# 游戏设计原理百条思维框架

## 1. 适用边界

| 应触发 | 不触发 |
|--------|--------|
| 设计新机制、系统、玩法 | 纯代码实现（不涉及设计决策） |
| 讨论平衡、数值、经济 | 纯 BUG 修复 |
| 讨论玩家体验、心流、教学、UI | 文档更新 |
| 评审或优化现有设计 | 资源管理等技术性任务 |
| 头脑风暴游戏创意 | 简单视觉微调（改颜色、调位置） |
| 设计奖励/惩罚/进度系统 | |
| 讨论叙事/主题/节奏 | |

---

## 2. 核心流程：三步法

### 第一步：识别设计维度

判断当前设计问题所属维度（可多选）：

| 维度 | 原理编号范围 | 典型问题 |
|------|-------------|---------|
| 创新 | 1-29 | 新机制、博弈设计、玩家分类、信息设计 |
| 创作 | 30-61 | 核心循环、主题、迭代、体验目标、叙事 |
| 平衡 | 62-79 | 数值、奖励、惩罚、难度曲线、心流 |
| 问题解决 | 80-100 | 教学、可用性、认知、节奏、成就感 |

### 第二步：调取相关原理

从项目设计参考文档中提取与当前问题最相关的 **3-6 条**原理。不要机械罗列全部100条，只引用真正相关的。

### 第三步：原理驱动的设计输出

将原理分析融入回复的思考过程，而非写成独立报告。输出格式：

```
【设计分析】
根据 [原理名] ...分析...
-> 建议: ...

根据 [原理名] ...分析...
-> 建议: ...
```

## 4. 使用原则

1. **原理是思考工具，不是教条**。引用原理用于支撑论点，不是凑数量。
2. **冲突时明确取舍**。例如"正反馈让人爽"与"负反馈防止碾压"同时成立时，需结合具体场景判断优先级并说明理由。
3. **结合项目上下文**。同一条原理在不同项目中可能导出不同建议。
4. **违反即指出**。若某个设计违反了某条原理，直接指出并解释为何可能构成问题。
