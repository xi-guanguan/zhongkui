# GDD 四阶段渐进式生成工作流

> **这是什么**: 将模糊的游戏创意通过四个递进阶段转化为可指导开发的结构化设计文档体系
> **原作者**: 节拍前夜
> **推荐度**: L2（有参考学习价值）
> **适用场景**: 从零开始设计游戏、整理已有创意为正式文档、建立可追溯的设计决策链

---

## 整体流程

```
用户创意 (模糊想法)
    |
[Phase 1] Brief Concept -- 快速概念，1-2 段话捕捉核心
    | 用户确认方向
[Phase 2] Game Concept Document -- 游戏概念文档，建立基本框架
    | 用户确认核心设计
[Phase 3] Game Design Document (GDD) -- 完整设计文档，可直接指导开发
    | 自动生成
[Phase 4] Systems Index -- 系统索引，拆分为独立子文档
```

关键原则：**不跳阶段**，每步都需要用户确认后才进入下一步。

---

## Phase 1: Brief Concept (快速概念)

**目标**: 用 1-2 段话捕捉游戏的核心本质。

**输出格式** (直接在对话中输出，不创建文件):

```
游戏概念: [游戏名称]

一句话描述: [用一句话说清楚这是什么游戏]

核心玩法: [玩家主要做什么？什么让它有趣？]

目标平台: [平台说明]

参考游戏: [1-3个类似的知名游戏]

独特卖点: [与参考游戏的关键区别]
```

**执行要点**:

1. 从用户的模糊描述中提炼出清晰概念
2. 主动提出 2-3 个方向性问题帮助用户明确想法
3. 等待用户确认后再进入下一阶段

---

## Phase 2: Game Concept Document (游戏概念文档)

**目标**: 展开核心设计，建立游戏的基本框架。

**输出**: `docs/game-concept.md`

**执行要点**:

1. 基于确认的 Brief Concept 展开详细设计
2. 重点关注三件事：**核心循环、关键机制、技术可行性**
3. 需要用户做决策的地方用 `[决策点]` 标记
4. 评估引擎的技术可行性，提前排除不可实现的设计

---

## Phase 3: Game Design Document (GDD)

**目标**: 完整的游戏设计文档，可直接指导开发。

**输出**: `docs/gdd.md`

**执行要点**:

1. 基于 Game Concept 展开完整设计
2. 所有系统都要有**具体的数值和规则**（不允许"适当的速度"这类模糊描述）
3. 标注技术实现方案
4. 包含具体的资源清单

---

## Phase 4: Systems Index (系统索引)

**目标**: 为每个游戏系统创建独立的详细设计文档索引。

**输出**: `docs/systems/README.md`

**执行要点**:

1. 从 GDD 中提取所有游戏系统
2. 为每个系统创建索引条目
3. 标注系统间的依赖关系
4. 按开发优先级排序

**文档目录结构**:

```
docs/
  game-concept.md          # Phase 2 输出
  gdd.md                   # Phase 3 输出
  systems/
    README.md              # Phase 4 输出 - 系统索引
    core-gameplay.md       # 核心玩法系统
    progression.md         # 进度系统
    ui-ux.md               # UI/UX 系统
    ...                    # 其他系统文档
```

---

## 四条设计原则

### 1. 技术约束前置

始终在设计阶段考虑引擎能力边界，而非编码阶段才发现不可行。涵盖渲染方案选择、性能约束（移动端/Web）、UI 技术栈等。

### 2. 渐进式细化

每个阶段都是上一阶段的展开和深化。不跳跃阶段，确保每步都有用户确认。所有需要用户决策的点都显式标记。

### 3. 可实现性优先

- 设计必须在目标引擎中可实现
- 优先考虑 MVP（最小可玩版本）
- 明确区分 "Must have" 与 "Nice to have"

### 4. 数值具体化

| 要求 | 正例 | 反例 |
|------|------|------|
| 给出具体初始值 | 移动速度 5.0 m/s | 适当的速度 |
| 标注调整范围 | 跳跃高度 2.0-4.0 m | 可调的跳跃高度 |
| 说明数值影响 | 攻击间隔 0.5s，DPS=20 | 合理的攻击频率 |

---

## 后续衔接

Phase 4 完成后，系统索引可直接映射为开发阶段的 Epic，进入生产管线：

```
GDD --> Epic 拆分 --> Story 细化 --> Sprint 规划
```
