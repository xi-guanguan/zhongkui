# 自由创作大世界游戏的存档架构

> **作者**: 点点 | **级别**: L2-参考 | **标签**: `存档` `云变量` `大世界` `增量存储`

---

## 核心目标

让玩家永远感知不到在等数据——"秒进世界、无感存档、断线不丢"。

---

## 三层 Key 架构

| Key 类型 | 前缀/命名 | 大小控制 | 内容 |
|----------|-----------|----------|------|
| world_meta | `wm_{worldId}` | < 2KB | 世界名、版本号、区块索引表、权限 |
| chunk | `ck_{worldId}_{chunkId}` | < 15KB | 地形格子、放置物件、NPC数据 |
| drawing | `dw_{worldId}_{drawId}` | ~4.5KB | 画布像素（v3编码，48色单字符） |

**设计思路**：大世界拆碎成多 Key，单 Key 上限 1MB，但控制在 15KB 以内保证读写速度。

---

## 增量脏标记写入策略

```
玩家操作 → 标记 chunk dirty
          → 不立即写入
          
定时器(30s) 或 离开区块 → 批量提交 dirty chunks
```

- **节流**：普通操作 30s 一次，切区块立即触发
- **合并**：同一 chunk 多次修改只提交最终态
- **优先级**：玩家所在 chunk > 相邻 chunk > 远处 chunk

---

## 客户端双写备份

```
写入 → clientCloud (主)
     → localStorage (备)

读取 ← clientCloud 优先
     ← localStorage 兜底（断网/超时）
```

**断线保护**：离线期间操作暂存 localStorage，上线后增量同步到云端。

---

## 登录优先级加载管线

| 阶段 | 加载内容 | 耗时目标 |
|------|----------|----------|
| Stage 1 | world_meta（索引） | < 200ms |
| Stage 2 | 玩家所在 chunk + 相邻 chunk | < 1s |
| Stage 3 | 其余可见 chunk（后台异步） | 不阻塞 |

**Push→Pull 模型**：
- 首次加载用 Pull（主动拉取 meta → 按索引拉 chunk）
- 运行时用 Push（其他玩家修改通过事件推送增量）

---

## v3 像素编码

解决"画布数据太大"的压缩方案：

- **48 色调色板**：单像素用 1 个可打印字符表示
- **编码密度**：256 字节可存一张 16×16 画布
- **空气压缩**：连续空白用 RLE 计数

```
原始 32×32 RGBA = 4096 字节
v3 编码后 ≈ 800~1200 字节（视内容复杂度）
```

## 关键数值

| 参数 | 值 | 说明 |
|------|-----|------|
| 单 Key 上限 | 1MB | 平台硬限制 |
| 推荐单 Key | < 15KB | 保证读写响应 < 100ms |
| 写入节流 | 30s | 防止频繁写入触发限流 |
| 画布像素 | 48色 | 平衡表现力与存储 |
| 首屏加载 | < 1s | Stage 1 + Stage 2 |

- 任何需要大量玩家生成内容持久化的游戏
