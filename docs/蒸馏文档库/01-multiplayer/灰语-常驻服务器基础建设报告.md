> **这是什么**: 基于引擎文档和业界最佳实践整理的常驻服务器架构全景报告，涵盖架构模型、通信协议、Tick 机制、场景复制、远程事件、玩家输入同步、兴趣管理、连接生命周期、三种运行模式、云变量持久化、网络调试和安全反作弊共 14 章。

## 一、架构总览

### 1.1 权威服务器模型

UrhoX 采用 **Server-Authoritative** 架构：

```
服务器 (Headless)
  - 运行完整游戏逻辑（移动、碰撞、AI、得分）
  - 无渲染，纯计算
  - 游戏对象使用 REPLICATED 模式创建
  - 从 connection.controls 读取玩家输入

客户端 (N个)
  - 负责渲染和用户输入
  - 通过 connection.controls 发送输入
  - ScriptObject:DelayedStart() 创建本地渲染组件
```

**核心原则**: 客户端发送意图，服务器验证合法性并决定结果。

### 1.2 推荐目录结构

```
scripts/
├── main.lua           # 入口（IsServerMode/IsClientMode 判断）
├── network/
│   ├── Shared.lua     # 共享（配置、事件名、工具函数）
│   ├── Server.lua     # 服务器逻辑
│   └── Client.lua     # 客户端逻辑
└── modules/           # 可复用模块
```

### 1.3 服务器全局变量

| 变量 | 类型 | 说明 |
|------|------|------|
| `SERVER_MAX_PLAYERS` | int | 最大玩家数 |
| `SERVER_TICK_RATE` | int | Tick 频率（帧率） |
| `SERVER_REGISTERED_PLAYERS` | int | 本局匹配到的玩家数量 |
| `SERVER_MODE` | string | 多人模式（如 `"server_authoritative"`） |
| `SERVER_PLAYER_AUTH_INFOS` | table | 玩家认证信息（含昵称） |
| `PERSISTENT_WORLD_KEY` | string | 常驻服房间标识（非空=常驻服模式） |

## 二、通信模型

### 2.1 通信层次

```
应用层 (Lua 脚本)       → Remote Events / Controls / Node Vars
同步层 (Scene Replication) → REPLICATED 节点自动同步 + SmoothedTransform 插值
引擎网络层               → 连接管理 / 序列化 / 压缩
传输层                   → UDP/TCP（引擎内部管理）
```

### 2.2 收发模式

| 方向 | API | 场景 |
|------|-----|------|
| 客户端→服务器 | `serverConnection:SendRemoteEvent(name, inOrder, data)` | 玩家操作 |
| 服务器→单客户端 | `connection:SendRemoteEvent(name, inOrder, data)` | 个人数据 |
| 服务器→全体 | `network:BroadcastRemoteEvent(name, inOrder, data)` | 全局广播 |
| 客户端输入(每帧) | `serverConnection.controls` | 移动、按钮 |
| 自动同步 | Node Vars / Position / Rotation | REPLICATED 节点属性 |

## 三、Tick 机制

### 3.1 Tick 循环

```
每个 GameTick:
  1. 接收客户端输入 (Poll Incoming)
  2. 处理远程事件 (Dispatch Events)
  3. 更新游戏逻辑 (Evolve World)
  4. 同步状态到客户端 (Enqueue Updates)
```

### 3.2 Tick Rate 参考

| 游戏类型 | 典型 Tick Rate |
|---------|---------------|
| 竞技射击 (Valorant) | 128 Hz |
| 主流射击 (CS:GO, OW) | 64 Hz |
| 大逃杀 (Fortnite) | 30 Hz |
| 休闲/社交 | 10-20 Hz |
| 回合制 | 1-5 Hz |

**选择原则**: 从低开始，按 1-2 Hz 递增，够用即可。

## 四、场景复制与状态同步

### 4.1 节点创建模式

| 模式 | 说明 | 创建端 |
|------|------|--------|
| **REPLICATED** | 自动同步到所有客户端（默认） | 仅服务器 |
| **LOCAL** | 仅存在于当前端 | 双端均可 |

**关键规则**: 客户端所有节点和组件必须用 LOCAL。

### 4.2 完整角色对照表

| 内容 | 服务器 | 客户端 |
|------|--------|--------|
| 游戏实体节点 | REPLICATED | （自动同步） |
| StaticModel | 不创建 | LOCAL |
| 材质/光照/相机 | 不创建 | LOCAL |
| 地面/墙壁 | LOCAL | LOCAL |
| 音效/粒子 | 不创建（发事件通知） | LOCAL |

### 4.3 节点变量同步 (Node Vars)

支持类型: Int, Float, Bool, String, Vector2, Vector3, Quaternion, Color

```lua
-- 服务器设置（自动同步）
node:SetVar("Health", Variant(100.0))
-- 客户端读取（需检查非空）
local var = node:GetVar("Health")
if var and not var:IsEmpty() then
    local health = var:GetFloat()
end
```

### 4.4 ScriptObject + DelayedStart 模式

**核心问题**: `NodeAdded` 触发时 Vars **尚未同步**。

**解决方案**: 服务器附加 ScriptObject，客户端在 `DelayedStart()` 中安全读取 Vars 并创建渲染组件。

### 4.5 关键规则

- **SmoothedTransform**: 引擎自动为 REPLICATED 节点创建位置插值，相机跟随不要加额外 lerp
- **节点删除**: 必须用 `node:Dispose()`，不要用 `node:Remove()`

## 五、远程事件系统

### 5.1 安全机制

远程事件必须**先注册后使用**（白名单机制）：

```lua
network:RegisterRemoteEvent("MyEvent")
SubscribeToEvent("MyEvent", "HandleMyEvent")
```

### 5.2 推荐: Shared.lua 统一管理

```lua
Shared.SERVER_EVENTS = { "ClientReady" }
Shared.CLIENT_EVENTS = { "AssignPlayer", "GameOver" }

function Shared.RegisterServerEvents()
    for _, name in ipairs(Shared.SERVER_EVENTS) do
        network:RegisterRemoteEvent(name)
    end
end
```

### 5.3 事件数据封装

```lua
local data = VariantMap()
data["Score"] = Variant(100)
data["Pos"]   = Variant(Vector3(1,2,3))
connection:SendRemoteEvent("MyEvent", true, data)
```

## 六、玩家输入同步

### Controls 结构

| 字段 | 类型 | 说明 |
|------|------|------|
| `yaw` | float | 水平旋转 |
| `pitch` | float | 垂直旋转 |
| `buttons` | uint | 按钮状态位标志 |

```
客户端每帧: 采集输入 → 计算 yaw/pitch/buttons → 写入 serverConnection.controls
服务器每帧: 遍历连接 → 读取 controls → 应用逻辑 → 更新 REPLICATED 节点
```

## 八、连接生命周期管理

### 8.1 关键时序（必须严格遵守）

```
客户端连接 → ClientConnected（记录连接，不设 scene）
→ 客户端设置 scene 并发送 Ready
→ ClientReady → 服务器设置 scene（触发全量同步）
```

**违反此时序**: `Can not handle LoadScene message without an assigned scene` 错误

### 8.2 认证事件时序

```
ClientConnected → ClientIdentity(user_id 可用) → ClientReady(自定义)
```

- `ClientIdentity` 后才能读取 `connection.identity["user_id"]`

## 九、三种服务器运行模式

| | 普通匹配 | 后台匹配 | 常驻服 |
|---|---|---|---|
| 开局方式 | 凑齐 N 人 | 脚本先加载，后台匹配 | 服务器先启动，随时加入 |
| 中途加入 | 不支持 | 不支持 | **支持** |
| 客户端获取连接 | Start() 直接获取 | 等 ServerReady 事件 | 等 ServerReady 事件 |
| 配置 | 默认 | `background_match: true` | `persistent_world.enabled: true` |

**常驻服关键差异**:
- 不能依赖 `SERVER_REGISTERED_PLAYERS`
- 用 `network:GetClientConnections()` 获取实时在线
- 玩家数据持久化需用 serverCloud

## 十、云变量与持久化存储

### 双层存储体系

| | clientCloud | serverCloud |
|---|---|---|
| 运行环境 | 客户端 | 服务端 |
| 操作对象 | 当前用户自己 | 任意用户（需传 userId） |
| API 范围 | Score + Rank | Score/Rank + Money/List/Item/Message/Quota + BatchCommit |

### 典型模式: 内存缓存 + 异步持久化

```
玩家加入 → BatchGet 加载到内存 → 游戏中操作内存（零延迟）
→ 关键节点 BatchCommit → 玩家离开 BatchSet 保存
```

### 使用限制

| 限制项 | 值 |
|--------|-----|
| 读/写请求频率 | 300 次/分钟 |
| 数据量 | 48 MB/分钟 |
| 单次 BatchCommit 操作数 | 1000 |

## 十二、安全与反作弊

- 权威服务器架构天然防护：客户端只发意图，服务器验证合法性
- 输入验证：方向向量 [0,1]、时间戳差 1-2 秒内、操作 ID 合法
- Ban 机制：`connection:Ban()`（重启后清空）
- 远程事件白名单：`RegisterRemoteEvent` 后才可接收
