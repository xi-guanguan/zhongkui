# 服务端权威架构三段式模式

> **这是什么**: 经过验证的服务端权威架构模式——Action 请求 → 服务端处理 → 状态广播，含 Shared 协议层、无状态 Manager 复用、saveDirty 定时存档等实践经验。  
> **原作者**: 心动  
> **推荐度**: L2（参考学习价值 — 已在实际项目中验证，适合从单机迁移到联机的中型项目参考）  
> **适用场景**: 服务端权威模式的多人游戏、需要单机/联机双模式支持的项目

---

## 核心架构：三段式

```
客户端 UI 操作
  ↓
Client.SendAction(action, params, callback)
  ↓
服务端 actionHandlers[action](playerInfo, params)
  ↓
修改 game state → pushStateToClient()
  ↓
客户端收到新状态 → UI 自动刷新
```

**好处**：所有业务逻辑集中在 `Server.lua` 的 `actionHandlers` 分发表中，客户端只做显示。

### 新增功能只需三步

| 步骤 | 文件 | 操作 |
|------|------|------|
| ① | `Shared.lua` | 在 `ACTIONS` 表中加一个常量 |
| ② | `Server.lua` | 在 `actionHandlers` 中加一个 handler |
| ③ | 客户端 UI | 调用 `Client.SendAction()` |

```lua
-- Server.lua —— actionHandlers 分发表
local actionHandlers = {
    [Shared.ACTIONS.PLACE_BID] = function(playerInfo, params)
        local result = AuctionManager.PlaceBid(playerInfo.game, params.itemId, params.price)
        pushStateToClient(playerInfo)
        return result
    end,

    [Shared.ACTIONS.COLLECT_REWARD] = function(playerInfo, params)
        local result = RewardManager.Collect(playerInfo.game, params.rewardId)
        playerInfo.saveDirty = true
        pushStateToClient(playerInfo)
        return result
    end,
    -- 30+ actions 以此类推...
}
```

## Shared.lua 协议契约层

创建 `Network/Shared.lua` 集中管理所有远程事件名和 Action 定义，服务端和客户端共同引用：

```lua
-- Network/Shared.lua
local Shared = {}

-- 远程事件名
Shared.EVENTS = {
    ACTION_REQUEST  = "ActionRequest",
    ACTION_RESPONSE = "ActionResponse",
    STATE_SYNC      = "StateSync",
    OFFLINE_REPORT  = "OfflineReport",
}

-- Action 常量（业务操作）
Shared.ACTIONS = {
    PLACE_BID       = "place_bid",
    COLLECT_REWARD  = "collect_reward",
    UPGRADE_BUILDING = "upgrade_building",
    -- ...共 30+ 个
}

-- 注册所有远程事件
for _, name in pairs(Shared.EVENTS) do
    RegisterRemoteEvent(name)
end

return Shared
```

### 好处

- **唯一定义点**：事件名/Action 名不会因拼写错误导致通信失败
- **一处添加，两端可用**：新增操作无需分别在客户端和服务端各写一遍
- **代码审查友好**：打开 `Shared.lua` 即可了解系统支持的全部操作

## 无状态 Manager 复用

将 Manager 改造为**无副作用的纯函数**，同一套代码在单机和联机模式下复用：

```lua
-- Manager 不持有状态，不主动触发存档
-- 只接收 game state，修改后返回
function BuildingManager.Upgrade(game, buildingId)
    local building = game.buildings[buildingId]
    if not building then return false, "not_found" end
    if game.gold < building.upgradeCost then return false, "no_gold" end

    game.gold = game.gold - building.upgradeCost
    building.level = building.level + 1
    return true
end
```

### 双模式调用

```lua
-- 单机模式（main.lua 直接调用）
function OnUpgradeButtonClick(buildingId)
    local ok, err = BuildingManager.Upgrade(game, buildingId)
    if ok then RefreshUI(game) end
end

-- 联机模式（Server.lua 在 handler 中调用同一个 Manager）
actionHandlers[Shared.ACTIONS.UPGRADE_BUILDING] = function(playerInfo, params)
    local ok, err = BuildingManager.Upgrade(playerInfo.game, params.buildingId)
    if ok then
        playerInfo.saveDirty = true
        pushStateToClient(playerInfo)
    end
    return { success = ok, error = err }
end
```

### 入口分流

```lua
-- main.lua
function Start()
    if IsServerMode and IsServerMode() then
        Module = require("Network.Server")
    else
        Module = require("Network.Client")
    end
    Module.Start()
end
```

同一套代码由引擎运行时决定角色（客户端/服务端）。

## saveDirty 定时批量存档

服务端不是每次操作都立即存档，而是标记脏数据后按固定间隔批量写入：

```lua
local SAVE_INTERVAL = 30  -- 秒

function ServerUpdate(dt)
    saveTimer = saveTimer + dt
    if saveTimer >= SAVE_INTERVAL then
        saveTimer = 0
        for _, playerInfo in pairs(onlinePlayers) do
            if playerInfo.saveDirty then
                SavePlayerData(playerInfo)
                playerInfo.saveDirty = false
            end
        end
    end
end
```

### 保底存档触发点

| 触发时机 | 说明 |
|---------|------|
| 定时器（30s） | 常规批量保存 |
| 玩家断线 | `ClientDisconnected` 事件中保存 |
| 服务器关闭 | shutdown 流程中遍历所有在线玩家保存 |

在性能和数据安全性之间取得平衡。

## 离线收益补偿

利用存档中的 `lastSaveTime` 计算离线时长，发放补偿：

```lua
function CalculateOfflineReward(playerInfo)
    local now = os.time()
    local offlineSeconds = now - (playerInfo.game.lastSaveTime or now)

    -- 设置上限，防止极端情况
    local OFFLINE_MAX_SECONDS = 24 * 3600  -- 最多算 24 小时
    offlineSeconds = math.min(offlineSeconds, OFFLINE_MAX_SECONDS)

    if offlineSeconds > 60 then  -- 至少离线 1 分钟才算
        local rewards = ComputeRewards(playerInfo.game, offlineSeconds)
        ApplyRewards(playerInfo.game, rewards)
        playerInfo.saveDirty = true

        -- 通过远程事件推送补偿明细给客户端展示
        local report = VariantMap()
        report["json"] = cjson.encode(rewards)
        playerInfo.conn:SendRemoteEvent(Shared.EVENTS.OFFLINE_REPORT, true, report)
    end
end
```

## 适用边界

- **适合**：需要单机/联机双模式的中型游戏、经营/策略/卡牌类需要服务端权威的项目
- **三段式优势**：逻辑集中、扩展简单、调试方便（只看 actionHandlers）
- **注意**：Action 数量多时（30+），`Shared.lua` 需要保持良好的分类注释
