# 多人服务器数据风暴治理指南

> **这是什么**: 多人常驻服务器在高并发场景下的数据读写风暴治理方案，从 API 减负到全局防御的 5 阶段渐进式架构演进。  
> **原作者**: 染墨轻尘  
> **推荐度**: L2（参考学习价值 — 来自实战经验，适合已有多人游戏基础的开发者深度参考）  
> **适用场景**: 多人常驻服务器、交易所/拍卖行等高频数据读写场景、百人同服优化

---

## 问题背景

多人常驻服务器在玩家并发量上升后，最先暴露的瓶颈往往不是逻辑复杂度，而是**数据读写风暴**：

- 每个玩家操作都触发一次 `serverCloud` 写入
- 关服/崩溃时大量保存请求同时涌入
- 多实例间数据竞争导致覆盖

本文将治理过程拆为 5 个阶段，每阶段解决一层问题。

---

## 阶段一：API 减负 — 从逐条写入到批量提交

### 核心武器：BatchSet

`BatchSet` 是降低 API 调用量的"核武器"——一次调用写入数百个 key：

```lua
-- ❌ 逐条写入（100 个 key = 100 次 API 调用）
for _, key in ipairs(keys) do
    serverCloud:Set(key, value)
end

-- ✅ 批量写入（100 个 key = 1 次 API 调用）
local batch = serverCloud:BatchCommit()
for _, key in ipairs(keys) do
    batch:Set(key, value)
end
batch:Submit()
```

### 关服生命周期重构

关服时的保存风暴是最常见的崩溃源。需要两层防护：

```
关服信号
  ↓
【阻塞保护层】
  - 立即拒绝新连接
  - 标记服务器为 SHUTTING_DOWN
  - 等待当前操作完成（超时兜底）
  ↓
【并发疏散层】
  - 对所有在线玩家并发执行 SaveAndKick
  - 每个玩家：BatchSet 保存 → 断开连接
  - 全部完成后才真正退出进程
```

**要点**：关服保存必须有超时兜底，单个玩家保存失败不能阻塞整个关服流程。

## 阶段二：玩家状态机 — 从无序到有序

### Player FSM

给每个玩家连接建立状态机，杜绝"数据还没加载完就开始写入"的竞态问题：

```
INIT → LOADING → READY → SAVING
 │                  │        │
 │                  │        └→ 保存完成 → 断开/回到 READY
 │                  └→ 正常游戏（可读写）
 └→ 连接建立，等待加载
```

**状态规则**：

| 状态 | 允许的操作 | 禁止的操作 |
|------|-----------|-----------|
| `INIT` | 无 | 一切读写 |
| `LOADING` | 读取云存档 | 写入、游戏操作 |
| `READY` | 读写、游戏操作 | — |
| `SAVING` | 等待保存完成 | 新的写入 |

### isDirty 延迟写入

不是每次数据变化都立即保存，而是标记脏数据，定时批量刷盘：

```lua
function MarkDirty(player)
    player.isDirty = true
end

-- 每 30 秒检查一次
function FlushDirtyPlayers()
    for _, player in pairs(onlinePlayers) do
        if player.isDirty and player.state == "READY" then
            BatchSavePlayer(player)
            player.isDirty = false
        end
    end
end
```

### 指令队列

当玩家处于非 READY 状态时，操作不丢弃而是排队：

```lua
function HandlePlayerAction(player, action)
    if player.state ~= "READY" then
        table.insert(player.pendingQueue, action)
        return
    end
    ExecuteAction(player, action)
end

-- LOADING → READY 时，回放队列
function OnPlayerReady(player)
    player.state = "READY"
    for _, action in ipairs(player.pendingQueue) do
        ExecuteAction(player, action)
    end
    player.pendingQueue = {}
end
```

## 阶段三：多实例协调 — Leader/Follower 架构

当服务器扩展到多实例时，共享数据（如交易市场）会产生写冲突。

### Leader 独裁模式

只有 Leader 实例拥有共享数据的写权限：

```
实例 A (Leader)    实例 B (Follower)    实例 C (Follower)
    │                   │                    │
    │← 转发操作 ────────┘                    │
    │← 转发操作 ─────────────────────────────┘
    │
    ↓
  写入共享数据
    │
    ↓
  广播结果给所有实例
```

### 微批次转发

Follower 不逐条转发，而是攒 0.5 秒的操作一次性发给 Leader：

```lua
-- Follower 端
local forwardBuffer = {}
local FORWARD_INTERVAL = 0.5

function QueueForward(operation)
    table.insert(forwardBuffer, operation)
end

function FlushForward()
    if #forwardBuffer > 0 then
        SendToLeader("BatchOperations", forwardBuffer)
        forwardBuffer = {}
    end
end
```

### 广播优化

共享数据（如订单簿）不广播全量，只广播摘要：

```lua
-- 订单簿广播（约 4KB，而非全量数据）
local summary = {
    topBids = GetTopN(bids, 10),    -- 买单前 10
    topAsks = GetTopN(asks, 10),    -- 卖单前 10
    lastPrice = lastTrade.price,
    volume24h = dayVolume,
}
BroadcastToAllInstances("OrderBookUpdate", summary)
```

## 阶段四：离线玩家处理 — GlobalOfflineMailbox

玩家下线期间发生的事件（交易成交、被攻击、收到邮件）不能丢失。

### 邮箱模式

```lua
-- 写入离线邮箱（玩家不在线时）
function SendToOfflinePlayer(playerId, event)
    local mailboxKey = "offline_mail:" .. playerId
    serverCloud:Append(mailboxKey, cjson.encode(event))
end

-- 玩家上线时消费邮箱（在 LOADING 阶段）
function ConsumeOfflineMail(player)
    local mailboxKey = "offline_mail:" .. player.id
    local events = serverCloud:Get(mailboxKey)
    if events then
        for _, event in ipairs(DecodeEvents(events)) do
            ApplyOfflineEvent(player, event)
        end
        serverCloud:Delete(mailboxKey)
        player.isDirty = true  -- 标记需要保存
    end
end
```

**执行时机**：在玩家 FSM 的 `LOADING` 阶段消费邮箱，消费完毕再进入 `READY`。

## 阶段五：防御性工程

### Optimistic UI（乐观更新）

跨实例操作有 3-5 秒延迟，UI 先乐观显示，后续修正：

```lua
-- 客户端：下单后立即显示 "Pending"
function OnPlaceOrder(order)
    AddToUI(order, "Pending")  -- 乐观显示
    SendToServer("PlaceOrder", order)
end

-- 服务端确认后更新状态
function OnOrderConfirmed(orderId)
    UpdateUIStatus(orderId, "已成交")
end

-- 超时未确认 → 标记异常
function CheckPendingTimeout()
    for _, order in ipairs(pendingOrders) do
        if Now() - order.time > 10 then
            UpdateUIStatus(order.id, "超时，请刷新")
        end
    end
end
```

### K 线冷热分离

高频数据和低频数据使用不同的保存周期：

| 数据类型 | 更新频率 | 保存周期 | 示例 |
|---------|---------|---------|------|
| 热数据 | 每秒级 | 30 秒 | 当前价格、实时订单簿 |
| 冷数据 | 分钟级 | 5-10 分钟 | K 线历史、统计报表 |

```lua
local SAVE_INTERVALS = {
    hot = 30,       -- 热数据 30 秒
    cold = 300,     -- 冷数据 5 分钟
}

function ScheduledSave(dataType)
    local interval = SAVE_INTERVALS[dataType]
    -- 按 interval 定时 BatchSet
end
```

### 全服静默模式

Leader 重新选举期间，暂停所有写操作，避免数据混乱：

```lua
function OnLeaderReElection()
    serverState = "SILENT"
    -- 所有写操作排队，不执行
    -- UI 显示"系统维护中，请稍候"
end

function OnNewLeaderElected()
    serverState = "ACTIVE"
    -- 回放队列中的操作
    FlushPendingOperations()
end
```

## 5 阶段总览

| 阶段 | 治理目标 | 核心手段 | 效果 |
|------|---------|---------|------|
| 1 | API 调用量 | BatchSet + 关服重构 | 调用量降 90%+ |
| 2 | 竞态条件 | Player FSM + isDirty | 消除无序读写 |
| 3 | 多实例冲突 | Leader/Follower | 单点写入，无冲突 |
| 4 | 离线事件 | GlobalOfflineMailbox | 事件不丢失 |
| 5 | 极端场景 | 乐观 UI + 冷热分离 + 静默模式 | 全方位防御 |

---

## 适用边界

- **适合**：常驻服务器、交易系统、拍卖行、大世界等高频读写场景
- **不适合**：纯单机游戏、房间制短局游戏（这些场景不需要这么重的架构）
- **注意**：Leader/Follower 模式需要多实例部署支持，单实例项目跳过阶段三即可
