## 一、连接管理：ConnectionManager 回调注册

### 问题
服务端需要在「玩家连接 / 断开」时执行多个业务逻辑（初始化存档、清理缓存、通知好友……），如果把所有逻辑写在一个函数里，耦合严重。

### 方案：回调注册模式

```lua
-- ConnectionManager 核心思路
local ConnectionManager = {}
local onConnectCallbacks = {}
local onDisconnectCallbacks = {}

function ConnectionManager.RegisterOnConnect(name, callback)
    onConnectCallbacks[name] = callback
end

function ConnectionManager.RegisterOnDisconnect(name, callback)
    onDisconnectCallbacks[name] = callback
end

-- 连接事件触发时，依次调用所有注册的回调
function ConnectionManager.OnPlayerConnect(player)
    for name, cb in pairs(onConnectCallbacks) do
        local ok, err = pcall(cb, player)
        if not ok then
            Log("ConnectionManager: [" .. name .. "] OnConnect error: " .. tostring(err))
        end
    end
end

-- 断开同理
function ConnectionManager.OnPlayerDisconnect(player)
    for name, cb in pairs(onDisconnectCallbacks) do
        local ok, err = pcall(cb, player)
        if not ok then
            Log("ConnectionManager: [" .. name .. "] OnDisconnect error: " .. tostring(err))
        end
    end
end
```

### 使用方式

各业务模块独立注册，互不干扰：

```lua
-- 存档模块注册
ConnectionManager.RegisterOnConnect("SaveSystem", function(player)
    SaveSystem.LoadPlayerData(player)
end)
ConnectionManager.RegisterOnDisconnect("SaveSystem", function(player)
    SaveSystem.FlushPlayerData(player)
end)

-- 好友模块注册
ConnectionManager.RegisterOnConnect("FriendSystem", function(player)
    FriendSystem.NotifyOnline(player)
end)
```

### 要点
- 每个回调用 `pcall` 包裹，一个模块出错不影响其他模块
- 回调有名字，方便日志追踪和动态注销
- 解耦各业务系统，新增模块只需 `Register` 一行

## 二、存档防丢：四层防御体系

### 问题
网游最怕丢存档。serverCloud 是远程 API，网络抖动 / 进程被杀 / 写入超时都可能丢数据。

### 四层防御架构

```
┌─────────────┐
│  L1: 内存缓存  │  ← 读写最快，进程内有效
├─────────────┤
│  L2: 写入队列  │  ← 攒批写入，减少 API 调用
├─────────────┤
│  L3: 自动刷盘  │  ← 定时 / 断线时强制写入 serverCloud
├─────────────┤
│  L4: 客户端缓存 │  ← 最后防线，服务端全丢时客户端可恢复
└─────────────┘
```

### L1：内存缓存

```lua
-- 玩家数据加载后缓存在内存，所有读操作走内存
local playerDataCache = {}  -- playerId → data

function SaveSystem.LoadPlayerData(player)
    local data = serverCloud:Get("player_" .. player.id)
    playerDataCache[player.id] = data or {}
end

function SaveSystem.GetData(playerId, key)
    return playerDataCache[playerId] and playerDataCache[playerId][key]
end

function SaveSystem.SetData(playerId, key, value)
    if not playerDataCache[playerId] then return end
    playerDataCache[playerId][key] = value
    WriteQueue.MarkDirty(playerId)  -- 标记脏数据，等待写入
end
```

### L2：写入队列（攒批 + 去重）

```lua
local WriteQueue = {}
local dirtyPlayers = {}  -- 待写入的玩家 ID 集合（去重）

function WriteQueue.MarkDirty(playerId)
    dirtyPlayers[playerId] = true
end

-- 批量写入：把多个脏数据合并为一次 BatchSet
function WriteQueue.Flush()
    local batch = {}
    for playerId, _ in pairs(dirtyPlayers) do
        local data = playerDataCache[playerId]
        if data then
            batch["player_" .. playerId] = data
        end
    end
    if next(batch) then
        serverCloud:BatchSet(batch)  -- 一次 API 调用写入多个玩家
    end
    dirtyPlayers = {}
end
```

### L3：自动刷盘

```lua
-- 定时刷盘（每 30 秒）
local FLUSH_INTERVAL = 30
local flushTimer = 0

function SaveSystem.Update(dt)
    flushTimer = flushTimer + dt
    if flushTimer >= FLUSH_INTERVAL then
        WriteQueue.Flush()
        flushTimer = 0
    end
end

-- 玩家断线时立即刷盘
ConnectionManager.RegisterOnDisconnect("SaveFlush", function(player)
    WriteQueue.FlushPlayer(player.id)  -- 只刷这一个玩家
end)
```

### L4：客户端缓存（最后防线）

```lua
-- 服务端定期把数据快照推送给客户端
function SaveSystem.PushToClient(player)
    local data = playerDataCache[player.id]
    SendToClient(player, "SaveSnapshot", data)
end

-- 客户端收到后存 clientCloud
-- 如果服务端完全丢失数据，可以从客户端恢复
```

### 要点
- **读操作永远走 L1 内存**，零延迟
- **写操作攒批到 L2**，减少 API 调用（省预算）
- **L3 定时 + 断线刷盘**，减少丢失窗口
- **L4 客户端兜底**，极端情况可恢复
- BatchSet 合并多个玩家的写入为一次 API 调用

## 三、serverCloud API 限流应对

### 限流规则

```
serverCloud 预算：300 次读 + 300 次写 / 每 60 秒
超限返回 HTTP 429
```

### 429 处理：FIFO 队列 + 去重 + 重试

```lua
local RequestQueue = {}
local queue = {}       -- FIFO 队列
local pendingKeys = {} -- 去重：同一个 key 只排队一次

function RequestQueue.Enqueue(key, operation, data)
    -- 去重：如果同一个 key 已经在队列中，更新数据而不是重复入队
    if pendingKeys[key] then
        -- 更新已排队的请求数据（latest-wins）
        for _, item in ipairs(queue) do
            if item.key == key then
                item.data = data
                break
            end
        end
        return
    end
    
    table.insert(queue, { key = key, operation = operation, data = data })
    pendingKeys[key] = true
end

function RequestQueue.ProcessNext()
    if #queue == 0 then return end
    local item = table.remove(queue, 1)  -- FIFO
    pendingKeys[item.key] = nil
    
    local ok, err = pcall(function()
        if item.operation == "set" then
            serverCloud:Set(item.key, item.data)
        elseif item.operation == "get" then
            serverCloud:Get(item.key)
        end
    end)
    
    if not ok and tostring(err):find("429") then
        -- 429 了，放回队首，等下一轮重试
        table.insert(queue, 1, item)
        pendingKeys[item.key] = true
        return "rate_limited"
    end
end
```

### 快速重连队列截断

```lua
-- 玩家断线重连时，清空该玩家的待执行队列
-- 避免旧请求和新请求冲突
function RequestQueue.ClearPlayer(playerId)
    local prefix = "player_" .. playerId
    for i = #queue, 1, -1 do
        if queue[i].key:find(prefix, 1, true) then
            pendingKeys[queue[i].key] = nil
            table.remove(queue, i)
        end
    end
end

ConnectionManager.RegisterOnConnect("QueueClean", function(player)
    RequestQueue.ClearPlayer(player.id)
end)
```

### 用 BatchGet / BatchSet 节省预算

```lua
-- ❌ 10 个玩家分别 Get = 10 次读
for _, pid in ipairs(playerIds) do
    serverCloud:Get("player_" .. pid)
end

-- ✅ BatchGet 一次搞定 = 1 次读
local keys = {}
for _, pid in ipairs(playerIds) do
    table.insert(keys, "player_" .. pid)
end
local results = serverCloud:BatchGet(keys)
```

### API 预算估算（100 人同服）

```
读预算：300 次/60秒
  登录加载：100 人 × 1 次 BatchGet = ~5 次（分批加载）
  实际利用率 ≈ 5%

写预算：300 次/60秒
  定时刷盘：每 30 秒攒批写入
  100 人 × 2 次/分钟 ÷ BatchSet 合并 ≈ 30 次/分钟
  实际利用率 ≈ 10%
```

**结论**：合理使用 Batch API + 攒批写入，100 人同服完全在预算内。

## 四、业务系统实战

### 4.1 商城：Quota API + BatchCommit

```lua
-- Quota API：原子性每日限购
-- 每个商品每天限购 N 次，Quota 自动计数+过期

-- 购买流程
function Shop.Buy(player, itemId)
    -- 1. Quota 检查 + 扣减（原子操作，防并发超买）
    local ok = serverCloud:QuotaConsume("shop_" .. itemId .. "_" .. player.id, 1)
    if not ok then
        return false, "今日已达购买上限"
    end
    
    -- 2. BatchCommit：扣钱 + 发货（事务性）
    serverCloud:BatchCommit({
        { op = "add", key = "gold_" .. player.id, value = -price },
        { op = "add", key = "item_" .. player.id .. "_" .. itemId, value = 1 },
    })
    
    return true
end
```

**要点**：
- `QuotaConsume` 是原子操作，天然防并发（多个请求同时买不会超限）
- `BatchCommit` 保证扣钱和发货要么都成功要么都失败
- Quota 自带过期，不用手动清理每日计数

### 4.2 邮件：名称缓存共享

```lua
-- 问题：邮件需要显示发件人名字，但频繁查询名字浪费 API
-- 方案：内存中维护名称缓存，多个模块共享

local nameCache = {}  -- playerId → playerName

function NameCache.Get(playerId)
    if nameCache[playerId] then return nameCache[playerId] end
    -- 未命中才查 serverCloud
    local name = serverCloud:Get("name_" .. playerId)
    nameCache[playerId] = name
    return name
end

-- 玩家上线时预加载
ConnectionManager.RegisterOnConnect("NameCache", function(player)
    nameCache[player.id] = player.name
end)
```

**要点**：邮件、好友、组队等多个模块都需要玩家名字，共享一份缓存避免重复查询。

### 4.3 组队：客户端 AI 队友（假多人）

```lua
-- 问题：匹配不到队友时体验差
-- 方案：客户端生成 AI 队友，看起来是多人，实际单机

-- 服务端只负责匹配，超时后通知客户端自己生成 AI
function Team.OnMatchTimeout(player)
    SendToClient(player, "SpawnAITeammates", {
        count = 2,
        difficulty = "normal",
    })
end
```

**要点**：
- AI 队友完全在客户端运行，不消耗服务端资源
- 对玩家来说体验一致（有队友一起打）
- 服务端只需要记录结算结果

### 4.4 师徒：List API

```lua
-- List API 适合有序列表场景（师徒列表、收徒记录）
function MasterDisciple.AddDisciple(masterId, discipleId)
    serverCloud:ListAppend("disciples_" .. masterId, {
        id = discipleId,
        time = os.time(),
    })
end

function MasterDisciple.GetDisciples(masterId)
    return serverCloud:ListGetAll("disciples_" .. masterId)
end
```

**要点**：List API 维护有序列表，比手动管理数组更可靠（原子追加、自动去重）。

### 4.5 结婚：BatchCommit 双向数据

```lua
-- 结婚涉及双方数据，必须事务性操作
function Marriage.Marry(playerA, playerB)
    serverCloud:BatchCommit({
        { op = "set", key = "spouse_" .. playerA, value = playerB },
        { op = "set", key = "spouse_" .. playerB, value = playerA },
        { op = "set", key = "marriage_time_" .. playerA, value = os.time() },
        { op = "set", key = "marriage_time_" .. playerB, value = os.time() },
    })
end
```

**要点**：双向数据必须用 `BatchCommit`，保证 A→B 和 B→A 同时生效，不会出现单方面已婚。

## 五、HeadlessMock：无头服务器兼容

### 问题
serverCloud 的某些 API 在 Headless（无头服务器）模式下不可用或行为不同，开发时需要 Mock。

### 方案

```lua
-- HeadlessMock：对不可用的 API 提供 stub 实现
local HeadlessMock = {}

if IsHeadless() then
    -- 替换不可用的 API 为内存实现
    function HeadlessMock.QuotaConsume(key, count)
        -- 简单的内存计数器模拟
        HeadlessMock._quotas = HeadlessMock._quotas or {}
        HeadlessMock._quotas[key] = (HeadlessMock._quotas[key] or 0) + count
        return true  -- 开发环境总是允许
    end
end
```

**要点**：开发阶段用 Mock 跑通流程，上线前替换为真实 API。

## 六、架构总结

### serverCloud API 使用策略

| 场景 | API | 说明 |
|------|-----|------|
| 单值读写 | `Get` / `Set` | 简单场景 |
| 批量读写 | `BatchGet` / `BatchSet` | **省预算首选** |
| 原子计数/限购 | `QuotaConsume` | 天然防并发 |
| 事务操作 | `BatchCommit` | 多 key 原子写入 |
| 有序列表 | `ListAppend` / `ListGetAll` | 师徒、日志等 |

### 性能关键数字

| 指标 | 数值 |
|------|------|
| serverCloud 读预算 | 300 次 / 60 秒 |
| serverCloud 写预算 | 300 次 / 60 秒 |
| 存档刷盘间隔 | 30 秒 |
| 100 人读利用率 | ~5% |
| 100 人写利用率 | ~10% |

### 核心原则

1. **读走内存，写攒批**——减少 API 调用
2. **Batch 优先**——BatchGet/BatchSet 把 N 次调用压缩为 1 次
3. **429 要排队**——FIFO + 去重 + 重试，不要直接丢弃
4. **存档四层防丢**——内存 → 队列 → 定时刷盘 → 客户端兜底
5. **双向数据用 BatchCommit**——结婚、交易等必须事务性写入
