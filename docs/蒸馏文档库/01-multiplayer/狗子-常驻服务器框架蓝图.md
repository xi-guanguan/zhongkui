# 常驻服务器框架蓝图

> **这是什么**: 一份完整的常驻服务器框架设计蓝图，涵盖目录结构、模块化通信（ServerDispatcher/ClientDispatcher）、存档管理（SaveManager）、Server/Client 完整生命周期、场景同步、心跳机制和合规检查清单。
> **原作者**: 狗子
> **推荐度**: L2（参考学习价值 — 框架设计详尽，适合搭建生产级常驻服架构）
> **适用场景**: 从零搭建常驻服务器框架、理解 Dispatcher/SaveManager 分层设计、建立多人游戏合规检查体系

---

## 一、目录结构与文件分类

```
scripts/
├── shared/          # 双端共享（配置、事件名、工具函数）
├── server/          # 服务端专用（Server.lua、ServerDispatcher、SaveManager）
├── client/          # 客户端专用（Client.lua、ClientDispatcher、UI）
└── network/         # 网络协议定义
```

**依赖方向**: shared ← server / client（server 和 client 互不 require）

## 二、Headless 模式 Mock

服务端运行在 Headless 模式下，渲染相关全局对象不存在，需要 mock：

```lua
-- 在 Server.lua 入口处注入
if graphics == nil then
    graphics = { width = 0, height = 0 }
end
if console == nil then
    console = { visible = false }
end
if debugHud == nil then
    debugHud = { mode = 0 }
end
```

## 四、核心术语

| 术语 | 含义 |
|------|------|
| **Toast** | 轻量浮窗提示（自动消失） |
| **Modal** | 模态弹窗（必须用户确认） |
| **LOGO 加载页** | 连接期间的 LOGO 遮罩页 |
| **模块化传输分片** | 超过 50KB 的消息自动拆分发送（60KB 硬限制） |
| **实时表 (Real-time Table)** | 内存中的活跃数据引用，SaveManager 管理 |
| **脏标记 (markDirty)** | 双重职责：即时推送 + 存档去抖（5秒定时器） |
| **会话版本 (sessionVersion)** | 防止异步回调写入过期会话的数据 |
| **断线遮罩 (Disconnect Mask)** | 断线时覆盖全屏的重连 UI |
| **isInGame** | 标记玩家是否在游戏内，守卫链依赖 |

## 五、ServerDispatcher（服务端数据推送）

### 职责

- 数据推送出口（唯一出口）
- 分片发送（>50KB 自动拆分）
- 缓存与重发
- 增量推送
- 通知广播

### 核心 API

```lua
-- 推送数据给单个玩家
ServerDispatcher:Push(connection, moduleName, data)

-- 增量推送（仅变更字段）
ServerDispatcher:PushDelta(connection, moduleName, delta)

-- 广播通知
ServerDispatcher:Broadcast(moduleName, data)

-- 分片发送（内部自动处理）
-- 超过 50KB 的消息自动拆分，60KB 硬限制
```

---

## 六、ClientDispatcher（客户端数据接收）

### 职责

- 数据接收与分片组装
- 订阅机制（UI 层按模块订阅数据变更）
- 数据补齐（断线重连后请求全量数据）

### 核心 API

```lua
-- 订阅模块数据变更
ClientDispatcher:Subscribe(moduleName, callback)

-- 获取当前模块数据
ClientDispatcher:Get(moduleName)

-- 请求补齐（断线重连后）
ClientDispatcher:RequestFullSync()
```

## 七、SaveManager（存档管理）

### 7.1 模块注册

每个游戏子系统注册为独立存档模块：

```lua
SaveManager:Register(moduleName, {
    getDefault = function() return {} end,       -- 新玩家默认数据
    onLoad = function(data) return data end,     -- 加载后处理
    onCleanup = function() end,                   -- 清理回调
})
```

### 7.2 实时表引用

```lua
-- 获取模块的实时表引用（内存中的活跃数据）
local playerData = SaveManager:GetRealTimeTable(moduleName)
```

### 7.3 脏标记（双重职责）

```lua
SaveManager:MarkDirty(moduleName)
```

**即时推送**: 触发 ServerDispatcher 推送变更给客户端
**存档去抖**: 5 秒定时器，批量提交到云端（避免高频写入）

### 7.4 模板匹配与字段迁移

- **模板匹配**: 加载数据时与 `getDefault()` 模板对比，补充缺失字段
- **字段迁移**: 旧版本数据字段名/结构变化时的升级处理

### 7.5 BatchCommit

```lua
-- 批量提交所有脏模块到云端
SaveManager:BatchCommit(sessionVersion)
```

**sessionVersion 守卫**: 异步回调返回时检查 sessionVersion 是否匹配当前会话，防止写入过期数据。

## 八、Server.lua 核心设计

### 8.1 守卫链

所有客户端请求经过三层守卫：

```
请求进入 → 频率限制 → 数据就绪检查 → isInGame 检查 → 业务逻辑
```

### 8.2 请求/响应协议

```lua
-- 客户端发送请求
serverConnection:SendRemoteEvent("Request", true, {
    action = "buyItem",
    itemId = 1001,
})

-- 服务端处理并响应
connection:SendRemoteEvent("Response", true, {
    action = "buyItem",
    success = true,
    message = "购买成功",
})
```

### 8.3 进入游戏流程

三条路径：

| 路径 | 条件 | 处理 |
|------|------|------|
| 新玩家 | 无云存档 | 使用 getDefault() 初始化 → BatchSet 保存 |
| 返回玩家 | 有云存档 | BatchGet 加载 → 模板匹配补字段 → 推送 |
| 断线重连 | 内存有数据 | 跳过云读取 → 直接推送内存数据 |

### 8.4 云存档处理

| 情况 | 处理 |
|------|------|
| 超时 | 重试 1 次，仍失败则踢出 + Toast |
| 失败 | 踢出 + 错误提示 |
| 成功 | 加载数据 → 模板匹配 → 推送客户端 |
| 数据修复 | 发现结构异常时自动修复 + 标记脏 |

### 8.5 断线保留

| 模式 | 保留时长 |
|------|---------|
| 单人 | 5 分钟 |
| 多人 | 10 分钟 |

超时后清理内存数据 + BatchCommit 存档。

## 九、Client.lua 核心设计

### 9.1 连接流程

```
启动 → LOGO 加载页 → 连接服务器 → ClientIdentity → 发送 ClientReady
→ 接收全量数据 → 关闭 LOGO 页 → 进入游戏
```

### 9.2 断线处理

```
检测断线 → 显示断线遮罩（全屏 UI）
→ 自动重试（30 秒间隔）
→ 手动重试按钮（无限次）
→ 重连成功 → ClientDispatcher:RequestFullSync() → 关闭遮罩
```

## 十、场景同步

- 游戏实体使用 REPLICATED 节点
- 客户端渲染组件使用 LOCAL
- NetworkPriority 控制更新频率
- 相机跟随规则：不加额外 lerp，直接使用 SmoothedTransform
