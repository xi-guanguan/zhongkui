# 移动端点击事件双触发防抖方案

> **这是什么**：解决移动端（Android/iOS/Web）手指点一次屏幕、UI 回调执行两次的问题，以及重启游戏后弹窗残留导致无法点击的问题。
>
> **原作者**：游玩
> **推荐度**：L1（实战验证，部署即生效，两个 BUG 都是生产环境高频踩坑）
> **适用场景**：使用 `urhox-libs/UI` 库的移动端游戏；任何用到 Modal/Alert 弹窗且有"重新开始"功能的游戏

---

## 问题一：点一次触发两次

### 现象

手机上点一下按钮，回调执行了两次——买道具扣两份钱、关弹窗关了两层、开局开了两把。电脑上一切正常。

### 根因：SDL 触屏转鼠标机制

UrhoX 底层使用 SDL 处理输入。SDL 在触屏设备上，每次手指触碰除了发 Touch 事件，还额外模拟一个 Mouse 事件（为了兼容只处理鼠标的老程序）。

```
电脑点击：MouseButtonDown → onClick ✅ 一次

手机触碰：TouchBegin     → onClick ✅ 第一次（正常）
          MouseButtonDown → onClick ❌ 第二次（SDL 模拟，多余）
```

UI 框架用统一的 PointerEvent 模型，Mouse 和 Touch 被视为独立指针源（pointerId 不同），框架无法判断 Mouse 事件是真实的还是 SDL 模拟的。

### 解决方案：Touch 后 100ms 防抖

**核心思路**：Touch 事件到达后，在 100ms 窗口内忽略 Mouse 事件。真实外接鼠标不会紧跟 Touch，不受影响。

```
0ms    Touch 到达 → 正常处理，记录时间戳
2ms    Mouse 到达 → 距上次 Touch < 100ms → 丢弃 ✅
5000ms Mouse 到达 → 距上次 Touch > 100ms → 正常处理 ✅
```

### 实现代码

在 `UI.Init()` 之后调用，对 UI 事件处理函数做 monkey-patch：

```lua
function PatchTouchMouseDedup()
    local platform = GetPlatform()
    if platform ~= "Android" and platform ~= "iOS" and platform ~= "Web" then
        return
    end

    local DEBOUNCE = 0.100  -- 100ms 窗口期（秒）
    local lastTouchTime = -1  -- 初始 -1，避免启动头 100ms 误拦

    local origMouseDown  = UI.HandleMouseDown
    local origMouseUp    = UI.HandleMouseUp
    local origMouseMove  = UI.HandleMouseMove
    local origTouchBegin = UI.HandleTouchBegin

    local function now()
        return time.elapsedTime  -- 引擎秒数，可靠递增
    end

    UI.HandleTouchBegin = function(touchId, x, y, pressure)
        lastTouchTime = now()
        origTouchBegin(touchId, x, y, pressure)
    end

    UI.HandleMouseDown = function(x, y, button)
        if now() - lastTouchTime < DEBOUNCE then return end
        origMouseDown(x, y, button)
    end

    UI.HandleMouseUp = function(x, y, button)
        if now() - lastTouchTime < DEBOUNCE then return end
        origMouseUp(x, y, button)
    end

    UI.HandleMouseMove = function(x, y)
        if now() - lastTouchTime < DEBOUNCE then return end
        origMouseMove(x, y)
    end
end
```

### 关键细节

| 决策 | 原因 |
|------|------|
| 用 `time.elapsedTime` 而非 `os.clock()` | `os.clock()` 在嵌入式 Lua 中可能返回 0，导致所有鼠标事件被永久拦截 |
| `lastTouchTime` 初始值 `-1` 而非 `0` | 初始为 0 时，启动头 100ms 内的鼠标点击会被误拦 |
| 窗口期 100ms | SDL 模拟 Mouse 在同帧或下帧到达（<16ms），100ms 留足余量；人类两次独立点击间隔通常 >200ms |

## 问题二：重启后无法点击

### 现象

游戏结束点"重新开始"，新一局所有按钮都无法点击。

### 根因：overlay 栈残留幽灵弹窗

UI 框架的 overlay 栈管理 Modal 弹窗。游戏过程中可能弹出多个窗口（Alert 等），用户只关闭了最上面的，然后调用 `UI.SetRoot(nil, true)` 销毁整棵 UI 树。

问题是：那些未被 `Close()` 的弹窗，DOM 节点虽然随旧 UI 树销毁了，但仍然留在 overlay 栈中。这些"幽灵弹窗"的 `isOpen_` 还是 true，`HitTest()` 返回 true，拦截所有点击。

### 修复：重启前清空 overlay 栈

```lua
function RestartGame()
    -- 清除残留的 overlay 弹窗
    local stack = UI.GetOverlayStack()
    for i = #stack, 1, -1 do
        UI.PopOverlay()
    end
    UI.ClearFocus()

    -- 然后才销毁旧 UI、创建新 UI
    UI.SetRoot(nil, true)
    CreateUI()
end
```

## 速查

| 问题 | 根因 | 解决 |
|------|------|------|
| 移动端点击触发两次 | SDL 触屏自动模拟鼠标事件 | Touch 后 100ms 内忽略 Mouse（monkey-patch） |
| 重启后无法点击 | overlay 栈残留已销毁的弹窗 | 重启前 `PopOverlay()` 清空栈 + `ClearFocus()` |

两个问题本质都是框架层面的状态清理不彻底。在框架无法修改时，用户代码通过 monkey-patch 和手动清理解决。
