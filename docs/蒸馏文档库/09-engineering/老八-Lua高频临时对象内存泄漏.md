# Lua 高频场景下的临时对象内存泄漏排查与修复

> **这是什么**：当游戏在高频调用路径（如每秒 6-12 次攻击）中反复创建临时 table 和拼接字符串时，GC 来不及回收导致内存持续增长直至闪退。本文记录了一个真实案例的排查过程和修复方案。
>
> **原作者**：老八（《挂机封神记》制作人）
> **推荐度**：L2（问题分析透彻，修复思路清晰且通用，但文中是针对特定项目的代码结构，需要自行定位自己项目中的高频热点）
> **适用场景**：游戏挂机/高频战斗时内存持续增长导致闪退；已做对象池但问题依然存在；需要排查 Lua 层的隐性内存分配

---

## 一、问题现象

- 游戏在挂机时频繁闪退
- 已经做了对象池复用、缓存优化，问题依然存在
- 修了 10+ 次都没根治

## 二、根因分析

### 关键认知：对象池 ≠ 万事大吉

对象池管住了池内的模块，但**池外的高频小函数**才是真正的内存杀手。

### 量化思路：频率 × 单次分配量 = 总分配量

案例项目中，3 倍速下攻击频率达到**每秒 6-12 次**（最小间隔 0.25s），以下函数成为泄漏热点：

| 函数 | 调用频率 | 每次分配 | 总分配量 |
|------|---------|---------|---------|
| `Economy.GetPanel()` | 60 次/秒 | 20 字段的新 table | 1200 字段/秒 |
| `Battle.GetMonsterConfig()` | 12 次/秒 | 5 字段新 table + 字符串拼接 | 60+ 对象/秒 |
| `MonsterSkill` 模块 | 12 次/秒 × 1-3 个字符串 | 数值字符串拼接 | 12-36 字符串/秒 |
| `SetSystem.OnMonsterDeath()` | 每次击杀 | 3 字段新 table | 随击杀频率增长 |

还发现一个**无效分配**重灾区：`onMonsterSkillTrigger` 回调是空函数，生成的字符串创建完直接等 GC 回收，完全没被使用。

### Lua 字符串的特殊性

Lua 中字符串是**不可变类型**，每次拼接（`..` 运算符）都会生成新的字符串对象。在高频场景下：

```lua
-- 每次调用都生成新字符串对象
local key = "monster_" .. id .. "_hp"  -- 3 个临时字符串
```

## 三、修复方案

**核心思路：消灭高频路径上的所有临时对象分配。**

### 3.1 预分配缓存表复用

```lua
-- 修复前：每次调用创建新 table
function Economy.GetPanel()
    return {
        gold = self.gold,
        exp = self.exp,
        -- ... 20 个字段
    }
end

-- 修复后：预分配缓存表，帧级复用
local panelBuf_ = {}  -- 模块级预分配

function Economy.GetPanel()
    panelBuf_.gold = self.gold
    panelBuf_.exp = self.exp
    -- ... 复用同一个 table
    return panelBuf_
end
```

同理对 `GetMonsterConfig()`、`OnMonsterDeath()` 等函数做预分配处理。

### 3.2 消除动态字符串拼接

```lua
-- 修复前：每次攻击都拼接字符串
local msg = "造成 " .. damage .. " 点伤害"

-- 修复后：使用常量字符串 + 数值分离
-- 方案 A：用格式化（减少中间字符串）
local msg = string.format("造成 %d 点伤害", damage)

-- 方案 B：如果字符串本身不需要，直接移除
-- （如日志回调是空函数的情况）
```

### 3.3 移除无效代码路径

空回调函数中仍在生成数据 → 直接移除数据生成逻辑，或在调用前检查回调是否为空。

## 四、排查清单

当游戏出现"已做对象池但仍然内存增长"的情况时，按以下步骤排查：

1. **识别高频路径**：找出每帧/每次攻击/每次事件触发的所有函数调用
2. **检查临时分配**：在这些函数中搜索以下模式：
   - `{}` 或 `{ ... }` — 创建新 table
   - `..` — 字符串拼接
   - `string.format` — 生成新字符串（比 `..` 好但仍有分配）
   - `tostring()` — 数值转字符串
3. **量化分析**：计算 `调用频率 × 单次分配量`，找出总分配量最大的函数
4. **检查无效分配**：回调是空函数但仍在生成数据的情况
5. **逐个修复**：预分配复用 / 常量替代 / 移除无效路径

## 五、通用避坑原则

1. **不要迷信对象池**：池复用管住了池内模块，但池外的高频小函数才是隐形杀手
2. **字符串拼接是隐形炸弹**：Lua 字符串不可变，每次 `..` 都是新对象，高频场景务必避免
3. **做"频率 × 分配量"的量化分析**：不要只看单次开销，要结合调用频率算总量
4. **注意倍速放大效应**：3 倍速意味着分配量也放大 3 倍，GC 压力成倍增长
5. **空回调也可能有成本**：如果在调用回调前就生成了数据，即使回调为空也有分配开销
