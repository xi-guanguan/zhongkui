# Agent 助力 3D 应用移动端性能优化 6 倍方法论

> **这是什么**：将 PC 端重度 3D 应用（近万星体实例、数万条连线）移植到移动端时，从 8FPS 优化到 60FPS 的系统性方法论。涵盖 AI Agent 辅助优化工作流、渲染优化、CPU 计算优化、内存管理、机型适配等全链路实战经验。
>
> **原作者**：iwae（老菜），《Cosmos》3D 星图应用开发者
> **推荐度**：L2（方法论有量化数据支撑，优化手段通用性强，但基于 Cocos Creator 引擎，移植到其他引擎需要适配）
> **适用场景**：3D 应用/游戏从 PC 移植到移动端遇到性能瓶颈；重度渲染场景需要系统性优化；需要建立 AI 辅助性能优化工作流

## 二、AI Agent 辅助优化工作流

### 2.1 创建专用优化 Agent

为性能优化任务建立专用 AI Agent，由多个 MD 文档组成（System Prompt + 优化案例库）。

**Agent System Prompt 核心结构**：

```
角色定义：精通 [引擎名] 的高级性能优化专家

核心优化目标：
1. 减少函数调用开销 — 合并高频调用函数、减少调用层级
2. 降低运算复杂度 — 简化算法、减少不必要计算
3. 最大化对象复用 — 对象池、避免频繁创建销毁

具体优化要求（按维度拆分）：
- 函数调用优化：内联化、缓存函数引用、减少 update 中的调用
- 运算优化：缓存结果、循环外提取、位运算替代、查表法
- 对象复用：对象池、复用临时向量/颜色对象、预分配数组
- 内存优化：及时释放、移除事件监听、清理定时器

输出要求：
- 提供完整优化代码 + 标注关键优化点 + 性能提升预估 + 前后对比
```

**关键实践**：在 Agent 中积累优化案例，让 AI 能找到更底层的 API 和优化方法。

### 2.2 Profiler 驱动的优化流程

```
引擎 Profiler / Chrome Profiler / 第三方 Profiler
  ↓ 定位热点函数
将 Profiler 数据 + 源码喂给 AI Agent
  ↓ AI 分析瓶颈并给出建议
人工评审 AI 建议 → 实施优化
  ↓
再次 Profiler 验证效果
```

**Profiler 分析要点**：

| 分析维度 | 关注指标 |
|---------|---------|
| Main Thread | 长任务（> 50ms）、Scripting 时间占比 |
| Call Tree | 按 Self Time 排序找最耗时函数 |
| Bottom-tree | 高频调用的叶子节点函数 |

### 2.3 Agent 分析结果示例

Agent 能从 Profiler 数据中精准定位瓶颈并按严重程度分级：

| 严重程度 | 瓶颈 | 耗时占比 | 根因 |
|---------|------|---------|------|
| 严重 | updateWorldTransform | 19.8% | 节点数量过多、层级过深 |
| 严重 | dispatch（事件分发） | 36.2% | 事件监听器过多 |
| 次要 | pyramidCulling（视锥剔除） | 9.5% | 需剔除判断的对象过多 |
| 次要 | 匿名函数调用 | 33.1% | 大量闭包、循环中创建函数 |

> **注意**：引擎源码级优化需要对底层架构非常熟悉，AI 给的方案大部分不能直接用，需要人工二次修改才能跑起来。

## 三、CPU 优化：更新节流与时间分片

### 3.1 更新频率分档节流

**问题**：数千星球每帧都执行完整的 update 逻辑（可见性判断、Label 透明度等）。

**方案**：按重要程度给不同逻辑分配更新频率档位，非关键逻辑不必每帧执行。

```
档位定义：
  SUPER_HIGH = 1帧    （实时：位置、碰撞）
  HIGH       = 3帧    （高频：动画混合）
  MEDIUM     = 5帧    （中频：AI 决策）
  LOW        = 7帧    （低频：UI 刷新）
  SUPER_LOW  = 10帧   （极低频：可见性、Label）
```

**实现模式**：

```
帧计数器 frameCounter 每帧 +1
当 frameCounter >= 目标间隔 时：
    frameCounter 归零
    执行实际逻辑（syncActivating / syncInstancing / syncNameLabel）
否则：
    跳过本帧
```

**效果**：减少 30-40% 的函数调用开销。

### 3.2 时间分片（Time Slicing）

**问题**：虽然做了分帧加载，但近万个实例的帧计数器初始值都是 0，导致它们在同一帧集中触发 update，形成性能尖峰。

**方案**：随机化每个实例的初始帧计数器，将更新操作均匀分散到不同帧。

```
实例创建时：
  frameCounter = 随机整数(0, SUPER_LOW)   -- 而非固定 0
```

**效果**：消除帧率尖峰，CPU 负载曲线从"锯齿形"变为"平滑线"。

> **通用规则**：任何「N 帧执行一次」的节流机制，都应该给每个实例随机化初始偏移，避免同步触发。

## 四、CPU 优化：减少 RTS（旋转/平移/缩放）开销

### 4.1 DirtyFlag 脏标记检查

**问题**：`updateWorldTransform` 占比 19.8%，数千星球每帧都触发世界坐标变换。

**方案**：在设置 Scale/Position/Rotation 前先检查值是否真的变了，避免冗余的 RTS 运算和子节点级联更新。

```
更新缩放时：
  计算新 scale 值
  如果 lastScale == scale：
      跳过（不调用 setWorldScale）
  否则：
      setWorldScale(scale, scale, scale)
      lastScale = scale
```

> **关键认知**：引擎的 `setWorldScale`/`setPosition` 等方法会触发子节点级联更新（`invalidateChildren`），即使值没变也会执行。在节点数量多的场景中，这个开销非常可观。

### 4.2 简化旋转计算

**问题**：`setRotationFromEuler(0, angle, 0)` 即使只旋转 Y 轴，引擎内部也会计算全部三个轴的三角函数和完整四元数。

**方案**：当只需要单轴旋转时，直接计算简化版四元数，跳过不必要的三角函数调用。

```
原始 setRotationFromEuler 内部流程（即使 X=0, Z=0）：
  计算 6 次三角函数（sin/cos × 3 轴）
  执行完整四元数合成公式

优化后（仅 Y 轴旋转）：
  当 X=0, Z=0 时，sinX=0, cosX=1, sinZ=0, cosZ=1
  四元数简化为：q = (0, sin(halfY), 0, cos(halfY))
  只需 2 次三角函数调用
```

**效果**：每次旋转操作减少 4 次三角函数调用。数千实例每帧旋转时收益显著。

## 五、CPU 优化：内联计算与函数调用削减

### 5.1 向量运算内联展开

**问题**：热点路径中的 `Vec3.subtract`、`Vec3.dot`、`lengthSqr` 等工具函数带来调用栈开销和临时对象分配。

**方案**：将向量运算展开为直接的数学计算，并使用平方距离避免 `sqrt`。

```
优化前（工具函数链式调用）：
  subtract(tempVec, targetPos, cameraPos)   -- 创建临时向量
  dot(tempVec, forward)                      -- 函数调用
  length = tempVec.length()                  -- sqrt 运算

优化后（内联 + 提前剔除）：
  dx = target.x - camera.x
  dy = target.y - camera.y
  dz = target.z - camera.z
  sqDis = dx*dx + dy*dy + dz*dz              -- 平方距离，避免 sqrt
  如果 sqDis > 阈值：直接剔除（快速返回）
  dot = dx*fwd.x + dy*fwd.y + dz*fwd.z      -- 内联点积
```

**关键技巧**：

| 技巧 | 说明 |
|------|------|
| 平方距离比较 | 用 `sqDis > threshold²` 替代 `dis > threshold`，省一次 `sqrt` |
| 提前剔除 | 先做最廉价的距离检查，通过后再做点积检查 |
| 消除临时对象 | 直接用 `dx/dy/dz` 局部变量代替临时 Vec3 |
| 缓存访问 | 将 `camera.forwardDir`、`camera.node._pos` 缓存到局部变量 |

## 六、渲染优化：Shader 与 Uniform

### 6.1 Global Uniform 减少逐对象更新

**问题**：太阳系数百个星体使用自定义光照 Shader，每帧需要逐对象同步太阳位置（近百次 `setUniform` 调用）。

**方案**：将太阳位置定义为 Global Uniform，全局只更新一次，所有使用该 Shader 的对象自动获取。

```
优化前：
  每帧对每个星体：pass.setUniform(handle, sunPosition)  -- N 次调用

优化后：
  每帧全局设置一次 solarParams -- 1 次调用
  Shader 中直接读取：data.L = normalize(-worldPos + solarParams.xyz)
```

> **老版本引擎**：如果不支持自定义 Global Uniform，可以 hack 一些未使用的内置全局 Uniform 来实现同样效果。

### 6.2 多个 Uniform 合并

**问题**：大量单 float 的 Uniform 调用。

**方案**：将多个相关的 float Uniform 合并成一个 `vec4`。

```
优化前：
  setUniform(handlePow10, pow10)     -- 1 次调用
  setUniform(handleRatio, ratio)     -- 1 次调用
  setUniform(handleAlpha, alpha)     -- 1 次调用

优化后：
  setUniform(handleParams, vec4(pow10, ratio, alpha, 0))  -- 1 次调用
```

### 6.3 精简 Shader 减少编译时间

**问题**：引擎默认 Shader 包含大量通用功能模块（多光源、阴影、反射、天空盒等），产生大量变体，首次加载编译卡顿。

**方案**：

- 只保留项目实际使用的光照模型和特性开关
- 移除未使用的光照计算模块
- 减少预处理宏定义和条件编译分支
- 优化 BRDF 计算（结合项目实际场景简化）

**参考**：轻量化光照 Shader 实现 — [github.com/iwae/LightingModel](https://github.com/iwae/LightingModel)

## 七、机型适配：GPU 性能分级

### 7.1 设备分级策略

通过 GPU 型号识别设备档位，动态调整渲染参数：

| 档位 | 代表 GPU | 目标帧率 | 策略 |
|------|---------|---------|------|
| LOW | Mali-G52 / Adreno 618 | 30 FPS | 激进剔除、最低几何精度 |
| MEDIUM | Mali-G78 / Adreno 730 | 45 FPS | 中等剔除、适度精度 |
| HIGH | 旗舰 GPU | 60 FPS | 标准剔除、高精度 |
| ULTRA | PC 独显 / 编辑器 | 60 FPS | 原始精度 |

### 7.2 几何精度分级配置

不同档位使用不同的几何体段数（Segments），在低端机上绘制更简单的几何形状：

| 几何体 | LOW | MEDIUM | HIGH | ULTRA | LOW 降幅 |
|-------|-----|--------|------|-------|---------|
| 圆环 | 16 | 24 | 32 | 46 | -65% |
| 椭圆 | 64 | 128 | 192 | 256 | -75% |
| 圆柱 | 8 | 12 | 16 | 16 | -50% |
| 圆弧 | 24 | 32 | 48 | 64 | -62% |

**实现模式**：

```
启动时：
  识别 GPU 型号 → 匹配档位白名单/黑名单
  设置全局质量等级（LOW / MEDIUM / HIGH / ULTRA）
  所有几何渲染器读取对应档位的段数配置

运行时：
  可动态切换档位（设置菜单）
```

> **关键认知**：手机 GPU 通常和 CPU 性能强相关，用 GPU 分级即可覆盖整体设备能力。低端机降低几何精度对视觉影响很小（远距离看不出差异）。

## 八、内存优化：纹理压缩

### 8.1 移动端纹理降档 + ASTC 压缩

**问题**：PC 端纹理 1024-2048px，直接用于移动端导致 GFX 纹理内存 305 MB。

**方案**：构建独立的移动端资源 Bundle，纹理分辨率降档 + ASTC 压缩。

| 纹理类型 | PC 分辨率 | 移动端分辨率 | 压缩格式 |
|---------|----------|------------|---------|
| UI 纹理 | 1024-2048 | 512-1024 | ASTC 5x5 |
| 图片纹理 | 1024-2048 | 256-1024 | ASTC 6x6 |

**效果**：GFX 纹理内存 305 MB → 65 MB（-78.7%），视觉质量可接受。

> **ASTC 格式选择**：块尺寸越大压缩率越高但质量越低。UI 用 5x5（质量优先），普通纹理用 6x6（平衡）或 8x8（激进压缩）。ARM 平台原生支持 ASTC。

## 九、渲染优化：LOD 策略

在已有 LOD 的基础上，针对移动端做更激进的面数和 Shader 优化：

```
远距离：最低面数 + 最简 Shader（可能替换为 Billboard）
中距离：中等面数 + 简化 Shader
近距离：最高面数 + 完整 Shader（只有近距离观察时使用最高档）
```

> **移动端 LOD 经验**：切换距离应比 PC 端更近（屏幕小，远处细节看不到）；最低 LOD 可以做得更激进（面数更少、Shader 更简单）。

## 十、性能优化检查清单

按优先级排序的优化手段速查表：

| 优先级 | 优化手段 | 预期收益 | 实施难度 |
|-------|---------|---------|---------|
| P0 | 更新频率分档节流 + 时间分片 | 减少 30-40% 函数调用 | 低 |
| P0 | RTS 脏标记检查 | 减少大量无效变换计算 | 低 |
| P0 | 纹理压缩（ASTC）+ 降档 | 显存降低 70-80% | 中 |
| P1 | 向量运算内联 + 平方距离 | 热点函数提速数倍 | 低 |
| P1 | Global Uniform 替代逐对象更新 | 减少 N 次 setUniform 为 1 次 | 中 |
| P1 | 单轴旋转简化计算 | 每次省 4 次三角函数 | 低 |
| P1 | Uniform 合并（float → vec4） | 减少 CPU-GPU 通信次数 | 低 |
| P2 | 精简 Shader / 减少变体 | 加快首次加载、减少编译卡顿 | 高 |
| P2 | GPU 分级 + 几何精度适配 | 低端机可玩性保障 | 中 |
| P2 | 移动端 LOD 激进优化 | 三角形数大幅降低 | 中 |
