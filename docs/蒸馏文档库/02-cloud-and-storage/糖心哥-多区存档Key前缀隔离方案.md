# 多区存档 Key 前缀隔离方案

> **这是什么**: 不依赖多套服务器、纯靠云变量 key 前缀实现多区服数据隔离的轻量方案，含切区防串数据、分区清档、排行榜隔离、三层存档容灾。
> **原作者**: 糖心哥（项目：《深渊炼狱：堕落之塔》，已稳定运营 8 个区服）
> **推荐度**: L2（思路简洁、实战验证 8 区零串区事故；适合轻量级多区需求，重度MMO可能不够）
> **适用场景**: 单机/轻联网游戏需要多区服但不想部署多套服务器，用云变量（clientCloud/serverCloud）做数据存储

> **同类对比**（02目录下存档相关）：
> - **游玩-多槽位云端存档系统设计**：多槽位是"同一玩家多个存档"，本篇是"同一玩家多个区服"，维度不同
> - **糖心哥-双通道云存档零丢档系统**：聚焦存档通道可靠性，本篇聚焦多区数据隔离
> - 可组合：双通道保不丢档 + Key前缀保不串区

---

## 一、核心思路：一个函数隔离所有

所有存取操作经过 `zKey()` 函数，按区服前缀隔离：

```lua
local zonePrefix_ = ""  -- 1区为空，2区为"z2_"，以此类推

local function zKey(baseKey)
    if zonePrefix_ == "" then return baseKey end
    return zonePrefix_ .. baseKey
end
```

效果：

```
1区：progress_chapter = 15        （无前缀，兼容老数据）
2区：z2_progress_chapter = 3
5区：z5_progress_chapter = 8
```

**为什么 1 区不加前缀？** 游戏上线初期只有一个区，已有大量玩家数据。加前缀 = 全量迁移，风险太大。1 区保持原始 key，2 区起才加前缀，老数据零迁移。

---

## 二、分区独立版本控制（精准清档）

每个区独立版本号，改哪个只清哪个区：

```lua
Config.SAVE_DATA_VERSION    = 2   -- 1区版本
Config.Z2_SAVE_DATA_VERSION = 1   -- 2区版本
Config.Z3_SAVE_DATA_VERSION = 1   -- 3区版本
-- ...以此类推
```

**运营场景**：5 区出了刷金 BUG → `Z5_SAVE_DATA_VERSION` 加 1 → 发版只清 5 区，其他 7 个区无感知。

---

## 三、切区防串数据（核心防护）

**最易出的 BUG**：玩家从 4 区切到 5 区，前缀已改成 `z5_`，但 4 区还有异步回调没执行完（广告奖励、邮件领取），回调触发保存时数据就写到 5 区了。

### 3.1 切区过渡锁

```lua
local zoneTransitioning_ = false

function CloudSave.BeginZoneTransition()
    zoneTransitioning_ = true   -- 切区开始，锁住所有 Save
end

function CloudSave.EndZoneTransition()
    zoneTransitioning_ = false  -- 新区数据加载完毕，解锁
end
```

切区期间所有 Save 操作被拦截并打日志，直到新区数据完全加载才解锁。

### 3.2 存档保护锁

云端加载失败时锁住保存，防止空数据覆盖真实存档：

> 宁可这局不存，也不能把人存档清了。

## 四、排行榜隔离

排行榜 key 同样走前缀：

```
1区：best_score       → 1区关卡排行榜
5区：z5_best_score    → 5区关卡排行榜
```

**跨服榜**：某些榜不加区服前缀（如每日广告次数），所有区玩家在同一个榜竞争。

**版本号清榜**：GM 清榜时递增版本号，key 变成 `best_score_v2`，旧榜自然失效，不需逐条删除。

---

## 五、GM 广播：一次写入全区可见

利用排行榜附带数据机制，GM 把公告写入每个区的排行榜：

```lua
local ALL_ZONE_PREFIXES = {"", "z2_", "z3_", "z4_", "z5_", "z6_", "z7_", "z8_"}
-- 遍历所有区，逐一写入广播数据
```

玩家进入任意区服时读取该区排行榜即可拿到最新公告，复用已有基础设施。

## 六、区服状态动态管理

选区界面支持基于时间自动切换的多种状态：

| 状态 | 含义 | 颜色 |
|------|------|------|
| 推荐 | 当前活跃区 | 绿色 |
| 已满 | 老区，人多生态稳定 | 橙色 |
| 即将开放 | 未到开服时间 | 灰色 |
| 停止注册 | 不再允许创建新角色 | 橙色 |

新区自动福利激励（如 2 区以上发月卡），引导玩家分流。

## 七、三层存档容灾

```
clientCloud（云变量直连）
    ↓ 不可用
Server Proxy（服务端代理转发 serverCloud）
    ↓ 不可用
本地文件（最终兜底）
```

本地存档文件名也跟着区服走：1 区 `save.json`，2 区 `z2_save.json`。即使完全离线，各区数据也不会混。

---

## 八、方案要点总结

| 机制 | 作用 |
|------|------|
| `zKey()` 前缀函数 | 零迁移成本隔离所有数据 |
| 分区版本号 | 精准清档，互不影响 |
| 切区过渡锁 | 杜绝异步回调导致数据串区 |
| 存档保护锁 | 加载失败时禁止写入 |
| 排行榜版本号清榜 | 无需逐条删除，key 自然失效 |
| 本地文件名带前缀 | 离线状态下各区也不混 |

**运营结果**：8 个区服稳定运营，零串区事故。
