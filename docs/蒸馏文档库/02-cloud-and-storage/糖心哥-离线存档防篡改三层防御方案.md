# 离线存档防篡改三层防御方案

> **这是什么**: 常驻服架构下支持断网游玩的存档安全方案——双写+三级Fallback保可用性，HMAC签名+服务端增量校验+离线收益上限防作弊。
> **原作者**: 糖心哥（项目：武侠 RPG，常驻服+单机离线混合架构）
> **推荐度**: L2（完整的离线存档安全体系，含实际运营数据验证；阈值参数需按项目调整）
> **适用场景**: 常驻服架构支持断网游玩的游戏，需要在"离线可用"和"防作弊"之间取平衡

> **同类对比**（同作者三篇存档文档）：
> - **云存档加载失败防护指南**（L1）：聚焦"加载失败≠新玩家"的致命BUG，5步修复方案，偏防御性编程
> - **双通道云存档零丢档系统**（L2）：聚焦三层存档通道互为备份的架构设计，偏可靠性/不丢档
> - **本篇**：聚焦"离线期间存档被篡改"的安全对抗，三层防御体系，偏安全性/防作弊
> - 三篇互补：加载防护 → 写入可靠 → 篡改对抗，可组合使用

---

## 一、核心矛盾

三个需求同时满足：

| 需求 | 约束 |
|------|------|
| 断网也能玩 | 客户端必须有一份可用的本地存档 |
| 网络抖动不丢进度 | 保存必须本地优先，不依赖网络 |
| 不能让玩家改档上线 | 本地文件天然不可信 |

**一句话**：可用性靠客户端兜底，安全性靠服务端把关。

---

## 二、存档架构：双写 + 三级 Fallback

### 2.1 保存流程（30秒自动 + 切场景立即）

```
G 对象序列化为 JSON
  │
  ├── ① 写入本地 save_cache.json（必定执行，无论网络状态）
  │
  └── ② 发送 C2S_SaveUpdate 到服务端（有网才发）
          │
          服务端 → BatchSet 写入 serverCloud（10字段原子写入）
              ├── 成功 → dirty=false
              └── 失败 → dirty=true，下线时入队重试
```

**关键**：本地写入永远第一步，serverConn_ 为 nil 也照写本地。

### 2.2 加载流程（三级 Fallback）

```
C2S_SaveLoad →
  ├── 服务端正常返回 status="loaded" → 用云端数据（刷新本地缓存）
  ├── 服务端返回 "no_save"/"error"   → 读本地 save_cache.json
  └── 8 秒超时无响应                 → 读本地缓存，进入离线模式
```

### 2.3 服务端离线写入队列

玩家断线时可能还有脏数据没写入云端：

```
玩家断线
  ├── 尝试立即写入 serverCloud
  │     ├── 成功 → 数据安全落云
  │     └── 429限流 → 降级入队
  │
  └── 队列消费（每 0.3 秒一条）
        ├── 去重合并：同一玩家只保留最新
        ├── 429重试：最多 5 次，间隔 2 秒
        └── 兜底刷写：每 5 分钟扫描所有脏缓存强制入队
```

**快速重连拦截**：断线 3 秒内重连 → 从队列捞最新存档直接返回，不读云端旧数据（避免"断线重连发现回档"）。

## 三、防作弊三层防御

### 第一层：本地签名校验（防文件篡改）

```
写入: save_cache.json + save_cache.sig
      sig = HMAC-SHA256(json_content, device_key + salt)

读取: 验证签名 → 不匹配 → 丢弃本地，强制走云端
```

- 签名密钥 = 设备指纹 + 硬编码盐（一机一密）
- **目标**：拦截 90% 手动改文件行为，不追求绝对安全
- **踩坑**：一开始用固定字符串做密钥，被提取后批量改档 → 改成设备指纹混合

### 第二层：服务端增量校验（防数值飞跃）

服务端收到 C2S_SaveUpdate 时，对比 liveCache 中的上次快照：

| 字段 | 允许方向 | 单次最大变化 | 超限处理 |
|------|---------|-------------|---------|
| coins（银两） | 增/减 | +50000/次 | 截断到阈值 |
| gold（元宝） | **仅减少** | 增长一律拒绝 | 回滚到服务端值 |
| level | 仅增长 | +1/次 | 截断 |
| exp | 增/减 | +当前升级所需经验 | 截断 |

**元宝特殊处理**：
- 只有服务端充值/商城流程能增加（quota:Add 原子操作）
- 客户端存档中的元宝字段只允许减少（消费）
- 加载存档时，iscores 中的商城货币覆盖存档值（双重校验）
- **元宝的"真相源"始终在 iscores，客户端值只是显示缓存**

**踩坑**：阈值卡太紧 → 高爆发 Boss 关被误判 → 加 1.5 倍容差 + 白名单关卡加成

### 第三层：离线时长 + 收益上限（防挂机刷）

```
离线时长 = 当前时间 - 上次云端保存时间（服务端时间）
理论最大收益 = 离线时长 × 每秒最大收益率
实际增量 = 新存档值 - 旧存档值

if 实际增量 > 理论最大收益 × 1.5:   -- 50% 容差
    截断到理论最大值
```

- **踩坑**：一开始用 `os.time()` 客户端时间 → 改系统时间可绕过 → 改用"上次服务端成功保存的时间戳"

## 四、完整数据流

```
┌──────────────────────────────────────────────────┐
│                    客户端                         │
│                                                  │
│  G 对象 ─序列化→ JSON ─HMAC签名→ 本地缓存        │
│    │                              │              │
│    └────── 发送到服务端 ──────────┘              │
└──────────────────┬───────────────────────────────┘
                   ▼
┌──────────────────────────────────────────────────┐
│                    服务端                         │
│                                                  │
│  收到 JSON → 增量校验 → 异常截断/正常放行         │
│    │                                             │
│    ├── 在线：写 serverCloud + 更新 liveCache      │
│    └── 离线：入队 → 去重合并 → 定时消费写入       │
│                                                  │
│  元宝/银两：iscores 原子操作，加载时覆盖存档      │
└──────────────────────────────────────────────────┘
```

## 五、踩坑清单

| # | 坑 | 解决方案 |
|---|---|---|
| 1 | 签名密钥用固定字符串，被提取后批量改档 | 设备指纹 + 盐混合，一机一密 |
| 2 | 增量校验阈值太紧，高爆发关卡误判 | 1.5 倍容差 + 白名单关卡加成 |
| 3 | 离线时长用客户端 `os.time()`，改系统时间可绕过 | 改用服务端上次保存时间戳 |
| 4 | 快速重连时重复校验，时间差小触发误判 | 断线 2 秒内重连直接返回队列存档，跳过校验 |

## 六、运营数据

上线半个月：
- 服务端增量校验拦截异常存档 **200+次**（大部分是同一批用户反复尝试）
- 签名校验拦截更多（客户端侧直接丢弃走云端，未精确统计）
- 正常玩家**完全无感知**——断网照常玩，上线自动同步
