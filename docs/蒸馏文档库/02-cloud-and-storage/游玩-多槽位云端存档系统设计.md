# 多槽位云端存档系统设计

> **这是什么**: 基于 clientCloud 的多槽位云端存档架构设计，包含云 key 布局、数据分片、校验机制、版本迁移和容错策略。  
> **原作者**: 游玩  
> **推荐度**: L2（参考学习价值 — 完整的存档系统架构方案，适合需要多存档槽位的中大型游戏）  
> **适用场景**: 需要多存档槽位、数据量较大需要分片、需要版本迁移和校验的游戏

---

## 架构概览

```
┌─────────────────────────────────┐
│         SlotSaveSystem          │  ← 唯一对外接口
├─────────────────────────────────┤
│  云端存储层 (clientCloud)        │  ← 主存储
│  本地缓存层 (File)              │  ← 快速读取 + 离线兜底
│  校验层 (DJB2)                  │  ← 数据完整性
│  版本迁移层 (MIGRATIONS)         │  ← 跨版本兼容
└─────────────────────────────────┘
```

**设计原则**：云端优先、本地缓存、单一入口、原子写入。

## 云 Key 布局

每个存档槽位使用一组 key，避免单 key 数据过大：

```
save_meta               → 元数据（槽位列表、最后使用的槽位等）
s_1_head                → 槽位 1 头信息（版本号、校验和、保存时间）
s_1_groupA              → 槽位 1 数据分组 A（角色属性、等级、金币等）
s_1_groupB              → 槽位 1 数据分组 B（背包、装备、技能等）
s_2_head                → 槽位 2 头信息
s_2_groupA              → 槽位 2 数据分组 A
s_2_groupB              → 槽位 2 数据分组 B
...
```

### 为什么分片？

| 策略 | 优点 | 缺点 |
|------|------|------|
| 单 key 存所有 | 简单 | 数据大时读写慢，容易超限 |
| 按槽位分 key | 槽位独立，互不影响 | key 数量多 |
| **按槽位+分组分 key** | 粒度适中，按需加载 | 需要管理分组 |

### 元数据结构

```lua
-- save_meta
{
    version = 2,                    -- 存档系统版本
    slots = {1, 2},                 -- 已使用的槽位列表
    lastSlot = 1,                   -- 最近使用的槽位
    maxSlots = 5,                   -- 最大槽位数
}

-- s_N_head（槽位头信息）
{
    version = 3,                    -- 数据版本（用于迁移）
    checksum = "a1b2c3d4",          -- DJB2 校验和
    savedAt = 1700000000,           -- 保存时间戳
    playTime = 36000,               -- 游戏时长（秒）
    slotName = "我的存档 1",         -- 玩家自定义名称
}
```

## 数据分组策略

根据数据的读写频率和关联性分组：

```lua
-- s_N_groupA: 核心属性（频繁读写）
{
    level = 25,
    exp = 12400,
    gold = 98000,
    hp = 350,
    mp = 200,
    position = { x = 100.5, y = 0, z = -30.2 },
    currentQuest = "quest_017",
}

-- s_N_groupB: 物品和装备（中频读写）
{
    inventory = { ... },   -- 背包物品列表
    equipment = { ... },   -- 装备栏
    skills = { ... },      -- 技能列表
}

-- s_N_groupC: 进度和成就（低频读写）
{
    achievements = { ... },
    completedQuests = { ... },
    statistics = { ... },
}
```

**分组原则**：
- 高频一起读写的数据放同一组
- 加载存档时只需读 head + groupA 就能显示存档列表
- 进入游戏后再异步加载 groupB/C

## DJB2 校验机制

用校验和检测数据是否损坏：

```lua
-- DJB2 哈希算法
function DJB2(str)
    local hash = 5381
    for i = 1, #str do
        hash = ((hash << 5) + hash) + string.byte(str, i)
        hash = hash & 0xFFFFFFFF  -- 保持 32 位
    end
    return string.format("%08x", hash)
end

-- 保存时计算校验和
function SaveSlotData(slotId, data)
    local json = cjson.encode(data)
    local checksum = DJB2(json)

    -- head 中存储校验和
    local head = {
        version = CURRENT_VERSION,
        checksum = checksum,
        savedAt = os.time(),
    }

    clientCloud:Set("s_" .. slotId .. "_head", cjson.encode(head))
    clientCloud:Set("s_" .. slotId .. "_groupA", json)
end

-- 加载时验证校验和
function LoadSlotData(slotId, callback)
    clientCloud:Get("s_" .. slotId .. "_head", function(success, headJson)
        if not success then callback(false, "load_failed"); return end
        local head = cjson.decode(headJson)

        clientCloud:Get("s_" .. slotId .. "_groupA", function(success2, dataJson)
            if not success2 then callback(false, "load_failed"); return end

            -- 校验
            local actualChecksum = DJB2(dataJson)
            if actualChecksum ~= head.checksum then
                print("[存档] 校验和不匹配！数据可能损坏")
                callback(false, "checksum_mismatch")
                return
            end

            local data = cjson.decode(dataJson)
            callback(true, data)
        end)
    end)
end
```

## 原子本地写入

本地缓存使用 .tmp + rename 模式，防止写入中断导致文件损坏：

```lua
function AtomicLocalSave(filename, data)
    local json = cjson.encode(data)
    local tmpFile = filename .. ".tmp"

    -- 先写临时文件
    local f = File(tmpFile, "w")
    f:Write(json)
    f:Close()

    -- 原子重命名（操作系统保证原子性）
    fileSystem:Rename(tmpFile, filename)
end
```

## 版本迁移链

数据结构变更时，通过迁移链逐版本升级：

```lua
local CURRENT_VERSION = 3

local MIGRATIONS = {
    -- v1 → v2: 添加 mp 字段
    [1] = function(data)
        data.mp = data.mp or 100
        return data
    end,

    -- v2 → v3: inventory 从数组改为 dict
    [2] = function(data)
        if type(data.inventory) == "table" and data.inventory[1] then
            local newInv = {}
            for _, item in ipairs(data.inventory) do
                newInv[item.id] = item
            end
            data.inventory = newInv
        end
        return data
    end,
}

function MigrateData(data, fromVersion)
    for v = fromVersion, CURRENT_VERSION - 1 do
        local migrator = MIGRATIONS[v]
        if migrator then
            data = migrator(data)
            print(string.format("[存档] 迁移 v%d → v%d 完成", v, v + 1))
        end
    end
    return data
end
```

### 加载时自动迁移

```lua
function LoadAndMigrate(slotId, callback)
    LoadSlotData(slotId, function(success, data)
        if not success then callback(false, data); return end

        local head = GetSlotHead(slotId)
        if head.version < CURRENT_VERSION then
            data = MigrateData(data, head.version)
            -- 迁移后立即保存新版本
            SaveSlotData(slotId, data)
        end

        callback(true, data)
    end)
end
```

## MarkDirty 合并机制

避免每次数据变化都立即保存，收集变更后统一写入：

```lua
local SlotSaveSystem = {}
local isDirty = false
local autoSaveInterval = 30  -- 秒
local autoSaveTimer = 0

function SlotSaveSystem.MarkDirty()
    isDirty = true
end

function SlotSaveSystem.Update(dt)
    if not isDirty then return end

    autoSaveTimer = autoSaveTimer + dt
    if autoSaveTimer >= autoSaveInterval then
        autoSaveTimer = 0
        SlotSaveSystem.Save()
    end
end

function SlotSaveSystem.Save()
    if not isDirty then return end
    isDirty = false
    -- 保存到本地缓存（即时）
    AtomicLocalSave(localPath, currentData)
    -- 保存到云端（异步）
    SaveSlotData(currentSlotId, currentData)
end

-- 强制立即保存（关闭游戏、切后台时调用）
function SlotSaveSystem.SaveNow()
    isDirty = false
    autoSaveTimer = 0
    AtomicLocalSave(localPath, currentData)
    SaveSlotData(currentSlotId, currentData)
end
```

## 容错策略

### 云端保存失败 → 后台重试

```lua
local pendingCloudSave = nil
local retryCount = 0
local MAX_RETRY = 3
local RETRY_DELAYS = { 5, 15, 30 }

function SaveToCloudWithRetry(slotId, data)
    clientCloud:Set(key, value, function(success)
        if success then
            retryCount = 0
            pendingCloudSave = nil
        else
            retryCount = retryCount + 1
            if retryCount <= MAX_RETRY then
                local delay = RETRY_DELAYS[retryCount] or 30
                pendingCloudSave = { slotId = slotId, data = data }
                -- delay 秒后重试
                SetTimeout(delay, function()
                    if pendingCloudSave then
                        SaveToCloudWithRetry(pendingCloudSave.slotId, pendingCloudSave.data)
                    end
                end)
            else
                print("[存档] 云端保存失败，已保留本地缓存")
                retryCount = 0
            end
        end
    end)
end
```

### 校验和不匹配 → 回退本地备份

```lua
function HandleChecksumMismatch(slotId)
    print("[存档] 校验和不匹配，尝试从本地备份恢复")
    local backup = LoadLocalBackup(slotId)
    if backup then
        -- 用本地备份覆盖云端（本地备份是上次成功保存的）
        SaveSlotData(slotId, backup)
        return backup
    else
        -- 无备份，提示玩家
        ShowCorruptionWarning(slotId)
        return nil
    end
end
```

## 完整 API 接口

```lua
local SlotSaveSystem = {}

-- 初始化（游戏启动时调用一次）
SlotSaveSystem.Init(config)
-- config = { maxSlots = 5, autoSaveInterval = 30 }

-- 加载指定槽位（异步）
SlotSaveSystem.LoadSlot(slotId, callback)
-- callback(success, data)

-- 创建新槽位（返回新槽位 ID）
SlotSaveSystem.CreateNewSlot(slotName, callback)

-- 删除槽位
SlotSaveSystem.DeleteSlot(slotId, callback)

-- 异步保存（合并到下次自动保存）
SlotSaveSystem.Save()

-- 立即保存（关闭游戏时）
SlotSaveSystem.SaveNow()

-- 标记数据已变更
SlotSaveSystem.MarkDirty()

-- 每帧调用（驱动自动保存定时器）
SlotSaveSystem.Update(dt)
```

## 适用边界

- **适合**：有多存档需求的 RPG、模拟经营、沙盒游戏
- **过度设计**：休闲小游戏（单 key 存一个 JSON 就够了）
- **注意**：clientCloud 有 key 数量和单 key 大小限制，分片数量需要控制
