## 核心架构：三层分离

将存档系统拆为三层，彼此独立演进：

```
游戏系统 ──→ GameState(内存单例) ←──→ 序列化模块 ←──→ ServerSave(防抖) ←──→ serverCloud
```

| 层 | 职责 | 关键特征 |
|----|------|---------|
| GameState（运行时状态） | 内存中全局单例，所有游戏系统只读写它 | 不关心数据怎么存、存在哪 |
| 序列化模块 | GameState ↔ 紧凑格式的翻译 | 每个业务领域一个模块，互不干扰 |
| ServerSave（持久化通道） | 与云端通信 | 防抖、合并、重试、容错封装在此 |

**实际收益**：中间换过一次云端存储方案，GameState 和所有游戏系统一行没改，只动了 ServerSave 里的几个调用。

## 分槽存储

按业务领域切成独立"存档槽"，每个槽有自己的 key、序列化/反序列化函数。

**为什么分槽**：
1. 局部修改只重写对应槽，减少写入量
2. 每个槽体积可控，不容易撞上云端单条 ~13KB 限制
3. 某个槽损坏不会连带其他数据

**注册方式**：统一注册表 `SLOT_DEFS`，声明 key + serialize + deserialize。新增槽只需加一行，加载/保存逻辑自动覆盖。

## 大集合分片

会随游戏进程增长的数据必须分片。

**分片粒度计算**：
1. 算单个元素序列化后的平均体积
2. 反推每个分片能放多少条
3. 留 20%~30% 余量应对可选字段膨胀

**无限增长数据**（如日志）：分片 + FIFO 上限（如 50 条），老条目自动丢弃。

## 短键名 + 默认值兜底

**序列化用短键名**：`guildName` → `gn`，`guildLevel` → `gl`，30 个成员的存档光键名省 30%~40% 体积。

**反序列化必须给默认值**：

```lua
GameState.guildName = data.gn or "无名工会"
GameState.guildLevel = data.gl or 1
```

- 每个字段无例外都写 `or 默认值`
- 老存档在新版本加载时，缺失字段自动填充，天然向前兼容
- 绝大多数版本迭代不需要迁移脚本

**何时需要版本号迁移**：仅在破坏性变更时（字段类型变了、嵌套层级改了），引入版本号做显式迁移。

## 防抖写入

每次数据变更不立即写入，而是标记槽为"脏"，定时器合并写出。

```
玩家 5 秒内点了 3 次升级 → guild_core 标脏（实际只标 1 次）
                         → 5 秒后合并成 1 次 BatchSet 请求
```

**inflight + pending 双缓冲**：
- 上一次 BatchSet 未返回时，新脏标记存入 pending 队列
- inflight 完成后再发 pending
- 任意时刻最多两次网络调用在路上

**generation 计数器**：每次 flush 递增 generation，旧的 inflight 回调发现 generation 不匹配时自动失效，防止过期回调干扰新状态。

---

## 断线快照

防抖的盲区：玩家在定时器触发前断线，脏数据丢失。

**解决方案**：检测到断线或关闭时立即 flush——先拍当前状态快照（序列化），再以最高优先级排队写入。快照用断线那一刻的状态，不等 inflight 回来。

---

## 容错加载

- 每个槽的反序列化包在 `pcall` 里
- 某个槽解析失败 → 打日志 + 默认值填充 + 继续加载其他槽
- 有依赖关系的槽先加载被依赖方
- 所有槽加载完成后统一执行"加载后逻辑"（跨日检测、过期清理等）

---

## 货币和配额走专用通道

金币（Money）和门票（Quota）不走 Score 槽位，走云端专用原子操作接口。

**原因**：数值型资源需原子加减（防并发余额错乱），`Money:Add` / `Money:Cost` / `Quota:Add` 自带原子保证和负数校验。

**注意**：修改后不标脏（不走 Score），直接调云端接口，但 GameState 缓存值要同步更新。

---

## 新增字段的标准流程

**给现有槽加字段（4 步，无需迁移）**：
1. GameState 加字段和默认值
2. `resetAll()` 加重置
3. 对应模块 `serialize()` 加序列化（短键名）
4. 对应模块 `deserialize()` 加反序列化（带 `or 默认值`）

**新增全新槽位**：创建序列化模块 → 注册 SLOT_DEFS → 加 key 常量 → BatchGet 加一行 → GameState 加字段。

## 端到端验证

```
填充已知数据 → 序列化 → 写入云端 → 清空 GameState → 读取 → 反序列化 → 逐字段 deepEqual
```

**测试数据集**：新手空档、中期半满、满编极端、故意缺字段的旧版模拟。每次改序列化逻辑后跑一遍。

---

## 设计检查清单

- [ ] 运行时状态和持久化是否分层（GameState vs ServerSave）
- [ ] 是否按业务领域分槽，单槽体积是否在配额内
- [ ] 大集合是否分片，是否有增长上限
- [ ] 序列化是否用短键名，反序列化是否每个字段都有默认值
- [ ] 写入是否防抖合并，是否有 inflight/pending 双缓冲
- [ ] 断线时是否有快照 flush
- [ ] 加载是否 pcall 容错，单槽失败不影响全局
- [ ] 货币/配额是否走专用原子接口
- [ ] 是否有端到端往返测试
