## 一、首尾帧衔接原理

### 核心规则

> **第 N 段视频的最后一帧 = 第 N+1 段视频的首帧参考图**

使用 Seedance 的 `first_last_frame` 模式，将上一段视频的实际截图作为下一段的首帧输入，保证画面自然过渡。

### 关键细节

| 要点 | 说明 |
|------|------|
| 首帧来源 | 用上一段视频的**实际最后一帧截图**，不要重新生成图片 |
| 风格一致性 | Prompt 末尾固定后缀，如"治愈系动画风格，竖屏 9:16，暖色调" |
| 时长计算 | `文字字数 × 0.16s + 0.6s 停顿`，空行 = 1.5s |
| 段数分配 | 11 段：A 播放器 5 段，B 播放器 6 段（交替播放） |
| 总成本 | 约 10 万积分，约 50MB，约 120 秒总时长 |

### 生成流程

```
第 1 段：一张原始图 → first_frame 模式生成
         ↓ 取最后一帧截图
第 2 段：上一帧截图 → first_last_frame 模式生成
         ↓ 取最后一帧截图
第 3 段：上一帧截图 → first_last_frame 模式生成
         ...依此类推
```

## 二、双播放器预加载与无缝切换

### 为什么需要双播放器

单播放器在切换视频时会出现加载间隙（黑屏/卡顿）。双播放器方案：

- **Player A** 正在播放 → **Player B** 后台异步加载下一段
- A 播完后 → 显示 B 并播放 → A 开始加载再下一段
- 如此交替，实现零延迟切换

### 关键实现

```
播放序列：
  A 播 #1 → B 预加载 #2
  B 播 #2 → A 预加载 #3
  A 播 #3 → B 预加载 #4
  ...

切换逻辑：
  当前播放器 Finished 事件 → 隐藏当前 → 显示另一个 → 播放
```

### generation 计数器防过期回调

视频加载是异步的，快速切换场景可能导致旧回调生效。解决方案：

```lua
-- 每次发起加载时递增
generationCounter = generationCounter + 1
local myGen = generationCounter

-- 回调中检查是否过期
function onVideoReady()
    if myGen ~= generationCounter then return end  -- 已过期，丢弃
    -- 正常处理...
end
```

## 三、标题页：一张图变动态背景

### 效果目标

用一张静态图生成两段视频：
1. **循环背景**：标题页不断播放的动态背景
2. **开门过渡**：点击"开始游戏"后的丝滑过渡

### 实现步骤

| 步骤 | 模式 | 说明 |
|------|------|------|
| 1. 生成循环视频 | `first_frame` | 一张图 → 生成自然循环的动态背景 |
| 2. 生成开门视频 | `first_last_frame` | 首帧 = 静态图，末帧 = 游戏场景入口（如门打开画面） |

### 循环播放设置

将循环视频的 `loop` 属性设为 true，标题页始终有动态效果。点击"开始游戏"后切换到开门视频，开门视频播完进入游戏场景。

## 四、手机端兼容修复

### 问题根因

> **blob URL 在 Android WebView 中无法播放视频**

PC 端用 blob URL（`blob:https://...`）加载视频正常，但手机端 WebView 不支持 blob 协议的视频源。

### 解决方案：CDN 直链

放弃 blob URL，改用 HTTPS 直链从 CDN 加载视频：

```
blob:https://xxx/abc123     ← PC 可用，手机不行
https://cdn.example.com/v.mp4  ← 双端都行
```

### ffmpeg 编码参数（手机兼容）

```bash
ffmpeg -i input.mp4 \
  -movflags +faststart \    # 元数据前置，支持流式加载
  -profile:v main \         # Main Profile（非 High），兼容性更好
  -bf 0 \                   # 禁用 B 帧，减少解码复杂度
  -b:v 1500k \              # 码率 1.5Mbps，画质与体积平衡
  output.mp4
```

| 参数 | 作用 |
|------|------|
| `-movflags +faststart` | moov atom 前置，支持边下边播 |
| `-profile:v main` | 避免 High Profile 在低端机的兼容问题 |
| `-bf 0` | 去掉 B 帧，减少解码器压力 |
| `-b:v 1500k` | 1.5Mbps 码率，50MB/11 段够用 |

### CDN URL 探测（video_url_resolver.lua）

视频上传后 CDN 地址有多种格式，需要逐一尝试找到可用的。约 280 行的探测器逻辑：

```
输入：视频资源 ID
尝试：6 种 CDN URL 模式
方法：依次发 HTTP HEAD 请求
输出：第一个返回 200 的 URL
缓存：成功的模式记住，后续直接用
```

### GetVideoWidth/Height 返回 0 的修补

手机端播放器可能在视频加载完成后仍返回宽高为 0，需要手动补丁：

```lua
function PatchVideoSizeForMobile(player, knownWidth, knownHeight)
    local w = player:GetVideoWidth()
    local h = player:GetVideoHeight()
    if w == 0 or h == 0 then
        w = knownWidth   -- 使用预设的视频尺寸
        h = knownHeight
    end
    return w, h
end
```

## 五、分布式预加载策略

### 思路

不要在同一时刻加载所有视频，而是利用不同的"空闲时段"分散预加载：

| 预加载时段 | 加载内容 |
|-----------|---------|
| 标题页展示期间 | 预加载第 1 段剧情视频 |
| 第 1 段播放期间 | 预加载第 2 段 |
| 剧情文字展示期间 | 预加载下一段 |
| 小游戏/交互环节 | 预加载后续需要的段 |

### 好处

- 避免一次性下载 50MB 导致卡顿
- 利用玩家阅读/操作的空闲时间
- 网络慢时也能流畅体验

## 六、降级回退方案

### 当视频不可用时

```
检测视频支持 → 不支持 → 降级为：
  1. 静态图片背景（原始首帧图片）
  2. ADV 文字模式（纯文本叙事）
  3. 跳过动画直接进入游戏
```

### 建议

始终准备一套静态图 + 文字的 fallback 方案，确保：
- 低端设备也能完整体验剧情
- 网络异常时不会卡死

## 八、踩坑备忘

| 坑 | 现象 | 解法 |
|----|------|------|
| blob URL | 手机端视频黑屏/无法播放 | 改用 CDN HTTPS 直链 |
| 重新生成首帧图 | 第 N+1 段与第 N 段画面跳变 | 必须用第 N 段实际最后一帧截图 |
| 单播放器切换 | 视频间有黑屏间隙 | 双播放器 A/B 交替 |
| 快速跳转 | 旧回调覆盖新状态 | generation 计数器 |
| GetVideoWidth 返回 0 | 手机端布局错位 | PatchVideoSizeForMobile 手动补宽高 |
| 一次加载全部 | 首次进入长时间白屏 | 分布式预加载 |
| High Profile 编码 | 低端安卓解码失败 | Main Profile + 禁 B 帧 |
