# 文档驱动关卡编辑流程

| 字段 | 值 |
|------|-----|
| 这是什么 | 无图形化编辑器环境下的关卡制作方法论——用 Markdown 文件画 ASCII 网格当地图、表格摆放实体，AI 负责转换为可运行代码。覆盖三文件互转架构、瓦片地图表示、自然语言微调、文档同步流程 |
| 原作者 | 夜晔（觅光，2D 平台解谜，11 个关卡） |
| 推荐度 | L2（方法论通用，具体格式需适配自己的游戏） |
| 适用场景 | 基于瓦片地图的 2D 游戏，无图形化关卡编辑器时的替代方案 |

---

## 核心思路：一个关卡，三种文件

| 文件 | 格式 | 用途 |
|------|------|------|
| `level_protocol_*.md` | JSON | 精确完整的关卡存档/备份 |
| `level_edit_*.md` | ASCII 网格 + 实体表格 | 日常编辑入口，人类可视化 |
| `levels.lua` | Lua 代码 | 引擎直接读取的运行时数据 |

三种文件可随时互相转换：
- 改了 edit → AI 同步到 levels.lua
- 代码里直接改了 → AI 重新导出 edit
- protocol 作为改坏时的恢复存档点

---

## 瓦片地图表示

每个关卡的地形是一个字符串数组，每行代表一行瓦片：

| 字符 | 含义 | 碰撞 |
|------|------|------|
| `0` | 空气 | 无 |
| `1` | 实体方块（地面/墙壁） | 有 |
| `2` | 木质平台 | 有（可站立） |
| `3` | 装饰块 | 无（纯视觉） |

---

## level_edit 可视化格式

包含两部分：

**1. 带坐标标尺的 ASCII 网格**：行列标号 + 瓦片字符，肉眼可看出地图结构。

**2. 实体表格**：列出每个实体的类型、ID、坐标、初始状态等参数。

---

## 关卡制作完整流程

```
第一步：描述设计（自然语言 or ASCII 字符串）
    ↓
第二步：AI 生成三个文件（protocol + edit + levels.lua）
    ↓
第三步：构建 & 试玩，发现问题
    ↓
第四步：自然语言微调（不碰代码）
    ↓
第五步：再次构建 & 试玩（满意→下一关，不满意→重复第四步）
    ↓
第六步：同步文档（"重新导出 edit 和 protocol 文件"）
```

---

## 自然语言微调示例

这是这套流程最高效的部分——改关卡完全不需要碰代码：

| 操作类型 | 指令示例 |
|---------|---------|
| 移动实体 | "Level3 的 key_01 从 (15, 9) 移到 (8, 12)" |
| 改地形 | "Level3 的 y=10 行，x=25 到 x=30 的实体块挖空改成空气" |
| 新增实体 | "Level6 新增一支火把 torch05，放在 (15, 18)，初始状态灭" |
| 删除实体 | "Level9 的 enemy02 删掉" |
| 改参数 | "Level3 的 torch_02 改成初始点亮" |
| 批量修改 | 一次说多条，AI 一起改 |

---

## 设计描述的精细度

可灵活控制：

- **模糊构思时**：说明结构层次和核心挑战，如"右侧山体作为终点屏障，门在山顶；左侧开阔降低压迫感"
- **明确想法时**：指定关键坐标，AI 补足尺寸和细节
- **最精确时**：直接画 ASCII 字符串网格

---

## 方法优势

| 优势 | 说明 |
|------|------|
| 零工具依赖 | 不需要 TiledMap/LDtk 等编辑器，文本编辑器 + AI 即可 |
| 改关卡不碰代码 | 一句"torch_01 移到 (25, 8)"，不用找代码改数组索引 |
| 设计意图和数据分离 | 描述"右边有座山"，AI 翻译成具体网格数据 |
| 改动可追溯 | 三种文件互为备份，任一出问题可从其他两个恢复 |
| 批量操作高效 | "所有关卡重新导出 edit 文件"一句话批量同步 |

---

## 适用前提

1. **定义清楚数据协议**：哪些字符代表什么瓦片、哪些字段代表什么实体
2. **让文档可视化**：ASCII 网格 + 坐标标尺，一眼看出地图结构
3. **用自然语言驱动修改**：比在代码里找对应行改数字快十倍
