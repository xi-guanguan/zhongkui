# AI 协作手册创建器

> **这是什么**：一套让 AI 为游戏项目一键生成完整协作体系的工作流——包括 CLAUDE.md（智能体准则）、docs/ 文件夹结构、以及所有预设文档（策划/编码/检查/框架规则），覆盖从"创建模式"到"更新模式"的全生命周期。
>
> **原作者**：嗒啦啦
> **推荐度**：L1（完整度极高，含全套预设文件内容、四套工作流程、四条核心原则、两种架构预设，开箱即用）
> **适用场景**：为新游戏项目建立 AI 协作体系；已有项目升级协作规则版本；想要标准化团队 AI 协作流程

---

## 一、定位与核心思路

这不是一篇"教你怎么写 CLAUDE.md"的教程，而是一个**创建器**：AI 读取本文档后，通过 6 步引导用户完成配置，然后自动生成整套项目级协作文件。

**核心理念**：
- 项目中只需放一个入口文件（本文档），AI 读取后引导创建所有其他文件
- 所有预设文件内容内嵌在本文档中，创建时提取写入对应路径
- 两种模式：**创建模式**（首次建立）+ **更新模式**（版本迭代）

## 二、创建模式 — 六步流程

### 第 1 步：游戏概念

| 用户情况 | AI 处理方式 |
|---------|-----------|
| 有文档 | 读取并提取概念，总结后让用户确认 |
| 有概念没文档 | 让用户描述，总结后确认 |
| 没概念 | 逐项引导：游戏类型 → 核心玩法 → 单人/多人 → 参考游戏 → 其他想法 |

产出：`docs/策划文档/游戏概念.md`

### 第 2 步：架构决策

先扫描 `scripts/` 和 `docs/` 判断存量/新项目，然后选择架构：

| 架构 | 适用场景 | 代码结构 |
|------|---------|---------|
| 服务端权威 | 云存档、排行榜、多人联机、防作弊 | shared/server/client 三层 |
| 纯单机 | 离线游戏、无服务端需求、轻量原型 | 扁平 scripts/ |

**AI 建议原则**：
- 单人但需要排行榜/云存档 → 建议服务端权威
- 纯离线 → 纯单机
- 不确定 → 建议服务端权威（更灵活）

选择服务端权威时追问联网模式：常驻世界（MMO）/ 房间匹配（对战）/ 单人游玩（云存档）。

### 第 3 步：屏幕方向

横屏（动作/射击/赛车/MOBA）或竖屏（卡牌/放置/社交/休闲）。

### 第 4 步：可选功能

- **UI 编辑器**：运行时可视化调整 UI 的开发工具。启用则额外创建 `关联UI编辑器规则.md` + `UI编辑器蓝图.md`。

### 第 5 步：确认并创建

展示即将创建的内容，用户确认后执行——生成文件夹结构 + 所有预设文件 + 组装 CLAUDE.md。

### 第 6 步：完成报告

列出所有创建的文件，建议开新对话开始正式开发。

## 四、CLAUDE.md 智能体准则 — 核心设计

生成的 CLAUDE.md 包含以下核心模块：

### 4.1 启动指令（TodoWrite 标准步骤）

收到任何消息时的强制流程：

1. **派 Task 扫描索引**：读策划文档表 + 模块注册表，筛选本次任务需要的文件路径
2. **输出任务准备报告**：列出固定规则文件读取状态 + 任务相关文件 + 本次关注的原则，等待用户确认
3. **输出任务分析**：确定任务类型（策划/编码/修复/其他）→ 进入对应流程

关键约束：每步都等用户确认后才继续，不自动推进。

### 4.2 四条核心原则

| 原则 | 核心规则 |
|------|---------|
| **禁止幻觉** | 没读过的文件不引用；接口调用前先读定义；结论标注来源文件路径/行号 |
| **对齐理解** | 先复述再执行；功能设计由用户定、实现方式由 AI 设计；收到纠正时复述完整模型 |
| **效率原则** | 自己写的文件不重读；已读文件不重读；未读文件必须读；关联修改一并完成 |
| **禁止硬编码** | 五类硬编码识别（值替代/链路断裂/默认值掩盖/逻辑复制/字符串契约断裂） |

### 4.3 四套工作流程

#### 策划流程
1. 扩展 TodoWrite → 2. 向用户提方案（循环讨论至确认完整）→ 3. 组织策划文档（核心逻辑 + 共享定义分文件）→ 4. 生成策划检查清单 → 5. 逐条检查（引用原文）→ 6. 修复未通过项 → 7. 收尾更新索引

#### 编码流程
1. 扩展 TodoWrite → 2. 输出编码方案（文件/职责/行数表格，含拆分规则）→ 3. 生成编码检查清单 → 4. 编码 + 检查 → 5. 修复 → 6. 循环至全部通过 → 7. 收尾更新索引

#### 修复流程
1. 定位 → 2. 追链路（排查优先级：自身代码90% → API理解偏差9% → 外部因素1%）→ 3. 修上游不修下游（检查补丁信号）→ 4. 自验 → 5. 收尾

#### 其他流程
扩展 TodoWrite → 执行 → 收尾更新索引

### 4.4 TodoWrite 扩展规则

- 一条 = 一个可独立验证的动作
- 同一改动涉及多文件 → 每文件单独一条
- 条目从流程步骤/模版步骤/涉及文件中提取
- 扩展完后自检：有"一条包含多个独立动作"则继续拆分

## 五、关键预设文件详解

### 5.1 AI 处理文档

**策划文档表**：Task 子智能体快速定位用。每个策划文件一条记录（路径 + 状态 + 核心内容），AI 拿到任务后先扫此表判断关联文档。

**模块注册表**：每模块两行内容（路径 + 依赖 + 功能描述），AI 定位目标模块后再读源代码获取接口详情。

**硬编码收集**：暂存无法立即修复的硬编码问题。条目格式：触发条件 + 目标文件 + 问题描述 + 修复方式。开发/重构相关模块时一并处理，修完删除。

### 5.2 检查系统

**策划检查创建模版**：AI 根据策划内容动态生成检查清单。核心能力：
- 识别策划类型（系统/剧情/数值/UI/关卡/角色/任务/社交/教程/商城/成就/存档，12 类）
- 按类型的分析侧重逐段深度分析
- 穷尽提取所有可判定的检查条目（拆细不合并）
- 两种检查模式：独立策划 / 规范产出物（对照规范比对）

**编码检查创建模版**：AI 根据编码任务动态生成检查清单。核心能力：
- 识别编码类型（功能实现/UI开发/网络多人/数值经济/数据存档/AI行为/性能优化/基础设施重构，8 类）
- 从准守清单和开发范式中逐节提取可验证的检查条目
- 两种检查模式：独立编码 / 策划驱动编码（对照策划比对）

### 5.3 UI 通用规范

覆盖以下维度：
- **安全区**：刘海屏适配 + TapTap 胶囊避让区（右上角 100×40px）
- **字号体系**：8 级（xs 12px → 3xl 36px），最小 12px，默认 14px
- **交互规范**：最小触摸区 44×44px，相邻可交互元素间距 ≥ 8px，按下必须有视觉反馈
- **内容溢出**：禁止静默裁切和缩小字号；文本换行高度自适应；数值不换行，布局时确保空间
- **布局空间分配 5 步**：列出元素 → 确定数据范围 → 最大值计算 → 固定元素先占位 → 验证剩余空间

### 5.4 服务端权威架构规则

完整的服务端权威架构技术规格，核心要点：

- **三层目录**：shared（定义+纯查询）/ server（状态+处理+持久化）/ client（请求发送+UI展示）
- **两条通道**：请求/响应（操作 → 结果）+ StateUpdate（状态推送，markDirty 触发）
- **数据分发层**：ServerDispatcher（拆分→缓存→发送）↔ ClientDispatcher（收齐→组装→分发），单消息 ≤ 50KB
- **存档系统**：单表模型 + 统一防抖（5秒）+ 模块 register 接入，markDirty 双职责（即时推送+存档防抖）
- **Handler 五步**：读实时表 → 校验 → 修改 → markDirty → 返回 {success, reason?}
- **客户端消费**：只从 ClientDispatcher 获取数据；response 回调仅处理失败提示
- **守卫链**：认证 → 数据加载 → 频率限制 → 路由到 Handler
- **场景同步**：REPLICATED 节点（空间/视觉）vs StateUpdate（游戏数据）
- **Headless Mock**：Server.lua 顶部注入 graphics/console/debugHud 模拟

### 5.5 UI 编辑器蓝图（可选）

运行时可视化 UI 编辑工具的完整技术规格：

- **6 模块架构**：UIEditor（主控）→ ElementStore（数据层）→ SelectionManager（选中）→ PropertyPanel（属性面板）→ PropertyDefs（元数据）→ Exporter（导出）
- **工作流**：AI 写 UI（带 elementId）→ 开发者 INS 进编辑模式 → 点击选中 → 修改属性 → 导出 JSON → AI 读日志更新源码
- **模板系统**：DEFAULTS 表（单一数据源）→ RegisterTemplate → RegisterElement → 框架属性 vs 内容属性分离
- **属性优先级**：覆盖值 > 模板属性 > 独立属性 > widget 当前值
- **Widget 重建机制**：部分属性（如 whiteSpace）仅 Init 时生效，需完整 props 重建

## 六、更新模式

已有协作体系的项目升级版本时：

| 文件类型 | 更新策略 |
|---------|---------|
| AI处理文档（策划表/模块表/硬编码） | 只更新头部格式，保留用户积累的内容 |
| AI规则文档 + AI框架文档 | 有就覆盖，没有就不创建 |
| CLAUDE.md | 新模板替换主体，保留 `## 本项目` 以下内容 |

## 七、两种架构预设对比

| 维度 | 服务端权威（预设 A） | 纯单机（预设 B） |
|------|-------------------|----------------|
| 目录结构 | shared/server/client 三层 | 扁平 scripts/ |
| 状态管理 | 服务端持有权威状态 | 模块内部管理 |
| 数据定义 | shared/模块名/Defs.lua | 跟随模块，通过公开接口暴露 |
| 模块连接 | Handler + markDirty + Dispatcher | Init(上游引用) + Observable on/emit |
| 网络通信 | 请求/响应 + StateUpdate | 无 |

## 八、禁止硬编码 — 五类速查

| 类型 | 症状 | 正确做法 |
|------|------|---------|
| 值替代 | 有常量定义却写字面量 | 引用定义常量 |
| 链路断裂 | 有数据源却手动拼输出 | 调用模块接口 |
| 默认值掩盖 | 源头未实现用 `or` 兜底 | 保持报错暴露缺失 |
| 逻辑复制 | 别处已有同逻辑却重写 | 调用已有函数 |
| 字符串契约断裂 | 同一字符串多处各写字面量 | Defs 定义常量统一引用 |

**引用尚不存在时**：保持引用让 build 报错 → 记入硬编码收集.md → 不准临时硬编码。

## 九、检查清单实战要点

### 策划检查的"穷尽提取"原则
- 每句话问"包含几个可独立验证的点"→ 每个点单独一条
- 一句话通常拆 3-8 条，只列 1 条大概率拆得不够
- 隐含要求也要提取（策划没明说但实际必然涉及的）
- 只检查策划内容完整性和逻辑自洽性，不检查编码实现

### 编码检查的领域检查
- 逐节读规则文件，对每条问"能否通过审查代码判定遵守与否"
- 能 → 转化为检查条目；不能 → 跳过
- 提取代码可验证的规则约束，不提取策划阶段的设计决策

### 修复流程的补丁信号
出现以下任一信号必须停下重新追因：
- 在下游加保护（nil 检查、flag、默认值绕过上游问题）
- 复制代码微调（复制一段稍改，不改原件）
- 反复小改同一区域（3 次以上还没解决 → 停下，报告根因分析）
