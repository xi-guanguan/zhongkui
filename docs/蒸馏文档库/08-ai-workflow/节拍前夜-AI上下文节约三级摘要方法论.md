# AI 上下文节约 -- 三级摘要方法论

> **这是什么**: 通过三级摘要索引和行为规范，减少 AI 会话中无效上下文的读入量，从而降低积分消耗
> **原作者**: 节拍前夜
> **推荐度**: L2（有参考学习价值）
> **适用场景**: 多文件/多模块的中大型项目；需要多轮对话持续迭代的项目；积分敏感的开发场景

---

## 核心原理

上下文窗口越大，每轮对话消耗的积分越多。减少无效上下文读入 = 直接省积分。

---

## 与项目记忆文档的分工

| 机制 | 读取对象 | 解决的问题 |
|------|----------|-----------|
| 项目记忆文档（如 `docs/` 下的迭代日志、设计文档） | 项目经验（字典） | 迭代历史、设计决策、踩过的坑 |
| 三级摘要（`_summary.md`） | 代码路由（目录页） | 定位该深入哪个模块、哪个文件 |

两者互补：记忆文档回答"为什么这样做"，摘要索引回答"代码在哪里"。

---

## 规则 1: 三级摘要，逐层深入

读代码前必须按层级逐步深入，禁止跳级。

```
L0: scripts/_summary.md（约 50 行，模块路由）
    每模块 2 行：第 1 行"它是什么"，第 2 行"谁用它 / 它用谁"
    -- 判断该深入哪个模块，不相关的直接跳过
        |
        v
L1: scripts/xxx/_summary.md（约 30-80 行，文件索引）
    每文件一行说明 + 关键依赖
    -- 定位到具体哪个文件需要修改
        |
        v
L2: 具体 .lua 文件（仅在必须时读取）
    -- 优先只读相关函数/段落，不读整个文件
```

每一层都是过滤器：L0 过滤模块，L1 过滤文件，最终只有真正需要的代码进入上下文。

### 缺少 `_summary.md` 时的处理

1. 先 `ls` + 快速扫描文件头了解结构
2. 完成当前任务后，为该目录生成 `_summary.md`（为下次会话省积分）

---

## 规则 2: 建索引代替重复搜索

同类数据被查询 2 次以上时，停下来整理索引写入 `_summary.md`，后续直接查索引。

```
第 1 次 --> 正常搜索
第 2 次同类 --> 停下，整理索引写入 _summary.md
后续 --> 直接读索引
```

适合建索引的典型数据：ScreenManager 注册名、configKey 列表、GameTheme 颜色名、事件名等高频查询项。

---

## 规则 3: 先对齐方案再编码

方案不明确时先用纯文字讨论（上下文开销低），确认后再一次性编码。避免边写边改方向导致的反复读写浪费。

---

## 规则 4: 长会话模块切换提醒

当前模块已完成且用户要切换到无关模块时，主动提醒开新会话。新会话的上下文窗口从零开始，每轮消耗更低。

---

## 规则 5: `_summary.md` 实时同步

代码变更后 `_summary.md` 必须同步更新，否则下次会话的路由判断会指向错误位置。

### 何时更新

| 触发条件 | 更新哪一级 |
|---------|-----------|
| 新增/删除/重命名文件 | L0（模块索引）+ 对应 L1（文件索引） |
| 某文件职责发生变化（拆分、合并、大幅重构） | 对应 L1 |
| 新增模块目录 | L0 + 为新目录生成 L1 |
| 仅修改文件内部逻辑（不改职责、不改接口） | 不更新（避免过度开销） |

### 更新原则

- **改了就更新**：不要攒到最后批量更新，随手同步
- **保持两行格式**：L0 每模块严格 2 行（是什么 + 依赖），不膨胀
- **行数/文件数可粗略**：不需要精确到个位，量级对即可
- **高频索引同步**：如果 L0 底部的"高频查询索引"涉及的数据变了（如新增屏幕注册名），一并更新

---

## 实践要点总结

| 要点 | 说明 |
|------|------|
| L0 控制在约 50 行 | 每模块严格 2 行，超出则说明粒度太细 |
| L1 控制在 30-80 行 | 每文件一行，附关键依赖 |
| L2 按需读取 | 只读相关函数，不读整个文件 |
| 重复搜索 >= 2 次则建索引 | 索引写入 `_summary.md` 末尾 |
| 方案未定不动手写码 | 纯文字讨论成本远低于代码读写 |
| 模块切换时提醒开新会话 | 避免旧上下文持续计费 |
| `_summary.md` 随改随更新 | 过时的索引比没有索引更危险 |
