# AI协作手册创建器

> 本文档是 AI 协作体系的唯一入口。新项目中只有这一个文件。
> AI 读取本文档，引导用户完成创建，生成 CLAUDE.md + docs/ 文件夹结构 + 所有预设文档。
> 所有预设文件内容已嵌入本文档中，创建时从这里提取写入对应路径。

---

## 使用方式

- **创建模式**：首次为项目建立协作体系
- **更新模式**：已有协作体系，更新规则版本

---

## 创建模式

### 第一步：游戏概念

询问用户："你已经有游戏概念了吗？或者有策划文档、概述文档？"

| 情况 | 处理 |
|------|------|
| 有文档 | 读取文档，提取游戏概念，AI 总结后请用户确认 |
| 有概念没文档 | 让用户描述，AI 总结后请用户确认 |
| 没有概念 | 逐项引导：游戏类型 → 核心玩法 → 单人/多人 → 参考游戏 → 其他想法 |

完成后将概念整理为摘要，写入 `docs/策划文档/游戏概念.md`。

### 第二步：架构决策

**项目扫描**：先扫描 `scripts/` 和 `docs/`，判断是存量项目还是新项目。
- 存量项目：推断架构模式，展示给用户确认
- 新项目：进入选择

**两个选项**（向用户解释由来后让其选择）：

| 选项 | 适用场景 | 代码结构 |
|------|---------|---------|
| 服务端权威架构 | 需要云存档、排行榜、多人联机、防作弊 | shared/server/client 三层 |
| 纯单机 | 离线游戏、无服务端需求、轻量原型 | 扁平 scripts/ |

**AI 建议原则**：
- "单人但需要排行榜/云存档" → 建议服务端权威
- "纯离线/不需要联网" → 建议纯单机
- 不确定 → 建议服务端权威（更灵活）

**选择服务端权威时追问联网模式**：

| 联网模式 | 配置 |
|---------|------|
| 常驻世界（MMO/沙盒） | `persistent_world.enabled = true`，追问 max_players |
| 房间匹配（对战/合作） | `match_info` 配置，追问 max_players |
| 单人游玩（云存档/排行榜） | 固定配置：`max_players=2, player_number=1, immediately_start=true, background_match=true` |

**记录架构选择**，决定后续使用哪个代码组织预设。

### 第三步：屏幕方向

> "游戏使用横屏还是竖屏？"

| 选项 | 说明 |
|------|------|
| 横屏（landscape） | 动作、射击、赛车、MOBA |
| 竖屏（portrait） | 卡牌、放置、社交、休闲 |

### 第四步：可选功能

**UI 编辑器**：

> "是否启用 UI 编辑器？UI 编辑器让你在游戏运行时可视化调整 UI（位置、颜色、字号等），调整结果导出后由 AI 自动更新代码。适合需要频繁微调 UI 的项目。"

| 选择 | 结果 |
|------|------|
| 启用 | 创建 `关联UI编辑器规则.md` + `UI编辑器蓝图.md` |
| 不启用 | 跳过这两个文件 |

### 第五步：确认并创建

向用户展示即将创建的内容，确认后执行。

**创建文件夹结构和预设文件**：

```
docs/
├── 策划文档/                          # 策划流程产出放这里
│   └── 游戏概念.md                    # 第一步已创建
├── 代码目录.md                        # 空文件树，开发时填充
└── AI协作手册/
    ├── AI处理文档/
    │   ├── 策划文档表.md              # 从【预设：策划文档表】提取
    │   ├── 模块注册表.md              # 从【预设：模块注册表】提取
    │   └── 硬编码收集.md              # 从【预设：硬编码收集】提取
    ├── AI规则文档/
    │   ├── UI通用规范.md            # 从【预设：UI通用规范】提取
    │   ├── UI编辑器准守清单.md         # ⚡ 可选：由 UI 编辑器蓝图构建后生成
    │   ├── 服务器框架准守清单.md       # 由服务器框架蓝图构建后生成
    │   ├── 策划检查创建模版.md         # 从【预设：策划检查创建模版】提取
    │   └── 编码检查创建模版.md         # 从【预设：编码检查创建模版】提取
    └── AI框架文档/
        ├── UI编辑器蓝图.md            # ⚡ 可选：启用 UI 编辑器时创建
        └── 服务器框架蓝图.md          # 从【预设：服务器框架蓝图】提取
```

除标记 ⚡ 的文件外，其余预设文件默认全部创建。⚡ 文件根据第四步的选择决定是否创建。内容从本文档的【预设文件内容】章节提取。

**组装 CLAUDE.md**：

从本文档的【预设：智能体准则模板】提取内容，替换变量：
- `{{CODE_ORGANIZATION}}` → 根据架构选择填入对应预设（见【代码组织预设】）

将结果写入项目根目录 `CLAUDE.md`。

### 代码组织预设

**预设 A：服务端权威架构**

```markdown
#### 架构模式

服务端权威架构：
- 服务端持有权威状态，执行所有状态变更、校验、计算
- 客户端只做两件事：发送操作请求、展示服务端返回的数据
- shared 放定义和纯查询，禁止状态变更、校验、权威计算

#### 文件归类原则

| 目录 | 运行端 | 放什么 | 判定标准 |
|--|--|--|--|
| `shared/模块名/` | 双端都加载 | Defs 定义 + 纯查询函数 | 只读数据，禁止状态变更、校验、权威计算 |
| `server/模块名/` | 仅服务端 | State + Handler + 权威计算 | 持有可变状态，执行所有状态变更、校验、权威计算 |
| `client/模块名/` | 仅客户端 | 请求发送 + UI 展示 | 只展示服务端发来的数据，不持有游戏状态，不做逻辑计算 |
| `network/` | 按文件区分 | 网络基础设施（连接、分发层） | 传输层，不含业务逻辑 |
```

**预设 B：纯单机**

```markdown
#### 模块规范

- 一个文件 = 一个功能，返回 table
- 模块通过 Init(上游模块引用) 建立连接，main.lua 按依赖顺序调用 Init
- 上游模块通过 Observable mixin 获得 on/emit 能力，下游在 Init 时订阅
- 数据定义跟随模块，通过模块公开接口暴露，不设独立 defs/ 层
- 代码文件头部一行注释说明职责
```

### 第六步：完成报告

展示创建结果：
- 列出所有创建的文件
- 架构选择和联网配置
- 屏幕方向

建议用户："开一个新对话开始正式开发。新对话会自动加载 CLAUDE.md。"

如果选了服务端权威架构，提示："第一个任务建议让 AI 按 `docs/AI协作手册/AI框架文档/服务器框架蓝图.md` 构建框架基础设施。"

## 更新模式

用于已有协作体系的项目，更新规则版本。

### 更新规则

1. **AI处理文档**（策划文档表、模块注册表、硬编码收集）：**只更新头部格式**，保留用户积累的内容
2. **其余预设文件**（AI规则文档、AI框架文档）：有就覆盖，没有就不创建
3. **CLAUDE.md**：重新从本文档的【预设：智能体准则模板】提取，替换主体部分（启动指令 + 原则），保留 `## 本项目` 以下的内容不动

### 更新步骤

1. 读取现有 CLAUDE.md，提取 `## 本项目` 以下的内容
2. 用最新模板重新组装主体部分
3. 拼接：新主体 + 旧"本项目"内容
4. 覆盖写入 CLAUDE.md
5. 按上述规则更新预设文件
6. 报告更新了哪些文件

## 启动指令

收到任何消息时：没有 TodoWrite → 强制建立；已有 TodoWrite → 跳过本指令。

**TodoWrite 标准步骤**：

1. 派 Task 扫描 `docs/AI协作手册/AI处理文档/策划文档表.md` 和 `docs/AI协作手册/AI处理文档/模块注册表.md`，Task 的职责仅为：根据用户消息，从两个索引表中筛选出本次任务需要读取的文件路径列表并返回

2. 主 AI 执行所有读取判断后，**必须使用以下格式输出，等待用户确认后才能继续**：

   **📋 任务准备报告**

   **固定规则文件**
   | 文件 | 决策 | 原因 |
   |------|------|------|
   | 服务器框架准守清单 | ✅ 已读取 / ⏭️ 文件不存在 / ⏭️ 已在上下文 | |
   | UI编辑器准守清单   | ✅ 已读取 / ⏭️ 任务不涉及UI / ⏭️ 文件不存在 / ⏭️ 已在上下文 | |
   | UI通用规范     | ✅ 已读取 / ⏭️ 任务不涉及UI / ⏭️ 文件不存在 / ⏭️ 已在上下文 | |

   **任务相关文件**
   | 文件 | 决策 | 来源 |
   |------|------|------|
   | `路径` | ✅ 已读取 | 索引命中 / 用户引用 / 内部引用 |
   | `路径` | ⏭️ 跳过 | 已在上下文 |
   | `路径` | ❌ 未读取 | [原因] |

   **本次需关注的原则**
   | 原则 | 关注 | 原因 |
   |------|------|------|
   | 原则一：禁止幻觉   | ✅ / — | |
   | 原则二：对齐理解   | ✅ / — | |
   | 原则三：效率原则   | ✅ / — | |
   | 原则四：禁止硬编码 | ✅ / — | |

   准备报告完毕，是否继续？

3. 用户确认后，分析任务，**必须使用以下格式输出，等待用户确认后才能继续**：

   **🔍 任务分析**
   - 任务类型：策划任务 / 编码任务 / 修复BUG / 其他
   - 将进入流程：XXX流程
   - 理解要点：
     - [要点1]
     - [要点2]

   是否继续进入 XXX流程？

   - 是 → 扩展 TodoWrite（**收尾步骤必须作为最后几条加入**），进入对应流程
   - 否 → 取消 TodoWrite
   - **策划任务** → 策划流程
   - **编码任务** → 编码流程
   - **修复 BUG / 修改代码** → 修复流程
   - **其他任务** → 其他流程

### TodoWrite 扩展规则

**一条 TodoWrite = 一个可独立验证的动作**。禁止用一条概括多个文件或多个步骤。

- 条目从流程步骤、模版步骤、涉及的文件中提取，不能只凭理解列
- 同一改动涉及多个文件时，每个文件单独一条
- 扩展完后自检：存在"一条包含多个独立动作"的情况则继续拆分

### 策划流程

1. 开始根据任务进行操作，按上方扩展规则扩展 TodoWrite 步骤
2. 基于已读取的资料，向用户提出策划方案，禁止开始写文档。循环执行以下过程直到用户明确确认完整：

   - 方案必须足够详细，完整到可以直接指导后续文档编写，不能遗漏任何重要设计点
   - 有任何不明确的地方，必须提出来让用户决定，不能自行假设填充
   - 每轮讨论后，判断最终文档体量是否会超过 800 行；若超过，提出拆分方案让用户确认
   - 用户明确确认"策划完整、可以开始写"后，退出循环，进入下一步
3. 组织策划文档到文件夹中，以策划名命名子文件夹。一个文件一个职责，按内容性质分为两类文件：
   - **核心逻辑**写入 `XX策划.md`（本策划独有的流程、规则、与其他系统的交互逻辑）
   - **共享定义**写入 `XX策划-定义.md`（只写本系统暴露的内容：数据结构、枚举、常量、接口等。不写本系统需要消费的内容）
   ```
   docs/策划文档/XX策划/
   ├── XX策划.md          # 核心逻辑（流程、规则、交互逻辑）
   └── XX策划-定义.md     # 共享定义（本系统暴露的定义，供其他系统读取）
   ```
4. 如果上下文中没有 `docs/AI协作手册/AI规则文档/策划检查创建模版.md`，则读取。为当前策划生成临时策划检查清单文件（`docs/策划检查清单.tmp.md`），检查完成后删除
5. 开始检查：主 AI 逐条对照策划检查清单检查策划文档，检查约束：
   - 每条判断必须引用原文（文件路径 + 具体段落/行号），不能凭记忆判断"通过"
   - 不确定的条目必须回到原文重新确认，不能跳过
6. 将检查结果更新到策划检查清单：
   - 通过 → 打勾
   - 未通过 → 记录问题和建议
7. 循环：对未通过的问题逐项列出选项和建议，与用户讨论，修改策划，更新策划检查清单，进行下一项。退出条件：全部打勾，或用户主动说暂不处理
8. 收尾：删除临时策划检查清单文件、更新策划文档表.md 及其他相关文档（模块注册表.md、代码目录.md、硬编码收集.md 等，按实际涉及情况处理）。**必须使用以下格式输出**：

   **✅ 收尾报告**
   | 文件 | 操作 | 状态 |
   |------|------|------|
   | 策划检查清单.tmp.md | 删除 | ✅ 已删除 |
   | 策划文档表.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |
   | 模块注册表.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |
   | 代码目录.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |
   | 硬编码收集.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |

### 编码流程

1. 开始根据任务进行操作，按上方扩展规则扩展 TodoWrite 步骤
2. 向用户输出编码方案，**必须使用以下格式输出，等待用户确认后才能继续**：

   **💻 编码方案**

   | 文件 | 职责（一句话） | 操作 | 预估行数 |
   |------|--------------|------|---------|
   | `路径/文件.lua` | [一句话描述] | 新建 / 修改 | [X] 行 |

   > **拆分规则**
   > - 职责一句话说不清 → 拆为独立命名的文件
   > - 代码文件 > 1000 行 → 拆为 `_1`、`_2`
   > - 定义文件 > 2000 行 → 拆为 `_1`、`_2`

   **待确认**：
   1. [关键设计决策：需要用户确认的实现方式]
   2. [不明确的点，附上自己的理解或选项]

   确认后开始编码

3. 如果上下文中没有 `docs/AI协作手册/AI规则文档/编码检查创建模版.md`，则读取。为当前编码任务生成临时编码检查清单文件（`docs/编码检查清单.tmp.md`），编码完成后删除
4. 开始编码，完成后检查：主 AI 逐条对照编码检查清单检查代码，然后更新编码检查清单：

   - 已有但未解决 → 保留未勾
   - 新问题 → 列进去
   - 之前打勾但又出现 → 去掉勾
5. 直接修复问题
6. 循环：再次检查 → 更新编码检查清单 → 修复。退出条件：编码检查清单全部打勾
7. 收尾：删除临时编码检查清单文件、更新相关文档（策划文档表.md、模块注册表.md、代码目录.md、硬编码收集.md 等，按实际涉及情况处理）。**必须使用以下格式输出**：

   **✅ 收尾报告**
   | 文件 | 操作 | 状态 |
   |------|------|------|
   | 编码检查清单.tmp.md | 删除 | ✅ 已删除 |
   | 策划文档表.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |
   | 模块注册表.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |
   | 代码目录.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |
   | 硬编码收集.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |

### 修复流程

1. 定位问题：根据用户提供的报错信息或 BUG 描述，定位相关代码文件和出错位置，报告给用户
2. 追链路：还原实际执行路径，找到第一个偏离预期的环节。排查方向按优先级：自身代码逻辑（~90%）→ API 理解偏差（~9%）→ 外部因素（~1%），禁止跳过自身代码直接怀疑引擎/框架
3. 在偏离处修复：修上游不修下游。过程中检查补丁信号——出现以下任一信号必须立即停下重新追因：
   - 在下游加保护（nil 检查、flag、默认值绕过上游问题）
   - 复制代码微调（复制一段代码稍作修改，而不是改原件）
   - 反复小改同一区域（同一块代码改了三次以上还没解决 → 停下，向用户报告修了几次、每次为什么没解决、根因分析，确认后最小化重写）
4. 自验：删掉所有下游保护，链路仍正确 = 根治
5. 收尾：根据任务更新相关文档（模块注册表.md、代码目录.md、硬编码收集.md 等）。**必须使用以下格式输出**：

   **✅ 收尾报告**
   | 文件 | 操作 | 状态 |
   |------|------|------|
   | 模块注册表.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |
   | 代码目录.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |
   | 硬编码收集.md | 更新 | ✅ 已完成 / ⏭️ 无变更 |

### 其他流程

1. 开始根据任务进行操作，按上方扩展规则扩展 TodoWrite 步骤
2. 收尾：根据任务更新相关文档。**必须使用以下格式输出**：

   **✅ 收尾报告**
   | 文件 | 操作 | 状态 |
   |------|------|------|
   | [涉及的文件] | [操作类型] | ✅ 已完成 / ⏭️ 无变更 |

## 原则一：禁止幻觉

- **没读过的文件**：不能凭猜测引用其内容，先读再说
- **使用接口前必须验证**：调用任何函数/接口前，必须先读过其定义或文档，不管是否自认为知道
- **结论必须标注来源文件路径或代码行号**：给出技术结论时，必须注明"读的哪个文件的第几行"或"哪段代码"。无法标注来源的结论必须注明"未经验证，需确认"
- **禁止伪造来源**：不能说"根据 XX 文件"但实际没读过该文件

---

## 原则二：对齐理解，决定权归用户

- **开始前必须对齐**：收到任务后，先复述理解要点，用户确认后再执行
- **功能要不要做、做成什么样由用户决定；怎么实现由 AI 设计**：用户提出设计想法时，AI 总结吸收，完善为完整设计。不合理时必须指出，不能盲从
- **AI 可自行决定的范围**：仅限不影响功能行为和用户可见结果的改动（如变量命名、内部数据结构选择、代码组织方式）。其余一律问用户
- **收到纠正时复述完整模型**：不能只复述被纠正的那一点，把关联的设计决策一起列出让用户确认

---

## 原则三：效率原则

- **AI 写入的文件不重读**：本轮对话中自己写的文件，不读，内容就是自己写的
- **已读过的文件不重读**：本轮对话中已读过的文件，不再读，信任上下文
- **从未读过的文件必须读**：不能凭上下文推测未读文件的内容
- **压缩应对**：检测到上下文压缩时，立即停下 → 重读当前步骤所需文件 → 从 TodoWrite 确认进度 → 继续
- **关联修改一并完成**：当前修改导致其他文件出现错误、接口不匹配、或引用失效的，必须在本次任务中一并修复，不得遗留到后续任务
- **变更前告知用户**：创建或修改文件前先告知用户意图，用户确认后再执行。例外：build、TodoWrite、读取文件、用户说"直接做"后的当次执行

---

## 原则四：禁止硬编码

报错能发现问题，硬编码会藏住问题。数据链断裂是最危险的 bug。

### 五类硬编码

| 类型 | 症状 | 正确做法 |
|--|--|--|
| **值替代** | 有定义常量却直接写字面量 | 引用定义常量 |
| **链路断裂** | 有数据源却没追到，手动拼输出 | 调用模块接口或引用已有映射 |
| **默认值掩盖** | 源头模块未实现，用 `or` 兜底 | 保持报错，不掩盖缺失 |
| **逻辑复制** | 别的模块已有相同逻辑，重写一遍 | 调用已有函数 |
| **字符串契约断裂** | 同一个字符串在多个文件各写字面量 | Defs 定义常量，所有文件引用 |

### 判定标准

定义文件是数据的唯一归属地，业务代码必须引用定义：
- 数据有来源却绕过来源 → 硬编码
- 值无其他来源、仅服务当前文件局部逻辑 → 不算硬编码

### 常见错误示例

```lua
-- ❌ 值替代：Defs 定义了武器类型，却直接写字符串
if weaponType == "sword" then ...

-- ✅ 引用定义
if weaponType == WeaponDefs.SWORD then ...
```

```lua
-- ❌ 链路断裂：伤害计算已有模块，手动重写公式
local damage = atk * 2.5 - def * 0.8

-- ✅ 调用已有接口
local damage = CombatCalc.calculateDamage(attacker, defender)
```

```lua
-- ❌ 默认值掩盖：模块未实现就用 or 兜底
local name = ItemDefs.getName(itemId) or "未知物品"

-- ✅ 保持报错，让问题暴露
local name = ItemDefs.getName(itemId)
```

```lua
-- ❌ 字符串契约断裂：action 名在客户端和服务端各写一遍
-- client.lua
sendRequest("buy_item", params)
-- server.lua
actionHandlers["buyItem"] = Handler.handleBuyItem  -- 拼写不一致！

-- ✅ 统一定义
-- shared/Actions.lua
Actions = { BUY_ITEM = "buy_item" }
-- 双方引用 Actions.BUY_ITEM
```

### 引用尚不存在时的处理

- **保持引用让 build 报错**，不准临时硬编码
- 模块已存在 → 补定义
- 模块未开发 → 保持报错，记入 `docs/AI协作手册/AI处理文档/硬编码收集.md`

### 数据定义规则

- **数据依赖消费者**：消费者已存在 → 从消费者读取的字段反推结构，程序化生成
- **手动逐条定义仅限少量无规律数据**：单次不超过 10 条
- **批量数据的数量是设计决策**：须与用户确认后写入 `_CONTRACT`，配套 `verify()` 在 require 时断言
- **程序化生成**：循环和公式实现，配合 `_CONTRACT` + `verify()` 验证链。人工输入仅限名称、描述等无法推导的创意数据

|------|---------|

### 内容溢出与布局空间分配

核心原则：**容器适配内容，不是内容适配容器**。

**禁止项**：

| 禁止行为 | 原因 |
|----------|------|
| 静默裁切 | 文本/数值被截断玩家看不到，说明布局有问题，必须修正 |
| 缩小字号 | 字号由体系定义，缩小破坏视觉层次 |

**文本溢出**：超出容器宽度时换行，高度自适应。省略号仅限非关键长文本（背景描述、简介），核心信息禁止省略。

**数值溢出**：数值不能换行（拆成两行无法阅读），必须在创建布局时就确保空间足够。

#### 布局空间分配（AI 创建横向排列 UI 前必须执行）

这是 AI 最常犯的布局错误：用当前测试数据验证布局，测试时正常，数据增长后溢出。

**5 步流程**：

1. **列出所有元素**：明确一行中有哪些元素（标签、数值、按钮、输入框等）
2. **确定数据范围**：每个动态内容的最大值是多少？不知道就问用户
3. **用最大值计算空间需求**：用最大数据范围验证布局，不是用当前测试值
4. **固定元素先占位**：按钮、图标等固定宽度元素先分配空间，动态内容拿剩余空间
5. **验证剩余空间是否足够**：放不下 → 调整布局方案（减少每行元素、改纵向、拆多行），禁止硬塞

**示例 — 角色属性面板**：

```
一行元素：[标签] [数值] [−按钮] [+按钮] [输入框]

数据范围：
  标签：最长 4 汉字（"暴击伤害"）≈ 64px
  数值：最大 9999999（7 位）≈ 63px
  按钮：各 36px 固定
  输入框：80px 固定

最大总需求：64 + 63 + 36 + 36 + 80 + 间距(8×4) = 311px
容器宽度 350px → 311 < 350 ✅

容器只有 280px → 311 > 280 ❌
→ 按钮和输入框换到第二行
```

### 页面独立性

所有 UI 页面独立实现。页面之间不共用组件，除非布局结构完全一致。

**判定标准**：两个页面的某个区域，HTML 结构、样式属性、数据绑定方式都相同 → 可共用组件。只是"看起来像"但结构不同 → 各自实现。

local DEFAULTS = {

    cardHeight = 160,
    cardRadius = 8,

    nameSize = 14,

**第 2 步：widget 构造引用 DEFAULTS**

```lua
local card = UI.Panel {
    elementId = PAGE_ID .. ".card",
    width = DEFAULTS.cardWidth,
    height = DEFAULTS.cardHeight,
    borderRadius = DEFAULTS.cardRadius,
    backgroundColor = DEFAULTS.cardBgColor,
}
```

**不写字面量**，所有框架属性从 DEFAULTS 取。

```lua
function Widget.Register()
    UIEditor.RegisterTemplate(tplId, name, defaults, pageId)
end

**第 4 步：创建元素后注册元素**

```lua
UIEditor.RegisterElement(eid, pageId, tplId, widget)
```

**第 5 步：内容属性直接写在元素上**

```lua
label.text = playerName
avatar.backgroundImage = avatarUrl
```

**框架属性 vs 内容属性**：

| 类型 | 放哪里 | 举例 |
|--|--|--|
| 框架属性 | DEFAULTS | 尺寸、颜色、间距、圆角、字号 |
| 内容属性 | 直接写在元素上 | text、backgroundImage、数据绑定 |

# 服务器框架规则

> 低频场景（进入游戏流程、断线重连、sessionVersion 守卫）见 `docs/策划文档/服务器框架蓝图.md`。

### 架构核心

只有一种架构：**服务端权威架构**。所有游戏逻辑都运行在服务端。

- **服务端权威**：服务端持有权威状态，执行所有状态变更、校验、计算。客户端代码运行在用户设备上，可以被篡改，只有服务端计算才是可信的
- **客户端只做两件事**：发送操作请求、展示服务端返回的数据
- **shared 放定义和纯查询**：Defs 定义 + 纯查询函数（只读数据，无副作用）
- **状态变更、校验、权威计算放 server/**：客户端可被篡改，只有服务端计算才可信
- **单人游玩**：玩家数量为 1，支持异步联机玩法（排行榜、异步对战等）
- **多人游玩**：多人实时同服
- 两者代码组织完全一致，区别仅在于玩家数量和网络层配置


### 目录结构与文件归类

```
scripts/
├── main.lua          # 模式路由（判断单人/多人，加载对应入口）
├── network/          # 网络基础设施
│   ├── Server.lua
│   ├── Client.lua
│   ├── ServerDispatcher.lua    # 服务端数据分发层
│   └── ClientDispatcher.lua    # 客户端数据分发层
├── shared/           # 双端共用：定义 + 纯查询
│   └── [模块名]/
│       └── Defs.lua            # 该模块的定义
├── server/           # 仅服务端：状态 + 处理 + 持久化
│   └── [模块名]/
└── client/           # 仅客户端：请求发送 + UI 展示
    └── [模块名]/
```

**文件归类判定**：代码按"谁运行"分目录——这段代码需要可变状态吗？需要在哪端运行？

| 目录 | 运行端 | 放什么 | 判定标准 |
|--|--|--|--|
| `shared/模块名/` | 双端都加载 | Defs 定义 + 纯查询函数 | 只读数据，禁止状态变更、校验、权威计算 |
| `server/模块名/` | 仅服务端 | State + Handler + 权威计算 | 持有可变状态，执行所有状态变更、校验、权威计算 |
| `client/模块名/` | 仅客户端 | 请求发送 + UI 展示 | 只展示服务端发来的数据，不持有游戏状态 |
| `network/` | 按文件区分 | 网络基础设施（连接、分发层） | 传输层，不含业务逻辑 |

**常见错误**：
- ❌ 在 `client/` 中计算伤害、判断是否可购买 → 应在 `server/` 中计算
- ❌ 在 `server/` 中写 UI 代码 → 应在 `client/` 中
- ❌ 在 `shared/` 中放校验或权威计算 → `shared/` 只放定义和纯查询，状态变更、校验、权威计算放 `server/`
- ❌ 把模块定义放在 `shared/Defs/` → 应放在 `shared/模块名/Defs.lua`，跟随模块

### 场景同步

引擎内置 Scene Replication，服务端创建的 REPLICATED 节点自动同步到所有客户端（位置、旋转、缩放、节点变量）。

**数据归属判断**：

| 数据类型 | 走哪条路 | 举例 |
|--|--|--|
| 空间/视觉状态 | REPLICATED 节点 | 角色位置、朝向、动作状态、棋子位置 |
| 游戏数据 | StateUpdate（markDirty） | 背包、属性、货币、关卡进度 |
| 操作请求 | 请求/响应 | 购买、使用道具、出牌 |

**规则**：
- 服务端：游戏实体用 `REPLICATED` 创建，渲染组件不创建（服务端无图形）
- 客户端：本地内容（相机、光照、环境）必须显式指定 `LOCAL`，否则默认 REPLICATED 会和服务端冲突
- 空间数据优先通过 REPLICATED 节点同步，避免通过 StateUpdate 重复传输
- 大量实体时用 `NetworkPriority` 控制更新频率，客户端上报观察者位置

### 两条通道

| 通道 | 方向 | 承载内容 |
|--|--|--|
| 请求/响应 | 客户端 → 服务端 → 客户端 | 操作请求，响应只含 `{success, reason?}` |
| 状态推送（StateUpdate） | 服务端 → 客户端 | 全量展示数据 |

两条通道完全分离。Handler 返回值禁止携带展示字段。StateUpdate 在 response 之前发送（WebSocket 保序）。


### 数据分发层

两端分发层构成对称的传输管道，业务代码不感知拆分细节：

```
业务流程 → ServerDispatcher（拆分、缓存、发送）→ 网络 → ClientDispatcher（收齐、组装）→ 页面
```

**ServerDispatcher**：服务端唯一的数据推送出口。接收完整数据包 → 按模块拆分传输 → 发送。per-UID 缓存最近一次全量数据，重连或补齐时从缓存重发。`markDirty` 触发增量推送时同步更新缓存。

**ClientDispatcher**：客户端唯一的数据接收入口。收齐拆分消息 → 组装完整数据包 → 分发到页面。2 秒未收齐 → 向服务端请求补齐。

**传输限制**：单条消息不能超过 60KB（TapTap 容器限制），安全阈值 50KB。超限模块继续拆分为分片消息。

**对 Handler 透明**：Handler 只管业务逻辑 + `markDirty`，推送由 `markDirty` 内部自动交给分发层完成，Handler 不需要知道分发层的存在。

### Handler 五步流程

每个 Handler 遵循固定模式：

```lua
function Handler.handleXxx(uid, params)
    -- 1. 读取实时表
    local data = SaveManager:getTable(uid, "moduleName")
    -- 2. 校验
    if not canDoXxx(data, params) then
        return { success = false, reason = "条件不满足" }
    end
    -- 3. 修改实时表（直接修改引用）
    data.someField = newValue
    -- 4. 标记变更（即时推送 + 存档防抖）
    SaveManager:markDirty(uid, "moduleName")
    -- 5. 返回结果（不携带展示数据）
    return { success = true }
end
```


### 存档系统

#### 核心概念

- **单表模型**：每个 scores key 一张实时表，没有备份表
- **统一防抖**：全局一个计时器（默认 5 秒），`markDirty` 确保计时器在跑（已在跑则跳过）
- **markDirty 双职责**：即时推送该模块的展示数据 + 启动存档防抖计时器

#### 模块注册

需要持久化数据的模块通过 `register` 接入存档系统。新增模块只需 register + getDefault，存档系统零改动。

```lua
SaveManager:register("moduleName", {
    key = "save_world",                           -- 可选，省略则归入默认 key save_data
    getDefault = function() return { ... } end,   -- 必填，子表默认值
    onLoad = function(data) ... end,              -- 可选，加载后的字段迁移
    onCleanup = function(uid) ... end,            -- 可选，会话清理时释放运行时状态
})
```

| 字段 | 必填 | 说明 |
|--|--|--|
| `key` | 可选 | 归入哪个 scores key。省略则归入 `save_data` |
| `getDefault` | 必填 | 返回模块子表的默认值。首次登录时用此值初始化。新增字段无需迁移——模板匹配自动补全 |
| `onLoad` | 可选 | 云端数据加载到实时表后调用。用于字段迁移（重命名、拆分、合并），必须幂等 |
| `onCleanup` | 可选 | 会话清理时调用。用于释放运行时状态（战斗状态、匹配状态等不存云的数据） |

#### 数据读写

| 操作 | API | 说明 |
|--|--|--|
| 读写自己的子表 | `SaveManager:getTable(uid, "moduleName")` | 返回引用，直接读写 |
| 读其他模块的子表 | `SaveManager:getTable(uid, "otherModule")` | 只读，不写别人的子表 |
| 标记变更 | `SaveManager:markDirty(uid, "moduleName")` | 即时推送 + 存档防抖 |

**常见错误**：
- ❌ 写别人的子表 — 跨模块只读
- ❌ 修改数据但没调 markDirty — 数据不会被推送，也不会被提交
- ❌ 货币通过独立 API 管理 — 货币是模块数据的普通字段，通过 getTable + markDirty 读写
- ❌ `_meta` 作为业务字段 — 这是存档系统保留名，业务模块禁止使用

#### Key 组织

所有 key 内部按模块名分子表，模块名即命名空间。`_meta` 是存档系统保留名。

```lua
-- 默认 key（未指定 key 的模块归入此处）
ScoreSet(uid, "save_data", {
    _meta = { lastSaveTime = ... },   -- 系统保留
    character = { ... },               -- character 模块
    item = { ... },                    -- item 模块
})

-- 自定义 key（数据量大的模块单独分 key）
ScoreSet(uid, "save_world", {
    worldInstance = { ... },
})
```

**分 key 规则**：未指定 key → 归入 `save_data`；数据量大 → 注册时指定自定义 key；单 key 接近 1MB → 必须拆分。

### 客户端数据消费

页面只从 `ClientDispatcher` 读取展示数据，禁止从其他来源获取。

```lua
-- 订阅模块数据变化（页面初始化时）
ClientDispatcher.subscribe("moduleName", refresh)

-- 读取当前数据（首次可能返回 nil，必须处理）
local data = ClientDispatcher.get("moduleName")

-- 取消订阅（页面销毁时，必须调用）
ClientDispatcher.unsubscribe("moduleName", callback)
```

**禁止**：
- params 传展示数据给其他页面（展示数据只从 ClientDispatcher 获取）
- 客户端计算游戏逻辑（逻辑只在服务端执行）
- response 回调中读取展示字段（回调中只处理失败提示）

### response 回调规则

| response 结果 | 客户端行为 |
|--|--|
| `success = true` | 无需操作（推送先于 response 到达，UI 已刷新） |
| `success = false` | 弹提示（Toast），显示 `reason` |

response 回调中不读取展示数据、不刷新 UI、不做逻辑计算。唯一用途是处理失败提示。

### 通知类型

框架中所有客户端通知分为两类，均根据游戏 UI 风格创建：

- **提示（Toast）**：顶部居中，一行文字，3 秒自动消失，点击直接关闭，不阻断操作
- **弹窗（Modal）**：屏幕居中 + 半透明遮罩，标题 + 说明 + 按钮，阻断背后操作，必须玩家点按钮关闭

### 模块规范

- 一个文件 = 一个功能，返回 table
- 模块通过 `Init(上游模块引用)` 建立连接，`main.lua` 按依赖顺序调用 `Init`
- 上游模块通过 Observable mixin 获得 `on/emit` 能力，下游在 `Init` 时订阅
- 数据定义放在 `shared/模块名/` 下，通过模块公开接口暴露
- 代码文件头部一行注释说明职责
- Handler 只管业务逻辑 + `markDirty`，推送由 `markDirty` 内部自动完成，Handler 不需要知道分发层的存在

### 状态分层

服务端玩家状态分三类：

| 状态类型 | 存储位置 | 存云 | 生命周期 | 举例 |
|--|--|--|--|--|
| 持久状态 | SaveManager 实时表 | ✅ | 跨会话持久 | 背包、属性、货币 |
| 运行时状态 | 模块自己的 state | ❌ | 随会话销毁 | 战斗状态、匹配状态 |
| 派生数据 | 不存储，现算 | ❌ | 每次推送时计算 | 战力值、解锁状态 |

三类状态均通过 `buildDisplayData` 发送给客户端。运行时状态随会话销毁，模块通过 `onCleanup` 释放。

### buildDisplayData 规范

每个模块负责提供自己的展示数据，供 `markDirty` 增量推送和全量推送调用：

```lua
function Module.buildDisplayData(uid)
    local data = SaveManager:getTable(uid, "moduleName")
    local rt = runtimeState[uid]  -- 模块自己的运行时状态
    return {
        -- 持久状态（从实时表读）
        gold = data.gold,
        items = data.items,
        -- 运行时状态
        inCombat = rt and rt.inCombat or false,
        -- 派生数据（现算）
        totalPower = calcPower(data),
    }
end
```

**三类数据来源**：持久状态（SaveManager 实时表）、运行时状态（模块内部 state）、派生数据（现算）。

### 页面生命周期

页面遵循 **创建 → 订阅 → 清理** 三步：

| 步骤 | 说明 |
|--|--|
| 创建 | 构建 UI 结构，定义 refresh 函数 |
| 订阅 | `ClientDispatcher.subscribe` + 立即调用一次 refresh（处理数据已到达的情况） |
| 清理 | `_cleanup` 中取消订阅，PageRouter 自动调用 |

### 守卫链

Server.lua 作为请求网关，在请求到达 Handler 前执行三层守卫：

1. `SessionManager.GetUserId(connection)` → nil → 丢弃（未认证）
2. `SaveManager:isLoaded(userId)` → 否 → 回复"数据加载中"
3. `SaveManager:checkRateLimit(userId)` → 超限 → 回复"操作过于频繁"

守卫全部通过后路由到 `actionHandlers[action](userId, params)`。

### Headless Mock

服务端无图形子系统（Headless 模式）。共享代码或工具库中引用 `graphics`/`console`/`debugHud` 会崩溃。

**规则**：
- Server.lua 最顶部、所有 `require` 之前，注入 Mock
- Mock 范围：`graphics`、`console`、`debugHud` 三个全局对象
- 共享代码引用 `graphics.width` 等属性时，Mock 提供默认返回值

```lua
-- Server.lua 最顶部（所有 require 之前）
if GetGraphics() == nil then
    local mockGraphics = {
        SetWindowIcon = function() end,
        GetWidth = function() return 1920 end,
        GetHeight = function() return 1080 end,
    }
    function GetGraphics() return mockGraphics end
    graphics = mockGraphics
    console = { background = {} }
    function GetConsole() return console end
    debugHud = {}
    function GetDebugHud() return debugHud end
end
```

### 云端约束

| 约束 | 数值 | 说明 |
|--|--|--|
| 提交频次 | 300 次 BatchCommit / 60s | 单服务端实例独享 |
| 单次操作数 | 1000 条 / 单次 BatchCommit | |
| 吞吐量 | 48MB / 分钟 | 读写总量 |
| 单 key 上限 | 1MB | 超限时必须拆 key |
| 频率限制 | 2 次请求/秒/玩家 | 守卫链拦截 |

**编码影响**：
- 单模块数据接近 1MB → 注册时指定独立 key，拆分存储
- 防抖周期默认 5 秒 → 提交预算 12 次/分钟，远低于 300 次上限
- 所有 key 合并进同一个 BatchCommit，不单独提交
- 频率限制 2 次/秒/玩家 → 客户端操作节流，服务端守卫链拦截超限请求

### 相机跟随 REPLICATED 节点

REPLICATED 节点在客户端自动挂载 `SmoothedTransform`（引擎行为），已内置平滑插值。

**规则**：客户端相机跟随 REPLICATED 节点时，直接使用 `node.worldPosition`，禁止额外 lerp。额外插值会叠加引擎插值，造成延迟或抖动。

```lua
-- ✅ 正确：直接跟随
cameraNode.position = playerNode.worldPosition + Vector3(0, 5, -10)

-- ❌ 错误：多余的 lerp
cameraNode.position = Lerp(cameraNode.position, playerNode.worldPosition + offset, dt * 5)
```

# 策划检查规则

> 主 AI 读取本文件，根据当前策划内容生成策划检查清单（放入策划文件夹）。
> 主 AI 最了解自己写的内容，因此由主 AI 深度分析策划后生成针对性的检查条目，并自行逐条检查。

## 提取原则

### 1. 穷尽提取

策划文档中每一条可判定的信息都必须提取，不能遗漏。宁可多列也不能少列。

### 2. 拆细，不要合并

一个功能点必须拆成多条，每条只描述一件事。

**拆分方法**：对策划文档中的每句话，问自己"这句话包含几个可独立验证的点？"——每个点单独一条。一句话通常能拆出 3-8 条。如果你一句话只列了 1 条，大概率拆得不够细。

### 3. 隐含要求也要提取

策划文档没有明说，但实际工作中必然涉及的内容，也要列出。不要只提取策划明确写出的内容。

### 4. 每条必须可判定

能明确判断"做了/没做"或"对/不对"。不能列"体验要好"这种模糊描述。

### 5. 只检查策划内容

检查范围限于策划文档本身的设计完整性和逻辑自洽性。不检查编码实现，除非策划文档中明确包含了编码相关的描述（如技术方案、接口定义等）。

---

## 生成步骤

### 第 1 步：确认检查模式

询问用户：该策划内容是否有对应的规范性文档（创建规范、模板、大纲等）？

- **有** → 请用户提供规范文档路径（及大纲路径，如有）→ 第 2B 步（规范产出物）
- **没有** → 第 2A 步（独立策划）

> 如果主 AI 在同一会话中刚按规范生成了产出物，已知规范路径，可跳过询问直接进入第 2B 步。

### 第 2A 步：独立策划 — 识别类型并深度分析

**识别策划类型**（一份策划可能跨多个类型，全部识别出来）：

| 类型 | 分析侧重 |
|------|---------|
| 系统/功能设计 | 状态流转完整性（状态机每条路径）、边界条件（上限/下限/溢出）、系统间交互（数据流转、触发关系）、异常处理（失败/中断/资源不足）、生命周期（创建/存续/销毁）、初始与默认状态 |
| 剧情/叙事 | 分支完整性（每个选择是否有结果，有无死胡同）、角色一致性（性格/动机/行为前后是否矛盾）、时间线一致性（事件顺序是否矛盾）、伏笔与回收（铺设的线索是否有回收）、对话触发条件（前置条件是否明确）、情感节奏（起伏是否合理） |
| 数值/经济 | 产出与消耗平衡（资源循环是否闭合）、成长曲线（节奏/上限/拐点）、公式变量来源（每个变量从哪来）、边际效益（递减/递增设计）、极端值推演（最高/最低/最快/最慢场景）、选项平衡（是否有明显最优解或无用选项） |

## 提取原则

### 1. 穷尽提取

策划文档中每一条可判定的实现要求，以及代码中每一个可验证的行为，都必须提取，不能遗漏。宁可多列也不能少列。

### 2. 拆细，不要合并

一个功能点必须拆成多条，每条只描述一件事。

**拆分方法**：对策划中的每句话或代码中的每个逻辑分支，问自己"这里包含几个可独立验证的点？"——每个点单独一条。一句话通常能拆出 3-8 条。如果你一句话只列了 1 条，大概率拆得不够细。

### 3. 隐含要求也要提取

策划没有明说但编码时必然涉及的内容，也要列出。不要只提取明确写出的内容。

### 4. 每条必须可判定

能明确判断"做了/没做"或"对/不对"。不能列"代码质量要好"这种模糊描述。

## 工作流

```
AI 编写初始 UI 代码（每个元素带 elementId）
       ↓
开发者运行游戏 → 按 INS 键进入编辑模式（仅开发者 UID 可用）
       ↓

       ↓
管理样式模板：新建/关联/解除关联
       ↓
点击"导出" → 修改通过 print() 输出到日志
       ↓
AI 读取日志中的导出 JSON → 直接分析并更新对应源码
```

### AI 更新规则

1. **读取方式**：导出内容通过 `print("[UIEditor] 导出内容: " .. jsonStr)` 输出到 `/opt/log/dev/user_script.log`，AI 从日志中读取
2. **直接修改源码**：AI 根据导出内容，定位源代码中对应 elementId / templateId 的属性字面量，直接修改源码中的值
3. **不展示分析表**：AI 读取导出 JSON 后，直接分析并修改源码，不向用户展示分析表或对照表
4. **全量快照**：每次导出包含当前所有修改（不是增量），AI 以导出内容为唯一真相源
5. **颜色格式转换**：导出 JSON 中颜色为 `#RRGGBB` 或 `#RRGGBBAA` 字符串，AI 写入源码时需转换为 `{r, g, b, a}` table 格式

## 模块详细规格

### 1. UIEditor.lua — 主控制器

#### 职责

- 初始化所有子模块、注册全局事件
- 管理编辑模式的进入/退出
- 提供 ApplyHook 供 PageRouter 调用
- 提供模板/元素注册接口供页面模块调用

#### 初始化

```lua
UIEditor.Init(rootContainer)
```

- 接收 UI 根容器引用
- 从 `settings.json` 的 `ui_editor.developer_uids` 读取开发者 UID 列表
- 注册 `KeyDown` 和 `Update` 全局事件
- 只初始化一次（`initialized_` 标志位）

#### 开发者身份验证

- 通过 `network:GetServerConnection().identity["user_id"]` 获取当前用户 UID
- 只有 UID 在 `developer_uids` 列表中的用户才能进入编辑模式

#### 编辑模式

- **进入** `EnterEditMode()`：
  1. 初始化 SelectionManager（传入 rootContainer）
  2. 设置选中/取消选中回调
  3. 初始化 PropertyPanel（传入 rootContainer）
  4. 创建全屏透明拦截层 overlay（zIndex=9998）
  5. 创建顶部指示栏（zIndex=9999），含"编辑模式"文字、"导出"按钮和"开发者工具"按钮

- **退出** `ExitEditMode()`：

### 2. ElementStore.lua — 数据层

#### 职责

- 模板注册与修改追踪
- 元素注册、覆盖值管理
- 子元素属性路径系统（`@{path}.{prop}`）
- Widget 重建机制（属性变更需重新初始化时）
- Label 自动计算值剥离（autoHeight_/autoMinHeight_）
- RegisterAllWidgets 自动注册与验证
- 应用修改到 widget 树

#### 数据结构

**模板注册表** `templates_[templateId]`：

| 字段 | 类型 | 说明 |
|------|------|------|
| name | string | 可读名称 |
| properties | table | 框架属性键值对（含 `@{path}.{prop}` 子元素属性） |
| pageId | string/nil | 所属页面 ID（用于按页面过滤模板列表） |
| _modified | boolean | 是否被修改过（导出用） |

**元素注册表** `elements_[elementId]`：

| 字段 | 类型 | 说明 |
|------|------|------|
| pageId | string | 所属页面 |
| templateId | string/nil | 关联的模板 ID |
| overrides | table | 对模板属性的覆盖值 |
| properties | table | 独立属性（无模板时） |
| widgetRef | table/nil | widget 引用（页面销毁后为 nil） |
| _originalTemplateId | string/nil | 注册时的原始模板 ID（导出时判断变化） |
| treePath | number[]/nil | 自动注册元素的树路径索引 |

#### 子元素属性路径系统

**格式**：`@{childIndex}.{prop}` 或 `@{idx1}.{idx2}.{prop}`（嵌套）

**示例**：
- `@1.fontSize` → 第 1 个子元素的 fontSize
- `@2.3.fontColor` → 第 2 个子元素的第 3 个子元素的 fontColor

**判断规则**：`@` 后第一个字符是数字 → 子元素属性

**路径解析**：
1. `parseChildProp("@1.fontSize")` → path={1}, realProp="fontSize"
2. `resolveChildByPath(widget, {1})` → widget.children[1]

**在模板中使用**：
- 注册模板时，`@{path}.{prop}` 格式的属性只有当 `realProp` 是框架属性（`PropertyDefs.IsFramework()`）时才保留
- 修改模板属性时，如果是子元素属性，会通过路径定位到子 widget 并调用 `SetStyle()`
- 同步到关联元素时，检查该元素是否有此属性的覆盖值

**公开工具函数**（供 PropertyPanel 等模块使用）：

| 接口 | 说明 |
|------|------|
| `IsChildProp(prop)` | 判断属性名是否为子元素属性（`@` 开头且后跟数字） |
| `ParseChildProp(prop)` | 解析子元素属性，返回 `path, realProp` |
| `ResolveChildByPath(widget, path)` | 根据路径索引数组定位子 widget |

#### Widget 重建机制

某些属性（如 Label 的 `whiteSpace`）仅在 `Init()` 中初始化内部状态，`SetStyle()` 无法更新。对这些属性，需要用完整 props 重建 widget 实例。

**需要重建的属性集合** `REINIT_PROPS`：

| 属性 | 原因 |
|------|------|
| `whiteSpace` | Label 的 multiline_/autoHeight_/autoWidth_/alignSelf 仅在 Init 设置 |

**重建流程** `rebuildWidget(widget)`：
1. 找到 widget 在父容器中的位置（index）
2. 浅拷贝当前 props
3. 剥离 Label.Init 自动计算的值（`autoHeight_` → `props.height = nil`，`autoMinHeight_` → `props.minHeight = nil`）
4. 从父容器移除旧 widget
5. 用完整 props 调用 `UI[className]()` 创建新 widget（走正规 Init 路径）
6. 剥离新 widget 中 Init 自动计算并写入 props 的值
7. 插回原位置
8. 更新 `elements_` 中的 `widgetRef` 引用
9. 释放旧 widget 的 Yoga 节点

**辅助函数**：

| 函数 | 说明 |
|------|------|
| `setWidgetProp(widget, prop, value)` | 调用 `SetStyle()` 后检查是否需要 rebuild |
| `clearWidgetProp(widget, prop)` | 将 `widget.props[prop]` 置 nil 后 rebuild（因为 `SetStyle({[k]=nil})` 在 Lua 中是空操作） |

#### Label 自动计算值剥离

Label 组件在 `Init()` 中会自动计算 `height` 和 `minHeight` 并写入 `props`。这些值不是用户设置的，如果不剥离会导致：
- PropertyPanel 误以为是用户设置的值（删除属性后又重新出现）
- 新 widget 实例误以为是用户显式设置的值

**剥离逻辑**：在 `RegisterElement()` 和 `rebuildWidget()` 中检查 `widget.autoHeight_` 和 `widget.autoMinHeight_` 标志位，为 true 则将对应 `props` 字段置 nil。

#### 属性优先级（从高到低）

1. 覆盖值（`overrides`）
2. 模板属性（`templates_[templateId].properties`）
3. 独立属性（`properties`）
4. widget 当前 props 值

通过 `GetEffectiveValue(elementId, prop)` 按此顺序查找。

#### RegisterAllWidgets

```lua
ElementStore.RegisterAllWidgets(pageId, pageWidget)
```

- 自上而下递归遍历整棵 widget 树
- 跳过编辑器自身的 widget（id 以 `__editor_` 开头）
- 对每个 widget：
  - 有 `elementId` prop → 用该 elementId 注册
  - 无 `elementId` → 生成 `{pageId}.auto_{pathStr}` 作为 elementId，**打印警告**
- 遍历结束后打印总计缺失 elementId 的数量
- **验证流程**：build 后检查日志中是否有 `⚠ 缺少 elementId` 警告，有则补上 elementId 直到 0 警告

#### ApplyToPage

```lua
ElementStore.ApplyToPage(pageId, pageWidget)
```

- 遍历该 pageId 下所有已注册元素
- 通过 `treePath`（自动注册元素）或 `FindById`（手动注册元素）定位 widget
- 调用 `ApplyToWidget()` 应用修改

#### ApplyToWidget

对单个 widget 应用修改，合并顺序：模板属性 → 独立属性 → 覆盖值。使用 `splitChildProps()` 分离子元素属性，分别调用 `SetStyle()`。

#### 模板操作

| 接口 | 说明 |
|------|------|
| `RegisterTemplate(id, name, props, pageId)` | 注册模板，只保留框架属性，pageId 指定所属页面 |
| `GetTemplate(id)` | 获取模板数据 |
| `GetTemplateIds()` | 获取所有模板 ID（排序） |
| `GetTemplateIdsByPage(pageId)` | 获取指定页面的模板 ID 列表（排序） |
| `SetTemplateProperty(id, prop, val)` | 修改模板属性，同步所有关联元素（无覆盖的） |
| `RemoveTemplateProperty(id, prop)` | 从模板定义中删除某属性，同步所有关联元素（无覆盖的）回退默认 |
| `CreateTemplate(elementId, widget)` | 从当前 widget props 提取框架属性创建新模板，自动关联 |
| `GetModifiedTemplates()` | 获取所有 `_modified=true` 的模板（导出用） |

#### 元素操作

| 接口 | 说明 |
|------|------|
| `RegisterElement(id, pageId, templateId, widgetRef)` | 注册元素（已存在则更新 widgetRef 并重新应用） |
| `UnregisterPage(pageId)` | 页面销毁时清除 widgetRef（保留修改数据） |
| `GetElement(id)` | 获取元素数据 |
| `FindElementId(widget)` | 通过 widget 引用反查 elementId |
| `SetOverride(id, prop, val)` | 设置覆盖值（实时更新 widget） |
| `ClearOverride(id, prop)` | 清除覆盖值，回退到模板值 |
| `SetProperty(id, prop, val)` | 设置属性（有模板→走覆盖，无模板→直接记录） |
| `RemoveProperty(id, prop)` | 移除元素属性（覆盖值+独立属性），回退到模板值或清除 |
| `GetEffectiveValue(id, prop)` | 获取属性生效值（按优先级） |
| `HasOverride(id, prop)` | 检查是否有覆盖值 |
| `LinkElement(id, templateId)` | 关联到模板（清空覆盖，应用模板属性） |
| `UnlinkElement(id)` | 解除关联（模板属性+覆盖值复制为独立属性） |
| `BatchLink(elementIds, templateId)` | 批量关联元素到模板 |
| `AutoRegister(pageId, pageRoot, widget)` | 自动注册未标记的元素（计算树路径） |
| `GetModifiedElements()` | 获取所有有修改的元素（导出用） |
| `GetLinkedElements(templateId)` | 获取模板关联的所有元素 ID |
| `GetSiblingElements(elementId)` | 获取同页面同类型兄弟元素 |
| `GetElementTemplateId(elementId)` | 获取元素关联的模板 ID |

### 3. SelectionManager.lua — 选中管理

#### 命中检测

SelectionManager.HitTest(x, y) → { elementId, widget } | nil

### 4. PropertyPanel.lua — 属性面板

#### 职责

- 显示选中元素的可编辑属性
- 根据属性类型创建对应控件
- 模板操作区（关联/解除/新建）
- 面板拖拽
- "添加属性"弹窗
- 长按 +/- 按钮的重复触发

#### 面板结构

```
Panel (position=fixed, zIndex=10000, 宽320, 最大高500)
├── 标题栏（可拖拽，显示 elementId 或 widget 类型，含关闭按钮）
└── ScrollView（滚动内容区）
    ├── 模板操作区
    ├── 属性行列表（按分类分组）
    └── "+ 添加属性"按钮
```

#### 模板操作区

根据元素的模板状态显示不同 UI：

| 状态 | 显示内容 | 可执行操作 |
|------|---------|-----------|
| 未注册到 store | widget 类型信息 | — |
| 已关联模板 | 模板名称 | "解除"按钮 |
| 未关联模板 | "独立元素" + **当前页面的**模板列表 + "+ 新建模板"按钮 | 关联已有模板，或从当前 widget 新建模板 |

> 模板列表通过 `GetTemplateIdsByPage(PageRouter.current())` 过滤，只显示属于当前页面的模板。

#### 属性控件类型映射

| PropertyDefs TYPE | 控件 | 交互方式 |
|-------------------|------|---------|
| NUMBER | +/- 按钮 + 数值显示 | 点击 +/- 调整，长按连续增减 |
| SIZE | +/- 按钮 + 数值显示 + px/% 切换按钮 | 同 NUMBER，额外可在 px 和 % 之间切换 |
| ENUM | 下拉选择（点击弹出选项列表） | 显示中文标签，选项弹窗中也显示中文 |
| COLOR | ColorPicker 组件 | 色盘 + 透明度 |
| BOOL | Toggle 开关 | 开/关 |
| STRING | TextField 文本输入框 | 直接输入 |
| SPACING | 若值为 table → 展开为 4 个数值（上右下左），否则单值 NUMBER | 同 NUMBER |

#### px/% 切换（SIZE 类型）

- 切换时根据**父元素尺寸**换算：
  - `px → %`：`percent = floor(px / parentSize * 100)`
  - `% → px`：`px = floor(percent / 100 * parentSize)`
- 父元素尺寸获取：
  1. 优先从 `currentWidget_.parent_:GetLayout()` 获取（Yoga 设计坐标）
  2. 兜底使用 `graphics:GetWidth() / UI.GetScale()`（屏幕设计尺寸）
- 方向判断：属性名含 `height`/`Height`/`top`/`bottom` → 取高度，否则取宽度

#### 属性行结构

每行包含：
1. **中文标签**（来自 PropertyDefs.GetLabel，宽 85px）
2. **控件**（根据类型创建）
3. **覆盖开关**（仅有模板的框架属性显示）
4. **删除按钮**（`×`，每行都有）

覆盖开关：
- 关闭（默认）→ 修改模板属性 → 影响所有关联该模板的元素
- 开启（橙色）→ 修改覆盖值 → 仅影响当前元素
- 点击切换时调用 `SetOverride` / `ClearOverride`，然后 `Refresh()` 重建面板

删除按钮（`×`）：
- 每个属性行末尾都有一个删除按钮（红色 `×`）
- 点击行为取决于当前属性的编辑状态：
  - **模板属性（覆盖未开启）** → 调用 `RemoveTemplateProperty()`，从模板中移除该属性，影响所有关联元素
  - **覆盖值或独立属性** → 调用 `RemoveProperty()`，回退到模板值或清除

#### 自动注册

首次修改未注册元素时自动触发：
- 调用 `ElementStore.AutoRegister(pageId, pageRoot, widget)`
- 生成 `{pageId}.auto_{treePath}` 形式的 elementId
- 注册失败则降级为直接 `SetStyle()`（不持久化）

#### 隐式属性补充

根据 widget 的 `_className` 自动补充应有但未显式设置的属性：

| className | 补充属性 |
|-----------|---------|
| Button | text, fontSize, textColor, variant, disabled, borderRadius |
| Label | text, fontSize, fontColor, textAlign, maxLines |
| TextField | placeholder, fontSize, fontColor |
| Toggle | checked, disabled |
| Image | src, backgroundFit |

#### 属性分类与排序

属性按 `PropertyDefs.CATEGORY_ORDER` 顺序分组显示：
尺寸 → Flex布局 → 间距 → 内边距 → 外边距 → 定位 → 背景 → 边框 → 显示 → 文字 → 变换 → 组件 → 按钮

未归类的属性显示在"其他"分类下。

#### "添加属性"弹窗

- 按钮位于面板底部
- 点击弹出包含所有**当前未设置**的 PropertyDefs 属性列表
- 按分类分组，按标签排序
- 点击属性项 → 设置默认值 → 关闭弹窗 → 刷新面板
- 弹窗实现：backdrop（zIndex=19999，点击关闭）+ content（zIndex=20000）作为兄弟节点

**添加属性的目标判断**：
- 元素已关联模板 → 通过 `GetElementTemplateId()` 获取模板 ID，调用 `SetTemplateProperty()` 将属性添加到模板（传播到所有关联元素）
- 元素未关联模板 → 调用 `SetProperty()` 添加为独立属性

#### 长按重复

- `onPointerDown` 时启动 `repeatState_`
- `Update()` 中检测：
  - 鼠标松开 → 清除状态
  - 首次延迟 0.35s → 之后每 0.05s 触发一次

#### 面板拖拽

- 标题栏 `onPointerDown` 开始拖拽，记录偏移量
- `Update()` 中跟踪鼠标位置更新面板 `left`/`top`
- 限制面板不完全移出屏幕

#### 公开接口

| 接口 | 说明 |
|------|------|
| `Init(rootContainer)` | 设置根容器引用 |
| `Show(elementId, widget, store, closeFn, isRefresh)` | 显示面板 |
| `Hide()` | 隐藏面板 |
| `Refresh()` | 刷新面板（保持同一元素，重建内容） |
| `IsVisible()` | 面板是否可见 |
| `GetPanel()` | 获取面板 widget |
| `Update(dt)` | 每帧更新（拖拽+长按） |

### 5. PropertyDefs.lua — 属性元数据

#### 职责

定义所有可编辑属性的元数据：类型、范围、中文标签、是否框架属性、分类。

#### 类型枚举 TYPE

| 类型 | 说明 | 值格式 |
|------|------|--------|
| NUMBER | 纯数值（px） | number |
| SIZE | 数值或百分比 | number 或 "50%" |
| STRING | 文本 | string |
| COLOR | 颜色 | {r, g, b, a}（0-255） |
| ENUM | 固定选项 | string |
| BOOL | 布尔值 | boolean |
| SPACING | 单值或四值 | number 或 {top, right, bottom, left} |

#### 框架属性 vs 内容属性

| 分类 | 说明 | 模板管理 | 示例 |
|------|------|---------|------|
| 框架属性 (`framework=true`) | 布局和视觉样式 | 可加入模板 | width, fontSize, backgroundColor |
| 内容属性 (`framework=false`) | 业务内容 | 不加入模板 | text, src, checked |

#### 属性分类 CATEGORY

| 分类 | 包含属性 |
|------|---------|
| 尺寸 | width, height, minWidth, minHeight, maxWidth, maxHeight, aspectRatio |
| Flex 布局 | flexDirection, justifyContent, alignItems, alignSelf, alignContent, flexWrap, flexGrow, flexShrink, flexBasis |
| 间距 | gap, rowGap, columnGap |
| 内边距 | padding, paddingTop/Right/Bottom/Left, paddingHorizontal, paddingVertical |
| 外边距 | margin, marginTop/Right/Bottom/Left, marginHorizontal, marginVertical |
| 定位 | position, top, right, bottom, left, overflow |
| 背景 | backgroundColor, backgroundImage, backgroundFit |
| 边框 | borderRadius, borderWidth, borderColor |
| 显示 | opacity, visible |
| 文字 | text, fontSize, fontFamily, fontWeight, fontColor, color, textAlign, verticalAlign, lineHeight, letterSpacing, maxLines, whiteSpace |
| 变换 | scale, rotate, zIndex |
| 组件 | variant, disabled, placeholder, checked, src |
| 按钮 | textColor, hoverBackgroundColor, pressedBackgroundColor |

#### 类型推断

当属性不在 `defs` 中时，通过值推断类型：
- `boolean` → BOOL
- `number` → NUMBER
- `string` 含 `%` → SIZE，否则 → STRING
- `table` 长度 4 且全是 number → COLOR，否则 → SPACING

#### 查询接口

| 接口 | 说明 |
|------|------|
| `Get(propName)` | 获取属性完整定义 |
| `IsFramework(propName)` | 是否框架属性 |
| `GetCategory(propName)` | 获取分类 |
| `GetLabel(propName)` | 获取中文标签 |
| `GetRange(propName)` | 获取数值范围（默认 {0, 500, 1}） |
| `InferType(value)` | 从值推断类型 |
| `GetEffectiveType(propName, value)` | 优先用定义，否则推断 |

### 6. Exporter.lua — 导出器

#### 职责

收集 ElementStore 中所有修改，序列化为 JSON，通过 `print()` 输出到日志。

#### 导出流程

1. 调用 `store.GetModifiedTemplates()` 获取所有 `_modified=true` 的模板
2. 调用 `store.GetModifiedElements()` 获取所有有修改的元素
3. 颜色值（`{r,g,b,a}` table）转换为 `#RRGGBB` 或 `#RRGGBBAA` 字符串
4. 元素按 `pageId` 分组
5. 使用 `cjson.encode()` 序列化
6. 通过 `print("[UIEditor] 导出内容: " .. jsonStr)` 输出

#### 导出 JSON 格式

```json
{
  "version": "3.0",
  "exportDate": "2026-03-24T12:00:00",
  "templates": [
    {
      "templateId": "tpl_primary_button",
      "name": "主按钮模板",
      "properties": {
        "width": 380,
        "height": 50,
        "@1.fontSize": 18
      }
    }
  ],
  "pages": [
    {
      "pageId": "main_menu",
      "elements": [
        {
          "elementId": "main_menu.title_area",
          "overrides": { "height": "25%" },
          "properties": null
        },
        {
          "elementId": "main_menu.logo",
          "templateId": null,
          "properties": { "width": "80%", "height": "100%" }
        }
      ]
    }
  ]
}
```

#### 修改判定

**模板被修改**：`_modified == true`（调用过 `SetTemplateProperty` 或 `CreateTemplate`）

**元素被修改**（任一条件成立）：
- `overrides` 不为空
- `properties` 不为空
- `templateId` 与 `_originalTemplateId` 不同

#### 导出规则

- 导出所有页面的所有修改，未修改的页面/元素不出现
- 颜色统一转为 hex 字符串
- 不写文件，仅输出到日志

## 设计决策

### 不支持撤销/重做

理由：增加复杂度，收益有限。替代方案：
- 手动改回原值
- 不导出（修改仅存于运行时）
- 重启游戏回到初始状态

### 导出方式选择 print 而非文件

理由：运行环境文件写入有沙箱限制，`print()` 输出到日志更可靠。AI 从日志中读取导出内容。

- 子元素不一定有 id，索引更通用

## 目录结构

```
scripts/
├── main.lua          # 模式路由（判断单人/多人，加载对应入口）
├── network/          # 网络基础设施
│   ├── Server.lua

│   ├── ServerDispatcher.lua    # 服务端数据分发层
│   └── ClientDispatcher.lua    # 客户端数据分发层
├── shared/           # 双端共用：定义 + 纯查询
│   └── [模块名]/
│       ├── Defs.lua            # 该模块的定义
├── server/           # 仅服务端：状态 + 处理 + 持久化
│   └── [模块名]/
└── client/           # 仅客户端：请求发送 + UI 展示
    └── [模块名]/
```

### 文件归类原则

代码按"谁运行"分目录。判断方法：这段代码需要可变状态吗？需要在哪端运行？

| 目录 | 运行端 | 放什么 | 判定标准 |
|--|--|--|--|
| `shared/模块名/` | 双端都加载 | Defs 定义 + 纯查询函数 | 只读数据，禁止状态变更、校验、权威计算 |
| `server/模块名/` | 仅服务端 | State + Handler + 权威计算 | 持有可变状态，执行所有状态变更、校验、权威计算 |
| `client/模块名/` | 仅客户端 | 请求发送 + UI 展示 | 只展示服务端发来的数据，不持有游戏状态 |
| `network/` | 按文件区分 | 网络基础设施（连接、分发层） | 传输层，不含业务逻辑 |

常见错误：

- ❌ 在 `client/` 中计算伤害、判断是否可购买 → 应在 `server/` 中计算
- ❌ 在 `server/` 中写 UI 代码 → 应在 `client/` 中
- ❌ 在 `shared/` 中放校验或权威计算 → `shared/` 只放定义和纯查询，状态变更、校验、权威计算放 `server/`
- ❌ 把模块定义放在 `shared/Defs/` → 应放在 `shared/模块名/Defs.lua`，跟随模块

## Headless 模式

服务端无图形渲染（Headless），共享代码或工具库中引用 `graphics`/`console`/`debugHud` 会崩溃。在 Server.lua 最顶部（所有 `require` 之前）检测并注入 Mock：

```lua
if GetGraphics() == nil then
    local mockGraphics = {
        SetWindowIcon = function() end,
        GetWidth = function() return 1920 end,
        GetHeight = function() return 1080 end,
    }
    function GetGraphics() return mockGraphics end
    graphics = mockGraphics
    console = { background = {} }
    function GetConsole() return console end
    debugHud = {}
    function GetDebugHud() return debugHud end
end
```

| 规则 | 说明 |
|--|--|
| 放置位置 | Server.lua 最顶部，所有 `require` 之前 |
| Mock 范围 | `graphics`、`console`、`debugHud` 三个全局对象 |
| 为什么需要 | shared 代码可能引用 `graphics.width` 等，服务端没有图形子系统会崩溃 |

## 术语速查

### 通知类型

框架中所有客户端通知分为两类，均根据游戏 UI 风格创建：

**提示（Toast）**：顶部居中，一行文字，3 秒自动消失，点击直接关闭，不阻断操作。

**弹窗（Modal）**：屏幕居中 + 半透明遮罩，标题 + 说明 + 按钮，阻断背后操作，必须玩家点按钮关闭。

### LOGO 载入页

| 项目 | 说明 |
|--|--|
| 风格 | 根据游戏 UI 风格创建 |
| LOGO | 有 LOGO 资源时居中显示，尽量大 |
| 提示文本 | LOGO 正下方，有内容就显示，没内容就隐藏 |

### 按模块拆分传输

服务端向客户端推送数据的传输机制。本质是一个完整数据包，因单条消息不能超过 60KB（TapTap 容器限制），所以按模块拆分为多条小消息传输。

| 项目 | 说明 |
|--|--|
| 安全阈值 | 每条消息 < 50KB（60KB 硬限制 - 10KB 开销余量） |
| 超限模块 | 单模块数据 > 50KB 时，继续拆分为多条分片消息，携带 `{ partIndex, totalParts }` |
| 小数据包 | 数据包 < 50KB 时不拆分，直接单条传输 |
| 拆分传输 | 数据包 > 50KB 时按模块拆分，首条携带批次头 `{ batchId, totalCount }` |

上层流程不感知拆分——两端分发层构成对称的传输管道：

```
业务流程 → 服务端分发层（拆分、缓存、发送）→ 网络 → 客户端分发层（收齐、组装）→ 等待数据的客户端流程
```

### 实时表

服务端内存中 per-UID 的玩家数据表。Handler 直接读写此表，防抖计时器到期时整表提交到云端。

### 云档

云端持久化的玩家存档数据（通过 serverCloud scores 存储）。登录时读取到实时表，运行期间由防抖机制定期写回云端。

### 会话

服务端为每个在线玩家维护的连接上下文，包含 UID、连接映射、实时表引用。断线后会话可保留（实时表不释放），等待重连。

### 断线遮罩

| 项目 | 说明 |
|--|--|
| 触发 | 客户端检测到连接断开（进入游戏后） |
| 层级 | 全局最高（zIndex = 9999） |
| 外观 | 半透明遮罩，阻断所有交互 |
| 文案 | 初始："正在连接服务器..."；多次失败后："连接失败" + 重试按钮 |
| 当前页面 | 保留在遮罩背后，不销毁 |
| 重连成功 | 遮罩消失，页面原地刷新（不弹回主菜单） |

### 页面清理（_cleanup）

页面创建时可在 widget 上挂载 `_cleanup` 函数，用于释放资源（取消订阅、清除计时器等）。

| 项目 | 说明 |
|--|--|
| 挂载方式 | `widget._cleanup = function() ... end` |
| 调用时机 | PageRouter 销毁页面前自动调用 |
| 必须清理 | ClientDispatcher 订阅、定时器 |

### 防抖（Debounce）

`markDirty(uid, "模块名")` 承担两项职责：

| 职责 | 触发时机 | 行为 |
|--|--|--|
| 即时推送 | 调用时立即执行 | 将该模块的展示数据交给服务端分发层推送，同时更新分发层缓存 |
| 存档防抖 | 调用时启动/复用全局计时器（默认 5 秒） | 计时器到期时，所有已注册 key 的实时表整表提交到同一个 BatchCommit |

Handler 只需调用 `markDirty(uid, "模块名")`，推送和存档均自动完成。

### isInGame 标志

由服务端维护（会话级布尔标志）。客户端成功进入游戏主界面时通知服务端置为 `true`，之前为 `false`。客户端本地也维护同名标志用于 UI 判断。

### 收齐检查

| 项目 | 说明 |
|--|--|
| 触发条件 | 客户端收到带批次头的拆分消息时启动 |
| 判定标准 | 已收到的条数 = 批次头中的 `totalCount` |
| 超时时间 | 2 秒 |
| 超时处理 | 向服务端分发层发送补齐请求，携带已收到的部分列表 |

## ServerDispatcher — 服务端数据分发层

### 职责

服务端唯一的数据推送出口。所有业务流程通过此层向客户端推送数据。

### 核心功能

| 功能 | 说明 |
|--|--|
| 拆分发送 | 接收完整数据包 → 按模块拆分传输（见术语） → 发送 |
| 缓存 | per-UID 缓存最近一次全量推送的完整数据 |
| 补齐响应 | 收到客户端补齐请求时，从缓存中取出缺失部分重发；缓存为空时走正常流程重新生成 → 缓存 → 发送 |
| 重连重发 | 重连成功时，直接从缓存重发；缓存为空时同补齐响应 |
| 增量推送 | `markDirty(uid, "模块名")` 触发时，立即推送该模块的展示数据，同时更新缓存中对应模块的数据 |

### 接口设计

```lua
-- 全量推送（进入游戏 / 重连）
ServerDispatcher.pushFullState(uid, allModulesData)

-- 增量推送（markDirty 触发）
ServerDispatcher.pushModule(uid, moduleName, moduleData)

-- 补齐响应
ServerDispatcher.handleResendRequest(uid, missingModules)

-- 重连重发
ServerDispatcher.resendFromCache(uid)

-- 清理缓存
ServerDispatcher.clearCache(uid)
```

## ClientDispatcher — 客户端数据分发层

### 职责

客户端唯一的数据接收入口。收齐拆分消息 → 组装完整数据包 → 分发到对应流程/页面。

### 核心功能

| 功能 | 说明 |
|--|--|
| 收齐组装 | 收集拆分消息，收齐后组装完整数据包 |
| 补齐请求 | 2 秒未收齐 → 向服务端分发层请求补齐 |
| 数据存储 | 缓存最新的各模块数据，供页面读取 |
| 订阅通知 | 页面订阅某模块数据变化，推送到达时回调 |

### 接口设计

```lua
-- 读取模块数据
ClientDispatcher.get("moduleName") → data or nil

-- 订阅模块数据变化
ClientDispatcher.subscribe("moduleName", callback)

-- 取消订阅
ClientDispatcher.unsubscribe("moduleName", callback)
```

### 工作流程

```
服务端分发层发送数据
  │
  └─ 客户端数据分发层（统一入口）
      │
      ├─ 单条消息（无批次头）→ 视为完整数据包，直接分发
      │
      ├─ 拆分消息（有批次头）→ 收集各条消息
      │   ├─ 收齐（已收到 totalCount 条）→ 组装完整数据包，分发
      │   └─ 2 秒未收齐 → 向服务端分发层请求补齐
      │       ├─ 补齐成功 → 组装完整数据包，分发
      │       └─ 补齐失败（连接断开）→ 走断线流程
      │
      └─ 增量推送（markDirty 触发的单模块变更）→ 更新缓存，通知该模块的订阅者
```

## SaveManager — 存档系统

### 职责

管理玩家持久状态的内存驻留和云端同步。核心模型：实时表（内存）+ 防抖写云（定期同步到云端）。

### 接口设计

```lua
-- 模块注册
SaveManager:register("moduleName", {
    key = "save_world",                           -- 可选，省略则归入默认 key save_data
    getDefault = function() return { ... } end,   -- 必填，子表默认值
    onLoad = function(data) ... end,              -- 可选，加载后的字段迁移
    onCleanup = function(uid) ... end,            -- 可选，会话清理时释放运行时状态
})

-- 读写实时表
SaveManager:getTable(uid, "moduleName") → table（引用）

-- 标记变更（双职责：即时推送 + 存档防抖）
SaveManager:markDirty(uid, "moduleName")

-- 查询加载状态（守卫链用）
SaveManager:isLoaded(uid) → boolean

-- 频率限制检查（守卫链用）
SaveManager:checkRateLimit(uid) → boolean
```

### 存储分配

| 存储工具 | 用途 |
|--|--|
| `scores`（ScoreSet） | 玩家全部游戏状态，含货币 |
| 排行榜 API（LeaderboardSubmit） | 排行榜数据，独立提交 |

货币作为普通字段存入 scores。两套存储互不干扰。

### Key 组织

所有 key 内部按模块名分子表，模块名即命名空间。`_meta` 是存档系统保留名。

```lua
-- 默认 key
ScoreSet(uid, "save_data", {
    _meta = { lastSaveTime = ... },   -- 系统保留
    character = { ... },               -- character 模块
    item = { ... },                    -- item 模块
})

-- 自定义 key（数据量大的模块单独分 key）
ScoreSet(uid, "save_world", {
    worldInstance = { ... },
})
```

**分 key 规则**：未指定 key → 归入 `save_data`；数据量大 → 注册时指定自定义 key；单 key 接近 1MB → 必须拆分。

### 模板匹配（数据加载）

```
各已注册模块 getDefault() → 按 key 组装模板
  │
  └─ 从云端批量读取所有已注册 key
        │
        ├─ 该 key 云端有数据 → 将云端数据覆盖到模板上
        │   └─ 模板中有但云端没有的字段 → 保留默认值（新增字段自动补全）
        │
        └─ 该 key 云端无数据（首次登录）→ 保持模板默认值
```

### 字段迁移（onLoad）

当模块数据结构发生变化时，在 `onLoad` 中处理迁移：

```lua
SaveManager:register("inventory", {
    getDefault = function()
        return { items = {}, capacity = 50 }
    end,
    onLoad = function(data)
        if data.bagSize then
            data.capacity = data.bagSize
            data.bagSize = nil
        end
    end,
})
```

| 规则 | 说明 |
|--|--|
| 执行时机 | 模板匹配完成后，实时表可用前 |
| 直接修改 | `data` 是实时表引用，直接修改即可 |
| 幂等性 | 迁移逻辑必须可重复执行 |
| 新增字段 | 不需要 onLoad，模板匹配自动补全 |

### 防抖机制

全局统一防抖周期（默认 5 秒），一个计时器。

`markDirty(uid, "模块名")` 的存档防抖职责：
- 确保该 UID 的防抖计时器在跑（已在跑则跳过）
- 窗口内后续变更不重置计时器
- 计时器到期时，该 UID 所有已注册 key 的实时表整表提交到同一个 BatchCommit

### 提交（写云）

```
update() 每帧检查
  │
  ├─ 该 UID 防抖计时器到期？
  │   └─ 否 → 跳过
  │
  └─ 是 → 构建 BatchCommit：
        各已注册 key：实时表整表 → ScoreSet
        save_data 的 _meta.lastSaveTime = 当前服务器时间
        Commit 提交（异步）
```

### 排行榜提交

| 规则 | 说明 |
|--|--|
| 触发时机 | 跟随存档提交——提交存档时检查距上次排行榜提交是否满 60 秒，满了就一并提交 |
| 提交方式 | 独立异步调用（LeaderboardSubmit），不混入存档 BatchCommit |
| 数据来源 | 从实时表现算现取 |
| 失败处理 | 忽略，不重试，不阻断存档 |

### 写云回调

```
回调返回
  │
  ├─ sessionVersion 守卫：会话已更迭？
  │   └─ 是 → 跳过后续操作
  │
  ├─ 成功 → 正常结束
  │
  └─ 失败
        ├─ 在线 → 弹提示（Toast）"存档上传失败"
        │         防抖继续运行，下次到期自动重试
        │
        └─ 断线 → 重试（上限 3 次，每次重试前校验 sessionVersion）
              └─ 超限放弃，清理会话
```

### 服务端关闭

```
收到关闭信号
  → 停止接受新连接
  → 所有在线玩家并行处理：
        ├─ 有待提交数据 → 立即提交
        │     ├─ 成功 → 完成
        │     └─ 失败 → 重试（上限 3 次）→ 超限放弃
        └─ 无待提交数据 → 直接完成
  → 等所有回调完成（超时上限 15 秒）
  → 超时未完成的放弃
  → 关闭服务
```

### 存档元数据（_meta）

`save_data` key 中保留 `_meta` 子表。业务模块禁止使用此名称。

| 字段 | 写入时机 | 说明 |
|--|--|--|
| `lastSaveTime` | 构建 BatchCommit 时 | 当前服务器时间戳 |

**离线时长计算**：玩家上线 → 从云端读取 `_meta.lastSaveTime` → `offlineDuration = 当前服务器时间 - lastSaveTime` → 服务端执行离线结算。

注意：`lastSaveTime` 在构建 BatchCommit 时写入实时表（不是写云成功后）。即使写云失败，下次提交时也会更新。

## 场景同步（REPLICATED）

引擎内置 Scene Replication：服务端以 `REPLICATED` 模式创建的节点自动同步到所有客户端（位置、旋转、缩放、节点变量）。无需手写序列化或消息分发。

### 机制概述

- 服务端创建 `scene_:CreateChild("Enemy", REPLICATED)` → 客户端自动出现同名节点
- 服务端修改 `node.position` → 客户端自动平滑插值（引擎自动挂载 SmoothedTransform）
- 服务端调用 `node:SetVar(key, value)` → 客户端通过 `node:GetVar(key)` 读取
- 节点在服务端 Remove → 客户端自动移除

### 服务端创建实体

```lua
-- 服务端：创建游戏实体（REPLICATED 模式）
local enemyNode = scene_:CreateChild("Enemy", REPLICATED)
enemyNode.position = spawnPos

-- 添加游戏逻辑组件（不添加渲染组件，服务端无图形）
-- 渲染组件由客户端监听 NodeAdded 自行挂载

-- 通过节点变量同步轻量状态
enemyNode:SetVar("HP", Variant(100))
enemyNode:SetVar("State", Variant("idle"))
```

### 客户端处理

```lua
-- 客户端：本地内容必须显式 LOCAL，否则默认 REPLICATED 会和服务端冲突
local cameraNode = scene_:CreateChild("Camera", LOCAL)
local lightNode = scene_:CreateChild("Light", LOCAL)

-- 监听服务端实体到达，挂载渲染组件
SubscribeToEvent("NodeAdded", function(eventType, eventData)
    local node = eventData["Node"]:GetPtr("Node")
    local name = node.name
    if name == "Enemy" then
        local model = node:CreateComponent("AnimatedModel", LOCAL)
        model.model = cache:GetResource("Model", "Models/Enemy.mdl")
        -- ... 挂载材质、动画控制器等
    end
end)
```

### NetworkPriority — 大量实体时的带宽控制

当场景中实体数量较多时，用 `NetworkPriority` 控制每个节点的更新频率：

```lua
-- 服务端：为实体添加 NetworkPriority
local priority = enemyNode:CreateComponent("NetworkPriority")
priority.basePriority = 100.0     -- 基础优先级
priority.distanceFactor = 0.01    -- 距离衰减因子
priority.minPriority = 10.0       -- 最低优先级
-- 距离观察者越远，更新频率越低
```

客户端需上报观察者位置，供服务端计算优先级：

```lua
-- 客户端：在 PostUpdate 中上报
local serverConnection = network:GetServerConnection()
if serverConnection then
    serverConnection:SetPosition(cameraNode.worldPosition)
end
```

### 相机跟随规则

REPLICATED 节点在客户端自动挂载 `SmoothedTransform`（引擎行为），已内置平滑插值。客户端相机跟随时：

- **直接使用 `node.worldPosition`**，不要额外加 lerp
- 额外插值会叠加引擎插值，造成延迟或抖动

```lua
-- ✅ 正确：直接跟随
cameraNode.position = playerNode.worldPosition + Vector3(0, 5, -10)

-- ❌ 错误：多余的 lerp
cameraNode.position = Lerp(cameraNode.position, playerNode.worldPosition + offset, dt * 5)
```

### 数据分工

| 数据类型 | 同步方式 | 举例 |
|--|--|--|
| 空间/视觉状态 | REPLICATED 节点属性 | 位置、旋转、缩放 |
| 轻量游戏状态 | 节点变量 `SetVar/GetVar` | HP、当前动作、阵营 |
| 持久游戏数据 | StateUpdate（markDirty） | 背包、货币、关卡进度 |
| 操作请求 | 请求/响应通道 | 购买、使用道具、出牌 |

### 注意事项

1. 服务端不创建渲染组件（StaticModel、Light 等），Headless 模式下无图形子系统
2. 客户端创建本地内容（相机、光照、UI）时必须指定 `LOCAL` 模式
3. 节点变量适合同步少量轻量状态；大量结构化数据仍走 StateUpdate
4. NetworkPriority 的 `distanceFactor` 需根据游戏视野范围调整

## Server.lua — 服务端主控

### 职责

服务端入口，负责连接管理、守卫链、请求路由、进入游戏流程编排。

### 进入游戏流程

进入游戏流程由三条路由组成。启动游戏路由是入口，根据会话状态分发到新老玩家路由或重连游戏路由。

```
启动游戏路由 → 服务端判定会话
  ├─ 新/老玩家 → 新老玩家路由（页面处理 → 云档加载 → 云档结果 → 进入游戏）
  └─ 重连玩家 → 重连游戏路由
      ├─ 继续游戏 → 恢复页面
      └─ 新游戏 → 刷盘清理 → 云档处理 → 进入游戏
```

#### 服务端进入游戏流程

```
收到连接和 uid
  │
  ├─ 旧会话存在？
  │   ├─ 存在
  │   │   ├─ 旧连接还活着（顶号）
  │   │   │   ├─ 异步通知旧客户端"账号在其他设备登录"
  │   │   │   │   └─ 旧客户端弹窗，只有"退出游戏"按钮（结束进程）
  │   │   │   └─ 清理旧会话（不残留，不刷盘）→ 当作新玩家，继续往下
  │   │   │
  │   │   └─ 旧连接已断（重连场景）
  │   │       ├─ 该 UID 实时表存在
  │   │       │   ├─ isInGame = true → 绑定新连接到旧会话，分发层从缓存重发全部展示数据 → 进入【重连游戏路由】
  │   │       │   └─ isInGame = false → 绑定新连接到旧会话，等待客户端驱动
  │   │       └─ 该 UID 实时表不存在 → 清理旧会话 → 继续往下
  │   │
  │   └─ 不存在 → 继续往下
  │
  └─ 发送初始化数据到客户端（动画配置、分区信息）→ 进入【新老玩家路由】
```

#### 服务端云档处理流程

```
收到云档读取请求
  │
  ├─ 云服务不响应（5 秒超时）→ 交给分发层推送 { status: "timeout" }
  │
  ├─ 读取失败 → 交给分发层推送 { status: "failed" }
  │
  └─ 读取成功 → pcall 加载数据到实时表（结构校验 + 字段修复）
      │
      ├─ 加载成功（数据已在实时表中）→ 进入【登录处理流程】
      │
      └─ 加载异常（结构严重损坏）→ 自动修复，不通知客户端
          │
          └─ pcall 修复（默认值打底，能捞的数据覆盖上去）
              ├─ 修复成功 → 写入实时表，立即写云（覆盖坏数据）→ 进入【登录处理流程】（tips 前缀"存档已修复"）
              └─ 修复也失败 → 完全重置，用默认值写入实时表 → 进入【登录处理流程】（tips 前缀"存档已重置"）
```

#### 登录处理流程

加载成功、修复成功、重置建档三条路径汇聚于此。实时表已就绪，执行统一的登录处理：

```
实时表已就绪
  │
  └─ pcall 执行离线结算及登录相关处理（结果写回实时表）
      │
      ├─ 处理成功 → 异步开启防抖窗口提交 → 交给分发层推送 { status: "success", tips: "离线奖励...", 各模块展示数据 }
      │
      └─ 处理异常 → 异步开启防抖窗口提交 → 交给分发层推送 { status: "success", tips: "登录处理异常...", 各模块展示数据 }
```

#### 服务端重连新游戏流程

```
收到客户端的新游戏请求
  │
  └─ 同步刷盘（最多重试 3 次，失败放弃）→ 清理旧会话 → 判断分区
      ├─ 是 → 发送分区数据 → 客户端显示分区页 → 玩家选择分区 → 发送选区请求 →【服务端云档处理流程】
      └─ 否 → 发送云档读取请求 →【服务端云档处理流程】
```

### 连接与断开处理

```
连接断开
  │
  └─ isInGame？
      │
      ├─ 否（进入游戏前）
      │   └─ 服务端：isInGame = false → 仅绑定连接，不路由，等客户端请求
      │
      └─ 是（进入游戏后）
          ├─ 服务端：保留会话和实时表，防抖计时器正常工作
          └─ 重连成功 → isInGame = true → 更新连接映射，分发层从缓存重发全部展示数据
```

### 断线保留策略

| 玩家数量 | 保留时长 |
|--|--|
| 单人 | 5 分钟 |
| 多人 | 10 分钟 |

保留期间：游戏逻辑继续运行、防抖正常工作、写云失败重试 3 次后放弃、不向断开客户端发送通知。

超时清理：提交剩余数据（如有）→ 清理会话、释放实时表。

### sessionVersion 守卫

每个玩家会话持有 `sessionVersion`，顶号时递增。所有异步写云回调发起时捕获当前值，回调中校验：不一致 → 跳过。

```lua
function commitKeys(uid)
    local capturedVersion = sessionVersions[uid]
    BatchCommit(keys, function(success)
        if sessionVersions[uid] ~= capturedVersion then
            return  -- 会话已更迭，放弃
        end
        -- 安全处理回调...
    end)
end
```

## Client.lua — 客户端主控

### 职责

客户端入口，负责连接管理、进入游戏页面流程编排、断线处理。

### 客户端进入游戏流程

```
玩家启动游戏
  │
  └─ 显示 LOGO 载入页，提示文本"正在连接中" → 连接服务端，发送 uid
      ├─ 连接成功 → 等待服务端响应
      └─ 连接失败 → 每 3 秒重试（30 秒超时）
          ├─ 连接成功 → 等待服务端响应
          └─ 超时 → 弹窗"游戏连接失败" + 退出游戏按钮
              └─ 点击 → engine:Exit()
```

### 客户端页面处理流程（新老玩家路由）

```
收到服务端初始化数据
  │
  ├─ 有动画 → 开始播放开头动画（并发，可点击跳过）
  │
  └─ 判断分区
      ├─ 是 →（若有动画则等播完）→ 显示分区页 → 玩家选择分区 → 弹窗"读取存档中" → 发送选区请求
      └─ 否 → LOGO 载入页提示文本改为"读取存档中" → 发送云档读取请求
```

### 客户端云档结果流程

```
收到完整云档结果（经 ClientDispatcher 收齐后交付）
  │
  ├─ status: "timeout" → 弹窗"服务器未响应" + 重试按钮
  │   └─ 点重试 → 弹窗改为"正在处理中..." → 发送重试请求
  │
  ├─ status: "failed" → 弹窗"存档读取失败" + 重试按钮
  │   └─ 点重试 → 弹窗改为"正在处理中..." → 发送重试请求
  │
  └─ status: "success" → 判断分区
      ├─ 是 → 关闭弹窗和分区页 → 进入游戏（有 tips 时弹 Toast）
      └─ 否 → 判断动画是否播完
          ├─ 已播完（或无动画）→ 进入游戏（有 tips 时弹 Toast）
          └─ 未播完 → 等动画播完 → 进入游戏（有 tips 时弹 Toast）
```

### 客户端重连页面流程

```
收到完整重连数据（经 ClientDispatcher 收齐后交付）
  │
  └─ LOGO 载入页提示文本改为"重连游戏中" → 弹窗"发现未结束的游戏进度" + 继续游戏 / 新游戏
      ├─ 继续游戏 → 关闭 LOGO 载入页，用重连数据恢复游戏页面
      └─ 新游戏 → 关闭弹窗，提示文本改为"开始新游戏中" → 发送新游戏请求
```

### 连接断开处理（客户端侧）

```
连接断开
  │
  └─ isInGame？
      │
      ├─ 否（进入游戏前）
      │   └─ 弹窗"连接已断开" + 重试按钮
      │       └─ 点重试 → 重连服务端
      │           └─ 重连成功 → 根据当前等待状态重发请求
      │               ├─ 在动画/分区页，未发过请求 → 不发请求，等玩家继续操作
      │               └─ 已发选区/云档请求，在等结果 → 重发同一个请求
      │
      └─ 是（进入游戏后）
          └─ 弹出断线遮罩"正在连接服务器..."
              └─ 多次失败 → 遮罩改为"连接失败" + 重试按钮
              └─ 重连成功 → 收到完整数据 → 关闭遮罩，刷新 UI
```

## 模块展示数据（buildDisplayData）

每个模块负责提供自己的展示数据，供 markDirty 和全量推送调用：

```lua
function Module.buildDisplayData(uid)
    local data = SaveManager:getTable(uid, "moduleName")
    local rt = runtimeState[uid]  -- 模块自己的运行时状态
    return {
        -- 持久状态（从实时表读）
        gold = data.gold,
        items = data.items,
        -- 运行时状态
        inCombat = rt and rt.inCombat or false,
        -- 派生数据（现算）
        totalPower = calcPower(data),
    }
end
```

三类数据来源：持久状态（SaveManager 实时表）、运行时状态（模块内部 state）、派生数据（现算）。
