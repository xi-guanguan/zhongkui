# 游戏接入大语言模型：从乱说话到演戏

> **作者**: 星河渔火 | **级别**: L1-实操 | **来源**: TapTap 开发心得 | **项目**: 《审讯室》

---

## 核心观点

LLM 游戏开发五条经验：
1. 不要让一个 LLM 做太多事 → 双模型架构
2. LLM 是演员不是计算器 → 数值交给服务端
3. Prompt 要写"动机"不要写"规则"
4. 免费额度是宝藏 → 共用模型池+自动切换
5. 证物是叙事的路标 → 剧本驱动

---

## 三大痛点

| 痛点 | 症状 | 根因 |
|------|------|------|
| LLM不是演员是公务员 | "根据我的懦弱性格，我应该说..." | 性格标签→机械执行 |
| 视角混淆 | 推荐话术以嫌疑人口吻输出 | 单模型同时扮两个角色 |
| 对话跳针 | 前面聊账本，出示证物后突然跳转 | 数值驱动无过渡 |

---

## 架构演进 V1→V5

| 版本 | 方案 | 效果 |
|------|------|------|
| V1 | 单LLM全包（角色+建议+数值） | 视角混淆，机械 |
| V2 | 加格式约束+正反面示例 | 治标不治本 |
| V3 | LLM只演戏，数值服务端算 | 表演质量提升 |
| **V4** | **双模型架构**（A嫌疑人/B参谋） | 视角问题彻底解决 |
| V5 | 模型池自动切换 | 几乎零成本运营 |

---

## 双模型架构（V4 核心）

**模型A**（嫌疑人）：只负责角色扮演，System Prompt 纯角色设定
**模型B**（警官参谋）：只负责生成推荐话术，完全警官视角

### 通信流程

```
玩家输入
  → 服务端基础数值修正
  → 调用模型A（角色设定 + 当前感觉 + 对话历史 + 玩家输入）
  → A返回嫌疑人对话
  → 用A的输出调用模型B（案件背景 + 当前阶段 + A的回应）
  → B返回3条推荐话术
  → 组合发给客户端
```

**关键**：B看到的只是"嫌疑人说了什么话"，不是A的内心想法。两个独立工作流互不干扰。

## Prompt 工程三原则

### 原则1：用"终极动机"替代"性格列表"

```
❌ 错误：你是懦弱的、狡猾的、自私的、爱女儿的
✅ 正确：活下来，保护女儿，为自己谋取最大利益。
        你可以在懦弱、狡猾、自私之间反复横跳。
```

性格列表 = 紧箍咒（逐条遵守），终极动机 = 驱动引擎（自己想办法）

### 原则2：用"感觉"替代"数值"

```
❌ 错误：fear=85, trust=15
✅ 正确：
  fear < 40 → "你感觉胸有成竹"
  fear 40-70 → "你开始心虚，但还想狡辩"
  fear > 70 → "你惊慌失措，快瞒不住了"
```

给LLM看数字 = 看体检报告。给它讲感觉 = 讲戏。

### 原则3：推荐话术的保底规则

```
第1条：直接回应当前对话（关联性最强）
第2条：可推进案情的替代问法
第3条：保底推进话术（确保总是可用）
```

## 成本控制：模型池策略

- 百炼平台约 150 个纯文本模型各有 100 万免费额度
- 共享一个 model_list + 共用 current_model_index
- 遇 403（额度用完）→ 自动跳下一个
- 遇 400（多模态不支持纯文本）→ 初始化时过滤
- 总免费额度约 1.5 亿 Token → 约 5000 次完整游戏
- 双模型成本只增加约 30%（B输出很短）
