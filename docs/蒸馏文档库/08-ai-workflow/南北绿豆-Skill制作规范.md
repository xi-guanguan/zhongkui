> **原作者**: 南北绿豆
> **推荐度**: L2（参考学习）
> **适用场景**: 为 Claude Code 编写模块化 Skill 扩展包时，遵循的结构、格式与设计原则

---

## 这是什么

一份 Claude Code Skill 的完整制作手册，涵盖目录结构、SKILL.md 格式、设计原则、渐进式披露策略、工作流模式、输出模式、6 步制作流程和验证清单。Skill = "上岗培训手册"，将通用 AI 变为领域专家。

## 1. Skill 能力矩阵

| 类型 | 说明 | 示例 |
|------|------|------|
| 专业工作流 | 多步骤领域操作流程 | PDF 表单填写、代码审查 |
| 工具集成 | 特定文件格式/API 交互指南 | DOCX 编辑、BigQuery 查询 |

## 2. 目录结构

```
skill-name/
├── SKILL.md              # [必须] 核心：元数据 + 指令
├── scripts/              # [可选] 可执行脚本（反复使用的代码）
├── references/           # [可选] 参考文档（按需加载，>100行的详细信息）
└── assets/               # [可选] 输出资源（模板/图片，不读入上下文）
```

**禁止创建**: README.md、CHANGELOG.md 等面向人类的辅助文档。Skill 只面向 AI。

## 3. SKILL.md 格式

### 3.1 Frontmatter（YAML，必须）

```yaml
---
name: my-skill-name          # 必须，hyphen-case，≤64字符
description: |                # 必须，≤1024字符，无尖括号
  做什么 + 何时触发。
  这是 Claude 判断是否加载的唯一依据。
license: MIT                  # 可选
allowed-tools:                # 可选
---
```

**`name` 规范**: 小写 `a-z0-9-`，不以 `-` 开头/结尾，无连续 `--`。

**`description` 关键规则**: 所有触发条件必须写在 description 中（不在 Body 中），Claude 仅凭 name + description 决定是否加载。

### 3.2 Body（Markdown 操作指南）

推荐结构：概述 → 核心概念 → 工作流程 → 参考资源链接。

四种组织风格：
- **工作流式** — 有明确步骤序列（PDF 处理）
- **任务式** — 多种独立操作（图片编辑）
- **参考式** — 规范标准指导（品牌指南）
- **能力式** — 多种关联功能（文档处理）

## 4. 核心设计原则

### 4.1 精简至上

默认 Claude 本身足够聪明，只添加它不具备的知识。对每段内容自问：
- "Claude 真的需要这段解释吗？"
- "这段话值得占用的 Token 吗？"

偏向简洁示例而非冗长说明。

### 4.2 自由度匹配

| 自由度 | 形式 | 适用场景 |
|--------|------|---------|
| 高 | 文本指令 | 多种方案都可行 |
| 中 | 伪代码/带参数脚本 | 有推荐模式但允许变化 |
| 低 | 具体脚本 | 操作脆弱易错、一致性至关重要 |

比喻：窄桥+悬崖 → 精确护栏（低自由度）；开阔平原 → 多条路线（高自由度）。

### 4.3 渐进式披露（三层加载）

```
第 1 层：元数据（name + description）    → 始终在上下文（~100 词）
第 2 层：SKILL.md Body                  → 触发时加载（<500 行）
第 3 层：scripts/ references/ assets/   → 按需加载
```

**红线**: SKILL.md Body ≤ 500 行。超过就拆到 references/，并在 SKILL.md 中引用说明。

---

## 5. 渐进式披露三模式

| 模式 | 做法 | 场景 |
|------|------|------|
| 概览+按需参考 | Body 放快速用法，详细内容链接到 references/ | 有高级功能需要深入时 |
| 按领域分文件 | references/ 下按变体分文件（aws.md / gcp.md） | 多平台/多变体场景 |
| 条件分支 | Body 放默认流程，复杂分支链接到 references/ | 有可选高级操作时 |

references/ 保持一层深度，超过 100 行的文件加目录。

---

## 6. 工作流与输出模式

**顺序工作流**: 拆分为清晰编号步骤，每步一个操作。

**条件工作流**: 先判断操作类型，再分支到对应子流程。

**输出模式**:
- **模板模式（严格）** — API 响应、数据格式等必须精确匹配
- **模板模式（灵活）** — 推荐格式但允许适应性调整
- **示例模式** — 通过 输入→输出 对展示预期质量（效果优于文字描述）

---

## 7. 六步制作流程

1. **理解需求** — 收集使用示例，明确功能范围和触发场景
2. **规划资源** — 反复编写的代码→scripts/，反复查阅的文档→references/，模板素材→assets/
3. **初始化** — 用 `init_skill.py` 生成骨架
4. **编写内容** — 先实现 scripts/references/assets，再写 SKILL.md，用祈使语气
5. **打包** — 用 `package_skill.py` 验证格式并生成 `.skill` 文件
6. **迭代** — 实际使用中发现问题→改进→再测试

## 8. 验证清单

### 结构
- [ ] SKILL.md 存在于根目录
- [ ] Frontmatter 含 name + description
- [ ] 无多余文档文件（README 等）

### name
- [ ] 小写 hyphen-case，≤64 字符

### description
- [ ] ≤1024 字符，无尖括号
- [ ] 含"做什么"+"何时触发"

### Body
- [ ] < 500 行，祈使语气
- [ ] 详细内容拆到 references/

### 资源
- [ ] scripts/ 经过测试
- [ ] references/ >100 行有目录，一层深度
- [ ] assets/ 只放输出用文件

## 9. 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 触发条件写在 Body 而非 description | Skill 不被触发 | 触发信息放 description |
| SKILL.md 超 500 行 | 挤占上下文 | 拆到 references/ |
| 添加 README 等辅助文件 | 噪音和混淆 | 只保留功能文件 |
| 信息在 SKILL.md 和 references/ 重复 | 浪费 Token + 不一致风险 | 信息只存一处 |
| description 过短/模糊 | 无法准确触发 | 写清具体场景和关键词 |
| 冗长解释代替简洁示例 | 浪费 Token | 用示例展示 |
