# 广告奖励防刷全流程方案：预发放 + 三层防护 + 递减机制

> **这是什么**：解决激励视频广告被玩家刷量的问题。采用"先发奖励、后确认广告"的预发放模式，配合客户端频率控制、服务端独立校验、异常行为检测三层防护，外加奖励递减机制，从根源遏制广告刷量导致的经济崩溃。
>
> **原作者**：石头
> **推荐度**：L3（方案思路完整，但涉及服务端校验和异常检测部分未经充分实战验证，部分逻辑需根据具体广告 SDK 和服务端架构调整）
> **适用场景**：游戏中有激励视频广告（看广告得金币/体力/道具），且已出现或预期会出现玩家刷广告量的情况

---

## 一、问题背景

激励视频广告的典型流程：玩家点击"看广告" → 播放广告 → 广告完成 → 发放奖励。

**被刷的场景**：
- 玩家每天看 500 次广告，每次得 100 金币，日入 50000 金币
- 正常玩家日均获取 2000 金币，经济体系被广告刷子彻底打破
- 刷子用脚本自动点击，全天候刷量

## 二、架构设计：预发放 + 确认/回滚

### 2.1 为什么用预发放

传统流程是"广告完成后发奖励"，但广告 SDK 的回调可能丢失（进程被杀、网络断开），导致：
- 玩家看完广告没拿到奖励 → 差评
- 为了保证体验，开发者倾向于"只要点了就给奖励" → 被刷

**预发放模式**：

```
玩家点击"看广告"
  ↓
[预发放] 立即给予奖励（金币 +100）
  ↓
播放广告
  ↓
├─ 广告正常完成 → [确认] 奖励保留，记录有效观看
├─ 广告中途退出 → [回滚] 扣回奖励（金币 -100）
└─ SDK 回调丢失  → [超时回滚] 30 秒无回调则扣回
```

好处：玩家体验零延迟（点击即看到奖励），同时有回滚兜底。

## 三、三层防护

### 3.1 第一层：客户端频率控制

**目的**：阻止 80% 的普通刷子，降低服务端压力。

```lua
local AdGuard = {
    COOLDOWN = 30,           -- 两次广告之间的冷却时间（秒）
    DAILY_CAP = 10,          -- 每日广告上限
    lastAdTime = 0,          -- 上次看广告的时间戳
    todayCount = 0,          -- 今日已看次数
    todayDate = "",          -- 今日日期（用于跨天重置）
}

function AdGuard:canShowAd()
    -- 跨天重置
    local today = os.date("%Y-%m-%d")
    if today ~= self.todayDate then
        self.todayDate = today
        self.todayCount = 0
    end
    
    -- 检查每日上限
    if self.todayCount >= self.DAILY_CAP then
        return false, "今日广告次数已用完，明天再来吧"
    end
    
    -- 检查冷却时间
    local now = os.time()
    local elapsed = now - self.lastAdTime
    if elapsed < self.COOLDOWN then
        local remaining = self.COOLDOWN - elapsed
        return false, string.format("请等待 %d 秒", remaining)
    end
    
    return true, nil
end

function AdGuard:recordAdView()
    self.lastAdTime = os.time()
    self.todayCount = self.todayCount + 1
end
```

**UI 反馈**：
- 冷却中：按钮灰显 + 倒计时文字（"30 秒后可用"）
- 达到上限：按钮消失或显示"明天再来"
- 接近上限：显示剩余次数（"剩余 2 次"）

### 3.2 第二层：服务端独立校验

**目的**：客户端可以被篡改，服务端做独立判断才能真正防刷。

```lua
-- 客户端：广告完成后通知服务端
function onAdCompleted(adType)
    serverCloud:Call("verifyAdReward", {
        adType = adType,
        clientTime = os.time(),
        todayCount = AdGuard.todayCount,
    }, function(result)
        if result.approved then
            -- 服务端确认，奖励保留
            log:Write(LOG_INFO, "广告奖励已确认")
        else
            -- 服务端拒绝，回滚奖励
            rollbackReward(result.reason)
        end
    end)
end
```

```lua
-- 服务端：独立校验逻辑
function Server:verifyAdReward(playerId, params)
    local player = self:getPlayer(playerId)
    local now = os.time()
    
    -- 1. 频率检查（容忍 5 秒网络延迟）
    local MIN_INTERVAL = 25  -- 比客户端的 30 秒略短，容忍延迟
    if now - player.lastAdTime < MIN_INTERVAL then
        return { approved = false, reason = "频率过快" }
    end
    
    -- 2. 每日次数检查（服务端独立计数）
    local today = os.date("%Y-%m-%d")
    if player.adDate ~= today then
        player.adDate = today
        player.adCount = 0
    end
    
    if player.adCount >= 10 then
        return { approved = false, reason = "超过每日上限" }
    end
    
    -- 3. 更新记录
    player.lastAdTime = now
    player.adCount = player.adCount + 1
    
    -- 4. 按递减规则计算实际奖励
    local reward = calculateDiminishedReward(player.adCount)
    
    return { approved = true, reward = reward }
end
```

### 3.3 第三层：异常行为检测

**目的**：抓出用脚本刷量的"职业刷子"。

```lua
-- 每周定时任务：扫描异常模式
function Server:weeklyAnomalyScan()
    for _, player in ipairs(self:getAllPlayers()) do
        local weeklyLogs = self:getAdLogs(player.id, 7)  -- 最近 7 天
        
        -- 规则 1：连续 7 天接近上限
        local nearMaxDays = 0
        for _, dayLog in ipairs(weeklyLogs) do
            if dayLog.count >= 9 then  -- 上限 10，9 次即接近
                nearMaxDays = nearMaxDays + 1
            end
        end
        
        if nearMaxDays >= 7 then
            -- 标记为可疑，人工审核或自动降权
            self:flagSuspicious(player.id, "连续7天广告接近上限")
        end
        
        -- 规则 2：广告观看时间异常（正常 15~30 秒，刷子可能 5 秒）
        -- （需要广告 SDK 提供观看时长数据）
    end
end
```

## 四、奖励递减机制

**目的**：即使没被防护拦截，大量看广告的收益也在递减，降低刷量动机。

```lua
-- 递减倍率表
local REWARD_TIERS = {
    { maxCount = 3,  multiplier = 1.0 },   -- 前 3 次：100% 奖励
    { maxCount = 6,  multiplier = 0.5 },   -- 第 4~6 次：50% 奖励
    { maxCount = 10, multiplier = 0.2 },   -- 第 7~10 次：20% 奖励
    -- 超过 10 次：被每日上限拦截，奖励 0
}

function calculateDiminishedReward(todayCount)
    local baseReward = 100  -- 基础奖励金币
    
    for _, tier in ipairs(REWARD_TIERS) do
        if todayCount <= tier.maxCount then
            return math.floor(baseReward * tier.multiplier)
        end
    end
    
    return 0
end
```

**UI 提示**：在广告入口显示当前倍率，让玩家知道收益在递减：
- "观看广告 +100 金币"（前 3 次）
- "观看广告 +50 金币"（第 4~6 次）
- "观看广告 +20 金币"（第 7~10 次）

## 五、注意事项

1. **绝不用广告发稀有道具**：广告奖励只给基础消耗品（金币、体力），稀有物品必须通过游戏内容获取，否则即使有防刷机制，经济体系也会失衡
2. **记录所有广告行为**：每次广告请求、完成、回滚都要记日志，便于事后审计和数据分析
3. **预发放的回滚要做好 UI 提示**：玩家看到金币加了又减了会困惑，需要清晰的文案说明（"广告未完成，奖励已收回"）
4. **客户端时间不可信**：频率检查以服务端时间为准，客户端时间仅作参考（客户端可以改系统时间）
5. **离线模式兜底**：如果游戏支持离线玩，服务端不可用时，只用客户端防护（第一层），上线后补校验
6. **递减要提前告知**：不要"偷偷"减奖励，在广告入口明确显示当前收益，降低玩家的负面感受
